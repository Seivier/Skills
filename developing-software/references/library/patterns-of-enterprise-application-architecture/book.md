## Preface

This preface section contains no substantive technical content on patterns or enterprise architecture principles. It is primarily contextual framing: the author recounts a challenging leasing application project that motivated the book; explains that enterprise applications deal with display, manipulation, and storage of complex data while supporting business processes (different from embedded, control, telecom, or desktop software); outlines the book's two-part structure where Part 1 offers narrative chapters introducing architectural problems and Part 2 provides detailed patterns as a reference to be consulted as needed; specifies primary topics covered (layering, domain logic, web UI, object-to-relational database mapping, session state, distribution principles); and notes significant topics excluded due to space constraints (validation, messaging, asynchronous communication, security, error handling, clustering, application integration, architectural refactoring, rich-client UI). The remainder is acknowledgments, colophon information, and publication history.

## Introduction

The Introduction positions enterprise applications as a distinct class of software facing unique complexities compared to other domains like telecom or embedded systems. The book concentrates specifically on enterprise applications—defined through examples like payroll, patient records, shipping tracking, and cost analysis—which share five critical characteristics: (1) Persistent data that outlives hardware and software versions, often requiring migration when systems change; (2) Large volume of data (moderate systems exceed 1 GB with tens of millions of records), managed through relational databases as a "subprofession"; (3) Concurrent access by many users (from dozens to millions in web systems), demanding careful transaction and locking management; (4) Hundreds of distinct UI screens presenting data in multiple ways for varying user expertise levels; (5) Integration with other heterogeneous enterprise systems built at different times with different technologies (COBOL files, CORBA, messaging), creating conceptual dissonance where the same business concepts have subtly different meanings across divisions. Business logic itself is described not as logical but as an accumulation of one-off special cases and negotiations that interact in surprising ways, making organization of business rules crucial as they inevitably change.

Fowler establishes that "architecture" means both the highest-level system breakdown and decisions hard to change, acknowledging it is subjective and evolves during a system's lifetime. He advocates for a layered architecture approach as the most widely useful, though notes that patterns can be architectural (significant decisions about decomposition) or design-level (realizing that architecture), without strict separation since this distinction is subjective.

Three concrete enterprise application examples illustrate why no single architecture suits all: a B2C online retailer prioritizing high volume, scalability, and generic web presentation with straightforward domain logic (order capturing, pricing, shipping); a leasing agreement automation system with few concurrent users but extremely complex business logic in calculating bills, handling returns, and validation; and a small expense-tracking system requiring rapid development with simple initial logic but potential for growth. Each faces fundamentally different challenges, proving that choosing an architecture requires understanding particular problems and selecting appropriate design based on that understanding—not blindly following templates.

The performance section addresses measurement-driven optimization while warning against applying generic advice without real-world validation on actual configurations. Fowler defines precise terminology to avoid ambiguity: Response time (seconds to process requests), Responsiveness (acknowledgment speed vs. completion), Latency (baseline time even for zero work, particularly problematic in remote systems), Throughput (work per unit time, measured as transactions per second), Load (stress measurement, usually concurrent users), Load Sensitivity (response time variance with load; degradation indicates poor load sensitivity), Efficiency (performance divided by resources), Capacity (maximum effective throughput), and Scalability (how adding hardware affects performance, including vertical scaling/scaling up on single servers and horizontal scaling/scaling out with more servers). The example comparing Swordfish and Camel illustrates that capacity, efficiency, and scalability are independent: Camel achieves better capacity and efficiency on fewer servers, but Swordfish scales out better per added server (15 tps vs. 10 tps per additional server), showing how architectural trade-offs depend on specific constraints.

Patterns themselves derive from Christopher Alexander's definition: recurring problems with core solutions that work repeatedly yet differently in each context. Fowler positions patterns as rooted in practice—discovered through observation rather than invented—serving as design vocabulary enabling communication (saying "use Remote Facade" conveys architecture instantly if colleagues know the name). Crucially, patterns are "half baked," requiring finishing in each project's oven; they are never blindly applied and always tweaked contextually. Patterns relate to each other—Class Table Inheritance only appears with a Domain Model—though he attempts to make each self-standing for reference use.

The pattern form Fowler uses consists of: name (crucial for vocabulary creation, e.g., Front Controller, Transform View), intent and sketch (one-sentence summary plus visual representation), motivating problem (a driving scenario, not exhaustive), How It Works (solution description with implementation issues and variations, platform-agnostic), When to Use It (trade-offs against alternative patterns—for example, Page Controller versus Front Controller), Further Reading (curated important references, not comprehensive), and examples in Java or C# (deliberately simple to isolate the pattern core, kept independent rather than connected running examples, focused on understandability over production realism, with error handling and business modeling deliberately minimized).

Finally, Fowler acknowledges these patterns are not comprehensive but serve as starting points. They reflect his current understanding based on observed practice, remain incomplete, and place responsibility on readers to adapt and complete them in their own system contexts. One certainty: software never stands still, and understanding will evolve.

## Chapter 1. Layering

Layering originates as a universal decomposition technique visible across machine architectures (from programming languages through operating system calls to CPU instruction sets), networking protocols (FTP over TCP over IP over ethernet), and enterprise software design. In a layered system, the principal subsystems form a layer cake where each layer rests upon a lower layer, with higher layers consuming services from lower layers while remaining opaque to them—typically, layer four uses layer three which uses layer two, but layer four is unaware of layer two.

The benefits of layering are substantial: individual layers can be understood as coherent wholes without deep knowledge of other layers (understanding FTP doesn't require knowing ethernet details); layers can be substituted with alternative implementations without affecting consumers (FTP runs unchanged over ethernet, PPP, or cable systems); dependencies between layers minimize significantly (changes to physical transmission don't affect FTP if IP remains functional); layers provide excellent standardization points (TCP and IP became standards precisely because they defined how their layers should operate); and layers enable reuse—TCP/IP serves FTP, telnet, SSH, and HTTP, avoiding duplication across protocols.

However, layering has documented disadvantages. Layers encapsulate some things well but not all, resulting in cascading changes—the classic example being adding a field that must appear in the UI, exist in the database, and therefore traverse every intervening layer. Extra layers incur performance costs from repeated transformations between representations, though this is often offset by optimization opportunities (transaction control layers, for instance, can improve overall system efficiency).

The evolution of enterprise layering reflects technological change. Early batch systems weren't consciously layered. The 1990s brought client-server systems (two-layer: client UI plus relational database server), popularized by tools like Visual Basic, Powerbuilder, and Delphi that made it easy to build data-intensive applications by directly binding UI controls to SQL. This worked well for simple display-and-update scenarios but failed with domain logic (business rules, validations, calculations). Such logic was either awkwardly embedded in client screens (leading to duplication) or pushed into database stored procedures (which lacked structuring mechanisms and weren't portable across database vendors).

The object-oriented community's answer was the three-layer system: presentation for UI, domain for structured business logic, and data source for persistence. This failed to gain immediate traction because many systems were simple and client-server tooling was compelling for simple problems. The Web became the catalyst for change—applications built with rich-client business logic suddenly needed Web interfaces, requiring complete rewrites. Conversely, three-layer systems could simply add a new presentation layer. Java as a mainstream object-oriented language and Web tools less tightly coupled to SQL enabled the three-layer approach.

Fowler distinguishes *layer* (logical separation, possible on one machine) from *tier* (physical separation across machines). A three-layer system might run with two physical nodes—domain logic on a desktop and database on a server—but remains three distinct logical layers.

The three principal layers are: **Presentation**, handling user interaction through command-line, text menu, rich-client GUI (Windows/Swing/fat-client), or HTML browser UI, responsible for displaying information and interpreting user commands into domain and data source actions; **Data source**, communicating with external systems (transaction monitors, other applications, messaging systems, databases) to persist and retrieve data; **Domain logic** (also called business logic), the application's core work involving calculations from inputs and stored data, data validation, and dispatching appropriate data source logic based on presentation commands.

Architectural variation is common: the domain layer may completely hide the data source from presentation, or presentation may directly access data stores. The latter, while less pure, often works better in practice—presentation interprets user commands, queries the data store for relevant data, then allows domain logic to manipulate that data before display. Single applications frequently have multiple instances of each layer: multiple presentations (rich-client and command-line interfaces), multiple data sources (different databases, packages), or partitioned domains.

A critical dimension emerges when considering systems without human users (Web services, batch processes, or programs driving the application): both presentation and data source layers represent interfaces to the outside world, similar in nature though inverted. Alistair Cockburn's Hexagonal Architecture visualizes systems as cores surrounded by symmetric external interfaces. Fowler argues for asymmetry—distinguishing between service interfaces you provide (presentation) and services you consume (data source)—because the difference in clients alters how you approach the service.

Practical separation depends on complexity. Simple scripts might keep all three layers in subroutines; moderate complexity justifies separate classes; high complexity warrants separate packages. The fundamental rule is to separate at *some* level.

A critical dependency constraint is that domain and data source must never depend on presentation—no subroutine calls from domain or data source code into presentation code. This enables substituting different presentations on the same foundation and modifying presentation without deep ramifications. The domain-data source relationship is more nuanced, depending on data source architectural patterns.

Identifying domain logic is challenging. Fowler offers an informal test: imagine adding a radically different presentation (command-line to a Web app). Any functionality requiring duplication indicates domain logic leaked into presentation. Similarly, would replacing a relational database with an XML file require duplicating logic? He illustrates with an example: a system highlighting products with sales increases over 10% month-to-month. Naive implementation puts comparison logic in presentation—when comparison exceeds 10%, set color to red. Correct separation places a domain method checking if a product has improving sales (returns Boolean), with presentation calling this method and highlighting accordingly if true. This separates the decision (is something highlightable?) from the presentation (how to highlight?). Fowler acknowledges discomfort with dogmatism—pure separation might be "the first step on a slippery slope to hell or a perfectly reasonable thing to do that only a dogmatic purist would object to."

**Choosing physical layer location** focuses on where to run processing—client desktop or server. Running everything on servers (via HTML front-end with Web browser) offers ease of upgrade, fix, deployment, and client compatibility management, though every user action requires server round-trips. Running on clients enables responsiveness and disconnected operation but demands network connectivity and widespread deployment/synchronization. The data source runs on servers except when duplicating functionality for disconnected clients requiring synchronization (out of scope here). Presentation location depends on interface type: rich clients run on client machines; Web interfaces run on servers; exceptions (remote client software, Web servers on desktops) are rare. B2C systems have no choice—all processing runs server-side with HTML output, accepting round-trip latency in exchange for universal browser compatibility.

Domain logic distribution choices are: all server (best for maintenance), all client (poor for upgrade/maintenance), or split (isolates small client pieces in self-contained modules). Splitting both locations requires isolating client logic from other system parts but complicates deployment. After choosing processing nodes, keep all code within a node in a single process or copied across cluster nodes; avoid separating layers into discrete processes without absolute necessity, as this degrades performance and introduces complexity (Remote Facades, Data Transfer Objects, network overhead).

Fowler identifies complexity boosters—distribution, explicit multithreading, paradigm chasms (object/relational), multiplatform development, and extreme performance requirements (100+ transactions/second)—that carry high development and maintenance costs. These should be adopted only when necessary.

## Chapter 2. Organizing Domain Logic

Transaction Script is the simplest approach, organizing domain logic as procedures where each script handles one user action. A single procedure takes input from presentation, validates and calculates using that input, stores data in the database, invokes operations on other systems, and returns results to presentation. The underlying organization is one procedure per business transaction—each action (checkout, add to cart, display status) gets its own script. Pieces can be separated into subroutines shared among different Transaction Scripts, but the driving force remains procedural. Advantages include simplicity that developers understand, compatibility with simple data source layers using Row Data Gateway or Table Data Gateway, and clear transaction boundaries that tools can manage automatically. However, as domain logic complexity grows, disadvantages emerge: duplicated code across scripts that's difficult to remove cleanly, tangled webs of routines without clear structure, and exponential difficulty in adding features.

Domain Model addresses complexity through object-oriented organization. Rather than procedures for each action, the Domain Model structures classes around domain nouns—lease, asset, shipment, contract, product. Logic for validations and calculations lives in the model objects themselves; for example, a shipment object contains logic to calculate shipping charges. Procedures may still exist (for calculating bills), but they quickly delegate to Domain Model methods. This represents the paradigm shift object-oriented development advocates: instead of one routine holding all logic for a user action, each object owns the logic relevant to it. The value becomes apparent with increasing complexity—adding new revenue recognition algorithms involves adding new strategy objects rather than adding conditional logic to scripts. The costs are substantial: developers new to rich object models need time to adapt (often several months of projects), and the data source layer becomes significantly more complex. Mapping richer Domain Models to relational databases typically requires sophisticated data source layers using Data Mapper patterns. This creates a fixed cost—substantial effort upfront—but once in place, it enables handling increasing complexity well. Fowler notes that once developers adapt to Domain Models, they typically become "infected for life," though a significant minority of developers struggle with the paradigm shift.

Table Module sits conceptually between Transaction Script and Domain Model. While it appears similar at first glance—classes for contracts, products, revenue recognitions—the critical difference is cardinality: Domain Model has one instance per database record, while Table Module has only one instance per table. Table Module is designed to work with Record Sets. A client queries the database to form a Record Set, creates a Table Module object, and passes the Record Set as argument. The client then invokes operations on the module, passing IDs when operating on individual records. This provides more structure than Transaction Script (organizing logic around tables rather than straight procedures, making duplication easier to find and remove) but sacrifices fine-grained OO techniques like inheritance, strategies, and other OO patterns that Domain Model uses. Table Module's biggest advantage is architectural fit: many GUI environments are built to work on SQL query results organized in Record Sets. Since Table Module also works with Record Sets, you can query, manipulate in the module, and pass results directly to GUI for display. This style is particularly attractive in environments like Microsoft COM and .NET with extensive Record Set tooling. Without such tooling, Table Module offers little advantage.

Choosing between patterns depends on domain logic complexity. Fowler presents a non-scientific graph showing effort versus complexity: with simple logic, Domain Model's learning cost and data source complexity make it unattractive. As domain complexity increases, other approaches hit walls where adding features becomes exponentially difficult, while Domain Model scales better. The challenge is measuring domain complexity—Fowler notes humorously that you should use Domain Model when complexity exceeds 7.42 (unquantifiable), so in practice, experienced teams must make judgment calls. Team familiarity lowers initial Domain Model costs (though not to Transaction Script levels due to data source complexity). Team experience should bias toward Domain Model. Table Module's attractiveness depends heavily on Record Set support in your environment—attractive with .NET or Visual Studio tooling, not worth considering without it. Fowler sees no reason to use Transaction Script in .NET environments. Refactoring paths matter: starting with Transaction Script and refactoring to Domain Model is straightforward; starting with Domain Model and reverting to Transaction Script is usually not worthwhile unless you can simplify the data source layer. These patterns are not mutually exclusive; many applications mix Transaction Script for some logic with Table Module or Domain Model for others.

Service Layer patterns place a layer over Domain Model or Table Module (rarely Transaction Script alone, which isn't complex enough to warrant separation). The presentation layer interacts with the domain purely through the Service Layer API. Service Layer also provides a convenient point for transaction control and security characteristics, often specified in a separate properties file or through language attributes. A key decision is behavior distribution. Minimal Service Layer acts as facade—all behavior in underlying objects, Service Layer just forwards calls, providing use-case-oriented APIs that are easier to consume while serving as transactional wrapper and security checkpoint. The opposite extreme puts most business logic in Transaction Scripts inside the Service Layer, with simple underlying objects. If Domain Model is used, it's one-to-one with database structure, allowing simpler data source layers like Active Record. A middle approach, controller-entity style, places transaction or use-case-specific logic in use-case controllers (Transaction Scripts), while behavior used in multiple use cases goes on domain entities. Fowler dislikes the controller-entity approach because use-case controllers encourage duplication; his view is that if you commit to Domain Model, you should make it dominant. An exception exists: if you've started with Transaction Script and Row Data Gateway, moving duplicated behavior to Row Data Gateways (making them simple Domain Models with Active Record) makes sense. However, starting that way is not recommended. Fowler's preference is the thinnest Service Layer possible, assuming you don't need one and adding it only when the application requires it. Many designers use Service Layers with substantial logic successfully, so this represents architect preference rather than absolute guidance.

## Chapter 3. Mapping to Relational Databases

The chapter opens by establishing relational databases as the dominant persistence mechanism and SQL as the standard interface. The core architectural decision involves whether to separate SQL from domain logic through gateway patterns—specifically, organizing classes around table structure rather than embedding SQL throughout the application.

Two primary gateway approaches exist. Row Data Gateway creates one instance per database row, naturally aligning with object-oriented thinking where each object represents a row. The alternative, Table Data Gateway, creates a single class per table that returns generic Record Sets, reducing mapping overhead and naturally fitting with Table Module domain logic. For simple applications with straightforward domain logic, Row Data Gateway or Table Data Gateway combined with Transaction Script or Table Module provides clear separation without excessive indirection.

However, when using Domain Model architecture with moderate business logic complexity, Active Record becomes appropriate. Each domain object handles its own loading and saving, eliminating gateway indirection. This works well when domain classes correspond one-to-one with database tables. Yet as domain logic grows richer, Active Record breaks down: the one-to-one class-to-table correspondence fails as developers factor logic into smaller classes; inheritance proves difficult without proper relational support; testing becomes harder without database isolation.

This complexity drives architects toward Data Mapper, the most sophisticated approach. Data Mapper completely isolates domain objects from database schema, with a dedicated mapping layer handling all loading and storage. This enables both domain model and database to vary independently, achieving complete isolation at the cost of implementation complexity.

The behavioral problem addresses tracking which objects have been modified and ensuring proper concurrency isolation. Unit of Work solves this by acting as a database mapping controller, keeping track of all objects read and modified, then sequencing all updates when commit is invoked. This centralizes complex commit logic in one place. Additionally, Identity Map maintains a record of every row read, preventing duplicate in-memory objects for single database rows and serving as an incidental cache. Lazy Load uses placeholders for object references, fetching related data only when actually accessed, preventing excessive object graph loading.

For reading data, finder methods wrap SQL select statements with method-structured interfaces like `find(id)` or `findForCustomer(customer)`. Finder placement depends on the pattern used: table-based patterns combine finders with insert/update, while row-based patterns require separate finder classes. Critically, finder methods operate on database state, not in-memory object state. Performance demands pulling multiple rows in single queries, using joins to fetch multiple tables, but never exceeding three or four joins per query. Database profiling is essential since general rules have surprising exceptions.

Structural mapping patterns handle the impedance mismatch between objects and relations. Objects use memory references; relations use foreign keys. Objects use collections; relations force single-valued references. Identity Field stores relational identity in objects as a lookup mechanism. Foreign Key Mapping translates between object references and relational keys using Identity Map as a lookup table. For collections, this requires issuing queries to find related rows, creating and adding objects to collections, tracking additions/removals—tedious work that metadata approaches simplify. Dependent Mapping reduces complexity for collections owned exclusively by their parent. Association Table Mapping handles many-to-many relationships through intermediate relational tables.

Value Objects like dates and money shouldn't become separate database tables. Instead, Embedded Value embeds all their fields into the owning object's row. Serialized LOB takes this further, storing entire object clusters as XML or binary data in single columns, useful for hierarchic structures (org charts, bills of materials) where fine-grained relational design causes excessive roundtrips, though at the cost of losing SQL querying ability.

Inheritance mapping presents three options: Single Table Inheritance stores all hierarchy classes in one table (simple modification, wastes space through empty columns, avoids joins but bottlenecks on table size); Concrete Table Inheritance uses one table per concrete class (avoids joins, brittle to superclass changes, complicates key management); Class Table Inheritance uses one table per class (simplest class-table relationship, requires multiple joins, reduces performance). The trade-off centers on data structure duplication versus access speed. Fowler favors Single Table Inheritance for resilience to refactoring. Multiple inheritance is handled through interface/superclass variations of these same three patterns.

Building mappings differs based on schema control. When designing schema for simple Transaction Script or Table Module logic, use Row Data Gateway or Table Data Gateway with classic database design. For Domain Model, design the model first without regard to database, then treat database as persistence mechanism—Data Mapper offers maximum flexibility but complexity, while Active Record works if isomorphic design makes sense. Within iterative cycles (six weeks maximum per iteration), integrate database design with each Domain Model piece rather than building a six-month database-free model.

With existing fixed schemas and simple logic, build Row/Table Data Gateway classes mimicking the database with domain logic layered on top. With complex logic, gradually build Domain Model plus Data Mappers for the existing database.

Double Mapping addresses pulling similar data from multiple sources with schema differences. The solution uses two-step mapping: first converting in-memory schema to logical data store schema (maximizing similarities), then mapping logical schema to physical schema (capturing differences). This overhead only pays off with significant commonalities.

Metadata Mapping eliminates repetitive handwritten code by boiling mapping into metadata files detailing column-to-field mappings. One metadata line can define read/write code, auto-generate joins, generate SQL, enforce relationship multiplicity, and compute write orders under referential integrity constraints. Query Object builds queries in object terms without requiring SQL or relational schema knowledge. Repository hides the database entirely, supporting both in-memory and database-retrieved objects equally. This foundation, while sophisticated, requires commercial O/R tools to be truly practical.

Database connection management ties heavily to transactions. Connections are expensive resources requiring closure after use. Two approaches exist: explicit parameter passing (tedious, clutters call stacks) or thread-scoped Registry pattern. Garbage collection offers backup but lacks timeliness—connections may linger. The best practice binds connections to transactions: open connection at transaction start, close at commit/rollback, with Unit of Work managing both. Fresh connections suit immutable data reads. Disconnected Record Sets allow opening a connection for loading data, closing for manipulation, then reopening for persistence, requiring concurrency control awareness.

Minor points address `select *` dangers (breaks with column reordering), the importance of column name indices for robustness, preference for static precompiled SQL over dynamic concatenated queries, batch query capability across platforms, and comprehensive create/read/update/delete test coverage for each database mapping structure to prevent SQL-code synchronization failures.

## Chapter 4. Web Presentation

Web applications have risen as the dominant presentation mode for enterprise systems, bringing advantages of no client installation and universal access. The fundamental architectural challenge is deciding how to structure server-side code. Scripts (CGI, Java servlets) are programs with methods handling HTTP calls; they parse request objects (via regular expression or keyword interfaces) and write responses as strings. This approach provides full programming power but makes writing HTML responses uncomfortable for programmers and nearly impossible for non-programmers accustomed to HTML editing. Server pages (PHP, ASP, JSP) invert this: the structure is the HTML page itself, with scriptlets of code embedded at execution points. This works well for simple responses ("show album #1234") but becomes messy with conditional logic requiring different display formats. Fowler's solution combines both: use scripts for request interpretation and server pages for response formatting, manifested through the Model-View-Controller pattern. MVC's input controller handles requests and invokes model objects to execute domain logic and gather response data; the model communicates with data sources and returns results; the controller then selects an appropriate view and forwards control with response data, often via an HTTP session object. The primary benefit of MVC is complete separation of models from Web presentation, enabling easier modification, additional presentations, and superior testability—especially critical when views are server pages prone to embedded logic. Fowler distinguishes input controllers from Application Controllers, a different layer handling application flow and screen sequencing. Application Controllers decide which screens appear in which order and may be presentation-independent to enable reuse across different UIs (beneficial for systems with identical navigation but different presentation), though different presentations often benefit from different flows. Application Controllers are necessary only when the machine controls screen flow; user-directed navigation requires none. The test: if flow logic exists, use Application Controller; if users can access any screen in any order, avoid it. On the view side, three patterns address rendering: Template View structures presentation around the page, embedding markers for dynamic content (ASP, JSP, PHP implementations), providing power and flexibility but risking messy, difficult-to-maintain code unless strict discipline prevents logic from pages, often using helper objects. Transform View uses transformation style (typically XSLT), effective when domain data is in XML format or easily converted to it; the input controller selects an XSLT stylesheet and applies it to XML gleaned from the model. Procedural scripts can be written in either Transform View or Template View style or an interesting mix. Single-stage views maintain one view component per screen, taking domain-oriented data and rendering HTML directly; similar logical screens may share views. Two Step View breaks this into two stages: first-stage views produce logical screens from domain data (one per screen), and a single second-stage view renders all logical screens as HTML. Two Step View's advantage is centralizing HTML rendering decisions in one place, enabling global presentation changes by altering a single object—effective for sites where different screens share basic layout. Two Step View excels with Web services used by multiple front-end customers (like multiple airlines using the same reservation system backend), where each front end differs in appearance but shares the same logical screen structure. It also handles multiple output devices (Web browsers, palmtops) with separate second stages, limited by the requirement that logical screens be shared, which may fail for vastly different UIs like browser and cell phone. For input controllers, two patterns predominate: Page Controller creates one input controller object per page, sometimes implemented as a server page combining view and controller roles, though separate objects often simplify implementation. The mapping isn't strictly one-to-one per page; rather, one Page Controller handles each action (button or link), and while actions usually map to pages, occasionally they map to multiple pages conditionally. Input controllers carry two responsibilities—handling HTTP requests and deciding what to do—making separation beneficial. A server page can handle requests while delegating to a helper object for decision logic. Front Controller centralizes further by having a single object handle all requests, interpreting the URL to determine request type and creating separate objects to process each; this avoids reconfiguring the Web server when the site's action structure changes and concentrates all HTTP handling in one location.

## Chapter 5. Concurrency

Concurrency Fundamentals. Concurrency creates two essential problems: lost updates (one user's changes overwrite another's because a second reader finishes later) and inconsistent reads (data read across multiple points in time is internally inconsistent—no single point in the past or future where both values coexisted). These failures of correctness (safety) are unavoidable without concurrency controls, but any control mechanism reduces liveness (concurrent activity). Fowler and Rice stress that you cannot simply choose correctness over liveness; you must accept tolerable failure rates to enable concurrent work.

Execution Contexts. Enterprise systems operate in multiple overlapping contexts. A request is a single external call requiring response; a session is a long-running interaction between client and server, typically consisting of multiple requests. Processes are heavyweight isolated execution contexts; threads are lightweight and share memory, creating concurrency hazards. Transactions are contexts that aggregate multiple operations to be treated atomically. Early systems often used process-per-request, later systems adopted thread-per-request despite synchronization burdens. The authors note that even with process-per-session ideal isolation, modern systems rarely implement it due to resource costs.

Isolation and Immutability. One solution partitions data so only one active agent accesses each piece—isolation at the operating system level with processes, file locks in productivity applications. This is vital because it creates zones where programmers need not worry about concurrency. The alternative is recognizing immutable data; while you cannot make all data immutable, identifying stable reference data or read-only views allows sharing without concurrency controls.

Optimistic Versus Pessimistic Locking. These represent the two fundamental strategies. With optimistic locking, multiple users edit copies and conflicts are detected only at commit time—David finishes first and commits cleanly; Martin discovers conflict when committing, must resolve, and retry. With pessimistic locking, whoever checks out first holds the lock—Martin locks the file, David waits until Martin commits. Optimistic locking is fundamentally about conflict detection; pessimistic locking about conflict prevention. Pessimistic locks reduce concurrency severely (frustrated users waiting for locks); optimistic locks avoid lock waits but force users to merge or restart after conflicts. The choice depends on conflict frequency and severity: rare conflicts favor optimistic (better liveness, simpler to implement); painful conflicts favor pessimistic (users discover problems early). Neither is free of secondary problems.

Preventing Inconsistent Reads and Version Markers. Inconsistent reads remain problematic even under locking. Pessimistic systems address this through read locks (shared) and write locks (exclusive): multiple readers coexist, but any writer blocks all access. Optimistic systems use version markers (timestamps or counters) comparing read and written data versions—if a read version differs from the shared version, conflict is signaled. However, controlling every read's version is burdensome. The authors recommend separating data truly "used" (like charges in a bill calculation) from merely "read" (like a product list where new entries are acceptable). Temporal Reads—returning data as it existed at a specified timestamp—offer another approach but require full temporal history (expensive in databases).

Deadlocks. Pessimistic systems face deadlock: Martin locks Customer and needs Order; David locks Order and needs Customer—neither proceeds. Solutions include victim selection (one transaction rolls back), timeouts (locks expire, victim loses work), lock ordering (all acquire locks alphabetically), or automatic victim making (simple, may victimize innocent sessions). The authors stress that deadlock-proof schemes are hard to verify; they recommend simple, conservative approaches even if they cause unnecessary victims, better than missing deadlock scenarios.

ACID Properties and Transactional Resources. The term "transaction" evokes financial exchanges—bounded work with start and end, resources in consistent states before and after, all-or-nothing completion. System transactions enforce ACID: Atomicity (all-or-nothing within transaction boundaries), Consistency (resources remain noncorrupt), Isolation (no partial visibility to other transactions until commit), Durability (committed data survives crashes). While databases are the primary transactional resource, message queues, printers, and ATMs can be transactional. Modern systems minimize transaction duration for throughput. Request transactions start at request entry and end at response; late transactions delay opening until updates, reducing contention time but leaving reads uncontrolled. Lock escalation—when row locks exceed database limits and elevate to table locks—severely damages concurrency and should be avoided (particularly for Layer Supertype tables holding all object types).

Isolation Levels and Trade-offs. SQL defines four isolation levels trading correctness for liveness. Serializable is strongest: concurrent execution produces results equivalent to serial execution (Martin's count is either 12 or 17, never invalid 15). Repeatable Read allows phantoms (inserted rows Martin didn't see before). Read Committed allows unrepeatable reads (Martin reads 7 for locking, David commits, Martin reads 9 for multiphase—if he rechecks locking he sees 9, not repeatable). Read Uncommitted allows dirty reads (Martin sees David's uncommitted changes, which may roll back). The authors note that serializable truly messes up liveness; developers must choose for each transaction based on acceptable risk, deciding what errors versus performance trade-off they can tolerate.

Business and System Transactions. System transactions are RDBMS-managed SQL command sequences. Business transactions are user-visible operations (online banking: login, select account, set payments, click OK). A business transaction ideally displays ACID properties, but implementing it as a single system transaction creates a long transaction, which databases handle inefficiently and which limits scalability. Breaking business transactions into multiple short system transactions leaves concurrency control to the application—offline concurrency. Between system transactions the application must ensure isolation and consistency using patterns like Optimistic Offline Lock or Pessimistic Offline Lock. Domain Model with Unit of Work patterns help track changes across requests; Transaction Script requires manual change tracking. Isolation between business transactions is the critical challenge—preventing one user's changes from corrupting another's and avoiding both lost updates and inconsistent reads.

Offline Concurrency Control Patterns. Enterprise applications often cannot fit business transactions into single system transactions or sustain long transactions at scale. Fowler and Rice recommend Optimistic Offline Lock as first choice for its ease of programming and best liveness, accepting late discovery of failures. When late failure discovery is unacceptable (users frustrated after hours of work), Pessimistic Offline Lock provides early detection but is harder to program and reduces liveness. Coarse-Grained Lock simplifies concurrency management by grouping objects so locks apply to groups rather than individuals. Implicit Lock saves developers from manual lock management, avoiding bugs when developers forget—concurrency bugs are notoriously hard to find. The choice of optimistic versus pessimistic affects the entire user experience and demands domain knowledge about conflict likelihood and user expectations.

Application Server Concurrency. Separate from transaction-based concurrency, application servers must handle multiple simultaneous requests. Explicit multithreading is dangerous—concurrency bugs are nearly impossible to reproduce, resulting in systems that work 99% of the time but crash randomly, frustrating users. The authors advocate avoiding explicit synchronization and locks. Process-per-session isolates each session completely—application programmers ignore multithreading. Process-per-request (pooled processes, each handling one request at a time) uses fewer resources while maintaining isolation; developers must ensure resources are released at request end. Thread-per-request runs multiple threads in a single process—most efficient resource utilization but no isolation between threads; any thread can access any data. The authors prefer process-per-request despite its inefficiency over thread-per-request, citing better robustness and fewer threading headaches, especially for less experienced teams. When using thread-per-request, create per-request object instances kept out of static variables or singletons, establishing isolated zones where multithreading concerns disappear. The Registry pattern provides thread-local storage appearing like global memory. Expensive objects like database connections may be pooled with synchronized access.

## Chapter 6. Session State

Fowler begins by clarifying terminology: a "stateless server" in distributed enterprise applications doesn't mean an object with no fields, but rather one that doesn't retain state between requests. While objects logically combine state and behavior, a stateless server maintains undefined field values across invocations. The example of a Web server returning book information by ISBN illustrates statelessness: the server object temporarily holds ISBN, title, and price during request processing, then discards these values; the next request is completely independent. 

The primary argument for statelessness centers on resource efficiency. Because stateless servers don't maintain state between requests, any available object can service the next request. This enables object pooling: a system with 100 concurrent users making one request every ten seconds, where each request takes one second to process, needs only 10 stateless server objects instead of 100 stateful ones (one per user). The idle time multiplication makes statelessness extremely valuable on high-traffic Web sites, and the pattern aligns naturally with HTTP's stateless protocol design.

However, Fowler emphasizes that many business interactions are inherently stateful. The shopping cart metaphor exemplifies this: a user browses multiple books and maintains a list of selections across multiple requests. The session state—the shopping cart contents—must persist between requests and remain isolated from other sessions until the business transaction commits. Session state is conceptually distinct from record data, the long-term persistent information visible across all sessions in the database. Crucially, session state belongs to a business transaction (which may span many system-level interactions) and exhibits ACID properties: atomicity, consistency, isolation, and durability requirements.

A critical design consequence is that session state need not obey normal consistency rules while being edited. An insurance policy being modified may temporarily violate validation rules; it achieves valid state only upon transaction commit. This is fundamentally different from record data, which must always be valid. Additionally, session isolation becomes complex: while one user edits a policy and views the associated customer record (which supplies a zip code affecting the policy's risk value), another user may modify the customer record, causing the editing user to read inconsistent data.

Fowler distinguishes session state from session cache: cache data improves performance but isn't required for correctness, whereas session state must persist between requests for the application to function correctly.

The chapter presents three storage approaches with nuanced trade-offs:

Client Session State stores data on the client via URL encoding, cookies, hidden form fields, or rich client objects. Advantages: eliminates bandwidth for idle sessions, gracefully handles user abandonment (state disappears with the client), survives server failures. Disadvantages: every request must transfer all relevant session data across the network (becoming problematic for large state—the author cites an example where session data approached one megabyte), creating security vulnerabilities if data isn't encrypted (allowing malicious users to edit session values), and requiring careful transformation of data to forms suitable for client transmission.

Server Session State maintains data in memory between requests or persists it to durable storage (the application server's local file system or a shared data source like a simple database table keyed by session ID). Immediate availability makes responsiveness excellent; Fowler notes this pattern is usually simplest for development. However, server affinity becomes necessary: the same server must handle all requests for a session, preventing load balancing and creating clustering complications. When a load balancer can't inspect calls to determine which session they belong to, it falls back to client IP address-based affinity, which becomes problematic behind corporate proxies where many clients share a single IP (e.g., AOL traffic concentrating on one server). Clustering and session migration create complexity: session migration allows sessions to move between servers for better load distribution, but retrieving session state from only the machine holding it becomes difficult.

Database Session State breaks up data into tables and fields, storing it like record data. Advantages: survives server crashes, client crashes, and network failures; enables server migration more naturally; handles large numbers of mostly-idle users well (like public retail systems). Disadvantages: requires database round-trips to retrieve state, causing latency increases; introduces significant complexity in isolating session data from record data; and demands additional development effort to maintain this separation.

The chapter examines session lifecycle management. User cancellation is straightforward with Client Session State but requires explicit cleanup and timeout mechanisms with other approaches. System failures affect each pattern differently: Database Session State handles all failure modes well; Server Session State's survival depends on whether backups exist; Client Session State survives server and network failures but not client crashes.

Fowler stresses that development effort varies significantly. Server Session State typically requires least effort, especially if persistence isn't necessary. Database Session State and Client Session State demand transformation code to convert between storage formats and usable object forms. Database Session State adds overhead in maintaining isolation from other database uses.

Importantly, the patterns aren't mutually exclusive—mixed approaches are common. However, non-Client approaches must maintain a session identifier in Client Session State to identify which session each request belongs to.

Fowler concludes with his preferences: Server Session State is preferred when the session memento is stored remotely for crash recovery; Client Session State is suitable for session IDs and small session data; Database Session State should be reserved for scenarios requiring failover, clustering, and when remote session storage isn't possible or session isolation isn't a concern.

## Chapter 7. Distribution Strategies

Fowler opens by recounting a recurring design pattern he observes: architects distribute objects by placing different classes (customers, orders, products, deliveries) on separate processing nodes for load-balancing and performance. However, vendors' middleware promises of transparent distribution obscure a fundamental truth—while transparency is valuable, performance transparency is illusory.

The primary constraint is architectural physics: local procedure calls within a single process are extremely fast, while inter-process calls are orders of magnitude slower, and cross-machine calls slower still. This disparity fundamentally shapes the required interface design. Local interfaces should be fine-grained with many small methods (separate getters/setters for city, state, zip on an address class), enabling flexibility and composability. Remote interfaces must be coarse-grained (e.g., getAddressDetails, updateAddressDetails) to minimize call overhead. This creates a dilemma: any class potentially used remotely must adopt the coarse-grained model, forcing an awkward programming model across the system and inevitably creating too many remote calls regardless.

The solution is clustering: place all classes in a single process and run multiple copies of that process on different nodes. Each process uses fast local calls internally and can employ fine-grained interfaces everywhere, improving maintainability. However, distribution boundaries are unavoidable in certain cases: client-server separation (desktop PCs versus shared data repositories), application-server-to-database separation, web-server-to-application-server splits, vendor package integration, and occasionally genuine application-logic splits. Fowler emphasizes being "parsimonious with object distribution"—fighting hard to eliminate distribution points whenever possible.

When distribution boundaries are necessary, use coarse-grained facade objects at those boundaries. The Remote Facade pattern places coarse-grained objects whose sole role is to delegate to fine-grained internal objects, making the distribution cost explicit to developers rather than hidden by transparency. Paired with this is the Data Transfer Object pattern: rather than sending domain objects (which contain fine-grained inter-object references), bundle the needed data into transfer objects that reference only other transfer objects and fundamental types like strings. This prevents clients from navigating object webs across the wire.

For interface mechanisms, traditional remote procedure call approaches contrast with XML-over-HTTP interfaces like SOAP. XML provides advantages for heterogeneous platforms, easy firewall passage, and human-readable wire format. However, XML adds processing overhead; if both sides use the same binary platform, direct RPC mechanisms are more efficient. Layering HTTP interfaces over object-oriented interfaces is possible but adds complexity and should only be done if both HTTP and OO APIs are needed or if security/transaction handling benefits justify it. Fowler notes that while his discussion assumes synchronous RPC-style interfaces, he increasingly favors asynchronous message-based approaches for distributed systems, including for web services.

## Chapter 8. Putting It All Together

The chapter opens with epistemic humility: Fowler acknowledges that architecture advice without full project knowledge is dangerous, and readers know their systems better than he does. He recommends three technical practices to mitigate the cost of architectural refactoring: continuous integration, test-driven development, and refactoring itself. These practices reduce organizational lock-in to architectural choices.

The central decision is domain layer selection, driven primarily by business logic complexity—a variable impossible to quantify precisely, though database connectivity difficulty also factors in. Fowler describes three main patterns:

Transaction Script represents the simplest approach, following a procedural model most developers find comfortable. It encapsulates each system transaction in a comprehensible script and integrates cleanly with relational databases. However, it handles complex business logic poorly and is susceptible to duplicate code. It works well for simple applications like basic catalog systems with shopping carts and straightforward pricing structures, but difficulties multiply exponentially as logic grows complex.

Domain Model stands at the opposite extreme. Fowler, identifying as an "object bigot," argues that complex domain logic is best handled by a rich Domain Model, and once developers become comfortable with domain models, even simple problems are tackled with ease. However, Domain Model has significant drawbacks: it requires real skill to implement well (done poorly it's a disaster), and its connection to relational databases is messy. While object database solutions exist, they're rarely practical in enterprise contexts. The complexity of O/R mapping patterns emerges from the poor fit between object and relational models.

Table Module represents an attractive middle ground. It handles domain logic better than Transaction Scripts without requiring the sophistication and O/R mapping complexity of Domain Models. Importantly, it fits extremely well with relational databases and thrives in environments like .NET, where tools orbit around Record Sets. If a platform provides robust Record Set frameworks, Table Module becomes particularly strong, leveraging the platform's strengths while representing reasonable domain logic factoring.

Fowler emphasizes a crucial insight: tools often constrain architecture as much as architecture choices determine tool selection. In practice, developers frequently must match architecture to available tools rather than choosing tools based on architectural ideals. Table Module's influence rises most when tools support it well, particularly in .NET environments where the platform is geared around Record Sets.

Data source layer decisions cascade from domain layer choices. For Transaction Scripts, Fowler recommends separating database logic from script logic even in simple applications. The choice between Row Data Gateway and Table Data Gateway depends on platform facilities and expected evolution. Row Data Gateway presents records as objects with clear, explicit interfaces requiring accessor code. Table Data Gateway reduces code volume by relying on implicit interfaces that access record set structures like maps. Platform Record Set tools tip the balance decisively toward Table Data Gateway.

Transaction Scripts rarely need complex O/R mapping patterns. Structural mapping issues are mostly absent since in-memory structures map well to database structures. Unit of Work is usually unnecessary since scripts easily track changes. Concurrency handling is simplified because scripts often correspond exactly to system transactions, allowing wrapping the entire script in a single transaction. The common exception occurs when one request retrieves data for editing while a later request saves changes—here Optimistic Offline Lock is almost always best, being easier to implement while matching user expectations and avoiding hanging sessions that lock resources.

For Table Module, the primary driver is the presence of good Record Set frameworks. Table Data Gateway becomes the natural pairing, the two fitting together almost perfectly. Record Sets may include built-in concurrency control, effectively implementing Unit of Work and reducing implementation burden.

Domain Model data source patterns depend on model complexity. Simple models with few classes closely matching the database structure can use Active Record. For modest decoupling, either Table Data Gateway or Row Data Gateway work, with the choice being relatively neutral. As complexity grows, Data Mapper becomes necessary, delivering on the promise of keeping Domain Model independent from other layers. However, Data Mapper is the most complicated pattern to implement. Unless the team is very strong or simplifications can ease mapping, Fowler strongly recommends acquiring a mapping tool. Once Data Mapper is chosen, most O/R mapping patterns come into play, and Fowler heartily recommends Unit of Work as a focal point for concurrency control.

The presentation layer is largely independent of lower-layer choices. The first decision is rich-client versus HTML browser interface. Rich clients offer better UI but require more control and deployment effort. Fowler prefers HTML when feasible and rich clients only when necessary, noting that apparent extra effort comes from sophistication rather than inherent technology complexity.

For HTML applications, Fowler recommends Model View Controller as the design underpinning. Within MVC, two decisions follow: controller pattern and view pattern. Tooling often makes these choices. Visual Studio developers most easily implement Page Controller and Template View. Java developers with frameworks like Struts arrive at Front Controller and Template View. Given freer choice, Fowler recommends Page Controller for document-oriented sites (especially those mixing static and dynamic pages) and Front Controller for sites with complex navigation and UI. For views, the choice between Template View and Transform View depends on whether teams use server pages or XSLT. Template Views currently dominate, though Transform View offers better testability. For sites needing consistent appearance with multiple looks and feels, Two Step View should be considered.

Communication with lower layers depends on whether they reside in the same process. Fowler prefers single-process architectures to avoid slow inter-process calls. When separation is necessary, Remote Facade wraps the domain layer, and Data Transfer Object handles communication with the Web server.

The chapter then applies patterns to specific technology platforms. In J2EE, the ongoing debate concerns Enterprise JavaBeans' value. Fowler notes that good J2EE applications don't require EJB, and excellent work happens with POJOs (plain old Java objects) and JDBC. Design alternatives vary by domain logic pattern. For Transaction Scripts with Gateways, the common EJB approach uses session beans as Transaction Scripts and entity beans as Row Data Gateways—reasonable for modest domain logic. However, this approach makes removing the EJB server difficult if not needed. The non-EJB alternative is POJOs for Transaction Scripts on top of either Row Data Gateway or Table Data Gateway. If JDBC 2.0 row sets gain acceptance as Record Sets, they lead toward Table Data Gateway.

For Domain Models in J2EE, current orthodoxy favors entity beans. For simple models matching the database well, entity beans work as Active Records, with session beans wrapping them as Remote Facades. If the Domain Model is complex, Fowler advocates keeping it entirely independent of EJB structure—POJOs for the Domain Model wrapped with session beans as Remote Facades, allowing write, run, and test of domain logic without EJB container vagaries. Alternatively, skip EJB entirely and run the whole application in the Web server. For complex POJO Domain Models, Fowler recommends POJOs for Data Mappers, using either O/R mapping tools or rolling custom solutions.

Fowler advises against giving entity beans remote interfaces, which he finds pointless. Entity beans typically serve as Domain Models or Row Data Gateways—both needing fine-grained interfaces—but remote interfaces must be coarse-grained. He keeps entity beans local only. Table Module is uncommon in the Java world but could gain viability if more tooling surrounds JDBC row sets. The POJO approach fits best there, though Table Modules can be wrapped with session beans as Remote Facades returning Record Sets.

In .NET, the dominant pattern is Table Module. Although object-oriented purists dismiss this, Table Module presents genuine value as compromise between Transaction Script and Domain Model, with impressive tools exploiting ubiquitous data sets as Record Sets. Table Module must therefore be the default choice for .NET. Even Transaction Scripts should rarely be used except in the simplest cases, and even then should operate on and return data sets. Domain Models are buildable in .NET as easily as in other object-oriented environments, but platform tools don't provide the extra help they do for Table Modules, so Fowler tolerates more complexity before shifting to Domain Model.

On stored procedures, debate is substantial. They're often the fastest approach, running in the same process as the database, reducing remote calls. However, stored procedure environments typically lack good structuring mechanisms, and stored procedures lock applications into particular database vendors. Oracle's approach of running Java inside the database process is interesting but still leaves vendor lockin. For modularity and portability, many developers avoid stored procedures for business logic. Fowler sides with this view unless strong performance gains justify it, in which case he moves clear performance-problem methods from the domain layer into stored procedures, treating this as optimization rather than architectural principle.

Web services are presented as tools for application integration rather than application construction. Fowler warns against breaking a single application into Web services that communicate with each other unless truly necessary. Instead, build the application and expose parts as Web services, treating those as Remote Facades. He emphasizes not forgetting the First Law of Distributed Object Design. While published examples show synchronous Web services resembling XML RPC, Fowler prefers asynchronous, message-based approaches, though he has no patterns for these yet.

The chapter concludes by comparing Fowler's three-layer approach against alternative layering schemes. The Brown model uses five layers: presentation, controller/mediator, domain, data mapping, and data source, placing mediating layers between the basic three. Fowler finds mediating layers useful sometimes but not always, describing them as patterns (Application Controller mediates between presentation and domain; Data Mapper mediates between domain and data source). He recommends adding mediating layers only when base layers get too complex.

CoreJ2EE patterns define five layers: client, presentation, business, integration, and resource. The resource layer comprises external services; the main difference is splitting the presentation layer between client and server sides. This split is often useful but not mandatory.

Microsoft DNA defines three layers: presentation, business, and data access, corresponding directly to Fowler's three layers. All layers operate on record sets from SQL queries, creating apparent coupling where business and presentation layers know about the database. Fowler interprets this as the record set serving as Data Transfer Object between layers. While unwieldy, this approach enables presentation layers to use data-aware GUI controls, even on business-layer-modified data. The domain layer is structured as Table Modules; the data source uses Table Data Gateways.

Marinescu proposes five layers: presentation is split into two (reflecting Application Controller separation), and domain is split into Service Layer built on Domain Model. This approach is common, reinforced by EJB limitations as Domain Model. The Service Layer separation distinguishes workflow logic (particular to single use cases and infrastructure communication) from pure domain logic. Whether to maintain separate services and domain layers is debated. Fowler views it as occasionally useful rather than mandatory, though designers he respects disagree.

Nilsson uses one of the more complex schemes with extensive stored procedures, including domain logic in stored procedures for performance. Fowler finds this uncomfortable as it complicates maintenance (occasional use as optimization is acceptable). Like Marinescu, he separates application and domain layers for domain logic. Both suggest skipping domain layer in small systems, aligning with Fowler's view that Domain Model is less worthwhile for smaller systems.

## Chapter 9. Domain Logic Patterns

Transaction Script organizes business logic as procedures where each handles a single request from presentation, making calls directly to the database or through thin wrappers. For simple cases, it requires minimal overhead in both performance and understanding. The pattern groups scripts by subject area into classes or optionally as separate Command pattern instances, one script per class. Key benefit is simplicity and natural fit for applications with small logic volumes. However, duplication emerges across transactions since each must be self-contained, making careful factoring necessary but eventually insufficient for complex domains. Scripts should be separated as much as possible from presentation and data source layers to ease modification and testing.

Domain Model creates an object-oriented layer incorporating both data and behavior, clustering processes close to the data they operate on. The pattern exists in two styles: simple Domain Models resembling database designs with one object per table, and rich Domain Models using inheritance, strategies, and other Gang of Four patterns with complex interconnected objects. Rich models better handle complex logic but require harder database mapping via Data Mapper pattern, while simple models can use Active Record. Critical design consideration is minimizing coupling to other system layers since business behavior changes frequently. Object graph scope varies from single-user (entire graph in memory) to multiuser systems requiring selective loading of relevant objects. The pattern addresses bloated domain objects by recommending retention of usage-specific behavior in the natural object fit rather than separation, observing that duplication costs exceed bloating risks in practice. Java-specific discussions note that entity beans with Container Managed Persistence are limited for rich Domain Models due to re-entrancy restrictions and fine-grained interface penalties on remote objects. POJO (plain old Java object) domain models with Data Mapper pattern provide better flexibility for complex logic while remaining independent of EJB infrastructure.

Table Module organizes domain logic with one class per database table, using a single instance to contain procedures acting on all rows. Primary distinction from Domain Model is one object handling all table rows rather than one object per row. Pattern packages data and behavior together while leveraging relational database strengths, particularly when using table-oriented backing data structures (typically SQL ResultSets held in Record Sets). Methods operate without object identity, requiring identity references (usually primary keys) to be passed for each operation. Multiple Table Modules frequently collaborate on same Record Set, and the pattern can be applied to database views and queries not just physical tables. Implementations use either instance (allowing initialization with existing ResultSets and enabling inheritance) or static methods. When using Table Data Gateway for queries, the application assembles data in a Record Set, creates Table Module with that Record Set, then Table Module performs business logic before passing modified Record Set to presentation. Testing benefits occur by creating in-memory Record Sets without database access. Pattern works well with table-aware widgets passing record sets between layers. Limitations include inability to support direct instance-to-instance relationships or polymorphism for complex logic, restricting its use to moderately complex domains. Particularly common in Microsoft COM and .NET environments where Record Set is primary data repository; less frequent in Java environments despite potential for increased use with row sets.

Service Layer, contributed by Randy Stafford, defines application boundary with coarse-grained operations establishing set of available operations and coordinating response in each operation. Pattern addresses requirement for multiple interface types (data loaders, UIs, integration gateways) to interact with application logic and data, preventing duplication of interaction logic across interfaces. Divides business logic into domain logic (problem domain aspects like revenue recognition calculation strategies) and application logic (application-specific responsibilities like notification and integration). Domain Models better prevent domain logic duplication and manage complexity, but commingling both logic types in domain objects reduces reusability across applications and complicates potential workflow tool reimplementation. Two implementation variations exist: domain facade approach implements Service Layer as thin facades over Domain Model with facades implementing no logic and Domain Model containing all logic; operation script approach implements Service Layer as thicker classes directly implementing application logic but delegating to encapsulated domain objects for domain logic. Operations grouped several per class defining subject area, forming application "services" extending Layer Supertype. Coarse-grained interfaces suit remote invocation from granularity perspective but incur object distribution costs requiring Data Transfer Objects and complex development effort. Recommendation is starting with locally-invocable Service Layer dealing in domain objects, adding remotability later via Remote Facades if needed. Web-based or Web-services UIs require no separate process for business logic. Service Layer operations identified from use case models and UI design, with CRUD (create, read, update, delete) use cases typically corresponding one-to-one with Service Layer operations. Service abstractions grouped by application subsystems, domain model partitions, or thematic behaviors (examples: ContractsService, ProductsService, RecognitionService). Java implementation prefers EJB 2.0 stateless session beans with local interfaces and operation script approach delegating to POJO domain classes, balancing distributed transaction control convenience against deployment/testing requirements. Distinguished from Session Facade pattern which focuses on avoiding remote invocation penalties on entity beans, while Service Layer addresses responsibility factoring for reusability across architectures. Benefits apparent when multiple client types exist or complex responses involve multiple transactional resources requiring atomic coordination. Less necessary for single-client applications with simple use case responses, though designing in Service Layer from beginning pays off upon envision of second client or transactional resource.

## Chapter 10. Data Source Architectural Patterns

Table Data Gateway holds all SQL for accessing a single table or view, providing methods for select, insert, update, and delete operations. Its interface is simple and stateless, pushing data back and forth through various return mechanisms. A key design decision is how to return query results: maps (weakly typed and error-prone), Data Transfer Objects (explicit and reusable), or Record Sets (convenient for .NET environments but conceptually messy). Table Data Gateway works particularly well with Table Module, where returned Record Sets provide natural data structures. When using Domain Model, the gateway can return domain objects directly, though this creates bidirectional dependencies between gateway and domain objects that the author finds problematic. A single gateway handles one table or view; update-only views use Table Data Gateway without update behavior. The pattern pairs naturally with stored procedures organized as gateways themselves. Fowler finds Table Data Gateway "probably the simplest database interface pattern," but uses it least with Domain Model because Data Mapper provides better isolation.

Row Data Gateway acts as an object mimicking a single database record. Each column becomes a field; the gateway hides all data source access details behind a normal programming language interface. It performs type conversion from database to in-memory types. The pattern requires separate finder objects to retrieve rows, with one finder class and one gateway class per table in relational databases. This avoids static find methods that preclude polymorphism. The critical distinction from Active Record is architectural: Row Data Gateway contains only database access logic with no domain logic whatsoever. Row Data Gateway candidates for code generation based on Metadata Mapping since database access code is mechanical and repetitive. Multiple Row Data Gateways on the same underlying tables can cause update conflicts that developers must manage manually. The pattern works particularly well with Transaction Scripts, factoring out reusable database access code. When using Domain Model with simple mappings, Active Record does the same job without an additional layer; with complex mappings, Data Mapper works better by decoupling data structure from domain objects.

Active Record is a Domain Model class where the object wraps a database row while adding domain logic directly to the class. The data structure matches the database exactly: one field per column, typed as SQL provides without early conversion. The class typically includes: construction from a result set, construction for insertion, static finder methods wrapping SQL queries returning Active Record objects, update and insert methods, field getters and setters, and business logic implementation. The getting and setting methods can perform intelligent conversions from SQL types to better in-memory types and return related Active Records via lookup. The pattern doesn't hide the database's presence; consequently, fewer object-relational mapping patterns appear alongside Active Record. Active Record closely resembles Row Data Gateway but critically includes both data source and domain logic, while Row Data Gateway contains only database access. Static find methods are common in Active Record due to close database coupling, though separate finder classes are possible and better for testing. Active Record works with views or tables, though view updates are harder. Fowler positions Active Record as good for simple domain logic (CRUD, single-record derivations and validations) but problematic for complex logic requiring object features like inheritance, relationships, and collections that don't map easily to isomorphic database schemas. It couples object design to database design, complicating future refactoring of either.

Data Mapper is a separate layer of software isolating in-memory objects from the database while transferring data between them. Unlike patterns that embed database access in objects, Data Mapper keeps objects entirely ignorant of databases, SQL interfaces, and schema structure; the database remains ignorant of objects. As a Mapper, Data Mapper itself is unknown to the domain layer, achieving true architectural separation. A simple Data Mapper example has a Person class and PersonMapper; clients call finder methods on the mapper, which checks an Identity Map before loading from the database. Updates involve clients requesting the mapper to save domain objects; the mapper extracts object data and shuttles it to the database. The entire mapper layer is substitutable for testing or supporting multiple databases against a single domain layer. Simple mappers map a table to an equivalent class on a field-by-field basis, but real mappers handle complexity: classes expanding to multiple fields, multiple tables per class, inheritance hierarchies, and object interconnections. Mapping patterns throughout the book address these complexities more effectively with Data Mapper than alternatives. The mapper layer must track changed objects, new creations, and destroyed instances, fitting the workload into transactional frameworks; Unit of Work pattern organizes this. Finding often generates graphs of loaded objects rather than single SQL queries; finders must understand client usage to minimize queries. Complex finds load multiple domain classes from single queries using joins, loading both order and order-line instances from one result set. Objects interconnect heavily, requiring stopping points to prevent loading entire databases; Lazy Load techniques address this while minimizing impact on objects, though in-memory objects must know about finders and the mapping layer's mechanisms.

Data Mapper can have one layer or several; handcoding favors one mapper per domain class or hierarchy root. Metadata Mapping enables single mapper classes, though find methods become the limiting problem in large applications, making sense to split finder methods by domain class or hierarchy root, yielding small finder classes easy to locate. All finders use Identity Maps maintaining object identity from database reads; either Registry holding Identity Maps or individual finders holding Identity Maps (with one finder per class per session) work. Handling finders involves initial object loading from the presentation layer, then domain layer movement from object to object using associations; Lazy Load enables loading additional objects when needed. Domain objects may invoke Data Mapper find methods, but good Lazy Load avoids this entirely. For simpler applications managing everything with associations and Lazy Load may not justify effort. Still, avoiding domain object dependency on Data Mapper is essential, solvable through Separated Interface: finder methods go into an interface in the domain package, not the data mapper package. Mappers accessing domain object fields face visibility problems: mappers need public methods not useful for domain logic. Solutions include packaging mappers closer to objects (same Java package) reducing visibility issues but confusing larger dependency pictures, using reflection bypassing visibility rules but slower, or using public methods guarded by status fields throwing exceptions outside database-load contexts, with names distinguishing them from regular getters/setters. Object creation offers two options: rich constructors creating objects with mandatory data, ensuring well-formed objects and enforcing immutable fields, or empty objects created with no-arg constructors later populated. Rich constructors create cyclic reference problems when two objects reference each other; each load attempt triggers the other, risking stack overflow, requiring special Lazy Load code that's messy. Empty object approach avoids cycles by creating a blank object with no-arg constructor and immediately inserting it into the Identity Map; cycles then stop since the map returns the already-created object. Empty objects require setters for truly immutable fields, solvable through naming conventions and status-checking guards or reflection for data loading. Metadata-based mappings store field-to-column correspondence as data rather than explicit code, either in a class or separate file. Metadata advantages: all mapper variation handled as data without additional source code, through code generation or reflective programming. The simplest and often best approach remains explicit code with one mapper class per domain object, using mapper method assignments and constant SQL strings for database access.

Data Mapper's primary use case is evolving database schema and object model independently, most commonly with Domain Model. Data Mapper's benefit is allowing domain model development while ignoring databases in design, build, and testing; domain objects know nothing of database structure because mappers handle all correspondence. Code becomes comprehensible since domain objects are understood without knowledge of storage; Domain Model and database modification happen independently. With complicated mappings, especially involving existing databases, this separation is invaluable. The price is the extra layer absent in Active Record; pattern choice rests on business logic complexity. Simple logic probably needs neither Domain Model nor Data Mapper. Complicated logic leads to Domain Model and therefore Data Mapper. Fowler wouldn't choose Data Mapper without Domain Model, but can Domain Model work without Data Mapper? If the domain model is simple and the database is under domain developers' control, domain objects accessing databases directly with Active Record is reasonable, effectively putting mapper behavior into objects themselves. As complexity increases, refactoring database behavior into a separate layer becomes better. Full-featured database-mapping layers are complicated to build; Fowler recommends buying rather than building for most cases.

## Chapter 11. Object-Relational Behavioral Patterns

Unit of Work solves the problem of tracking which objects have been created, modified, or deleted during a business transaction so changes can be written to the database efficiently. Rather than writing each change immediately (causing many small database calls and requiring long-running transactions), or manually tracking dirty objects (which becomes unmanageable), Unit of Work maintains three lists: newObjects, dirtyObjects, and removedObjects. The pattern supports caller registration (the client explicitly registers changes) versus object registration (objects register themselves when modified), with object registration being preferred despite the risk of forgotten registrations. Objects can find their current Unit of Work via ThreadLocal or a session object. Unit of Work handles registration through methods: registerNew() adds objects that must be inserted, registerDirty() marks objects needing updates, registerRemoved() marks objects for deletion, and registerClean() optionally tracks read objects for concurrency checking. At commit(), the Unit of Work opens a transaction, performs concurrency checks using Pessimistic or Optimistic Offline Lock patterns, and writes changes in proper order using a Data Mapper for each object type. A unit of work controller approach (like TOPLink) reads all objects through the Unit of Work and compares copies at commit time rather than tracking dirty flags, allowing selective field updates. The pattern naturally handles batch updates by accumulating multiple SQL commands for single remote submission and naturally extends to transactional resources beyond databases. Alternative approaches include explicit save-on-each-change (inefficient) or leaving all updates to end (difficult to track in complex Domain Models). Object registration benefits from code generation or aspect-oriented programming to inject registration calls automatically. Identity Map ensures that each database record corresponds to exactly one in-memory object instance during a business transaction, preventing anomalies where two different object instances represent the same database row. When loading objects, the Identity Map checks first—if the record already exists in memory, return it; otherwise load from database and add to map. Implementation requires choosing: (1) key strategy—surrogate primary keys work best as map keys, or encapsulate different key types behind a Key Object; (2) explicit vs. generic maps—explicit maps like findPerson(1) offer compile-time type safety and clarity but require new methods per entity, while generic maps like find("Person", 1) are reusable but lack type safety; (3) number of maps—one map per class/table, one per inheritance tree, or one session-wide map (works only with database-unique keys across entities); (4) placement—store in Unit of Work (best location), in a session Registry, or as process-wide and session-specific variants for read-only vs. updatable objects. Inheritance complicates Identity Maps because polymorphic lookups must search multiple maps unless using one map per inheritance tree. Thread safety is critical—Identity Maps belong to sessions and must never be shared across threads. Without Identity Map you risk modification anomalies and incur unnecessary database queries. Value Objects (immutable) don't strictly need Identity Map, though caching improves performance. Dependent Mapping objects don't need separate maps since their parent controls persistence. Lazy Load defers fetching related object data until accessed, balancing between eager-loading entire object graphs (expensive if much data unused) and forcing manual explicit loading (tedious). Four main implementations exist: lazy initialization checks if a field is null and loads on first access, working best with Active Record; virtual proxy is a placeholder object that looks like the target but loads on any method call, avoiding identity problems for collections but causing issues with domain objects due to identity mismatches; value holder wraps the actual object and loads on first access to getValue(), hiding lazy load from callers but losing type safety; ghost represents the real object in partial state (containing only ID initially), loading full state on first field access, allowing immediate Identity Map placement to preserve identity. Collection Lazy Loads risk ripple loading (querying database once per item instead of batching) unless the collection itself is lazy and loads all contents together. Lazy Load works best with Data Mapper since Active Record couples the object to database. Aspect-oriented programming and bytecode post-processing can inject lazy load behavior transparently. Different use cases need different loading strategies—separate mapper variants per use case or strategy objects determine what subset of object graph to load. Rarely need more than two degrees of laziness: complete load and minimal load for list identification. When to use Lazy Load depends on database call cost—if data resides in same row, eagerness doesn't cost extra; only worth using if an extra database call is required and the data often isn't used alongside the main object.

## Chapter 12. Object-Relational Structural Patterns

Identity Field establishes that relational databases distinguish rows by primary keys while in-memory objects use object identity. The pattern stores the database's primary key as a field in the object. Key design decisions include: (1) meaningful vs. meaningless keys (Fowler recommends meaningless numeric keys to avoid immutability and uniqueness issues), (2) simple vs. compound keys (simple keys reduce code but compound keys are sometimes necessary, like order number + sequence for line items), (3) key type selection (long integers are preferred for equality checking and incrementing), (4) table-unique vs. database-unique keys. New key generation strategies include auto-increment fields (problematic because you can't determine the generated value before transaction commit), database counters via sequences (Oracle sequences are ideal), GUIDs (globally unique but large and hard to type), table scans with max() (read-locks the entire table), and separate key tables (recommend storing one or few rows with name/nextID pairs, with batch allocation to reduce contention). Compound key handling uses a dedicated Key class that stores multiple fields and implements equality comparison, with convenience constructors for common configurations.

Foreign Key Mapping maps object references to foreign keys in the database. Single-valued references are straightforward: extract the referenced object's ID and store it. Collections present complications because databases support only single-valued fields. When an album has a collection of tracks, the foreign key is reversed: the track record contains the album ID. Updates to collections require tracking which tracks were added, removed, or moved. Three options exist: (1) delete and reinsert (simple to implement; only works if tracks are Dependent Mappings with no external references), (2) add a back pointer (makes the association bidirectional), (3) diff the collection (compare with current database state or original load state). The pattern also addresses cycles in reference graphs—when loading an order with links to customer and payments creates cycles back to the original order. Two strategies prevent infinite recursion: use Lazy Load at appropriate points to break cycles, or create empty objects and immediately insert them into an Identity Map so cycles reconnect to already-loaded objects rather than recursing.

Association Table Mapping handles many-to-many associations by creating a link table containing only the foreign key IDs of the two associated tables, with one row per association pair. The link table has no corresponding in-memory object and uses a compound primary key from the two foreign keys. Loading requires two conceptual queries: find link rows for the first entity, then find the related objects. Performance can be optimized by joining the skills table to the link table in a single query. Updating involves deleting all existing links and inserting new ones (safe because link tables should have no external references and can be treated as Dependent Mappings). The pattern is essential for many-to-many associations where Foreign Key Mapping is impossible due to the lack of a "single-valued end."

Dependent Mapping handles child objects that exist only in the context of a parent owner, with exactly one owner and no external references. The dependent class has no Identity Field and isn't stored in an Identity Map. Dependents are typically loaded and saved whenever the owner is accessed, though Lazy Load can defer expensive loads. Updates use delete-and-reinsert strategy because there are no external references to track. Dependents can form hierarchies where a dependent owns other dependents. The mapping code lives in the owner's mapper, not in a separate mapper for the dependent. UML composition notation appropriately represents owner-dependent relationships.

Embedded Value maps small objects (like Money or DateRange value objects) into multiple fields of their owner's table rather than creating separate tables. These objects have no Identity Field and are created and destroyed easily. The owning class handles all persistence operations. Embedded Value is appropriate for true value objects lacking identity semantics and for simple dependents that don't need separate Identity Fields. The alternative, Serialized LOB, works with more complex structures but sacrifices SQL queryability of the embedded data.

Serialized LOB persists complex object graphs by serializing them into a single large object (BLOB or CLOB) stored in a database field. BLOB (binary) is simple if the platform supports automatic object serialization and uses minimal space, but creates versioning problems if class structures change and makes the field unreadable without the object. CLOB (character/text) is human-readable and parseable, but uses more space and requires custom parsing. XML CLOBs overcome parsing issues with standard XML parsers and tools but are verbose. The pattern works best for self-contained object graphs not referenced from outside; care must be taken to avoid duplicating data (e.g., don't store customer snapshot in every order unless temporal snapshots are desired, rather store in a single customer table linked by multiple orders). References from objects outside the LOB to objects inside it become awkward and should be avoided.

Single Table Inheritance maps all classes in an inheritance hierarchy into one table with columns for all fields in all classes, leaving irrelevant columns empty. A type code (or class name) column indicates which class each row represents. Strengths: no joins needed for retrieval, table refactoring is easy (pushing fields up/down the hierarchy requires no schema changes), single table to manage. Weaknesses: sometimes-relevant fields confuse direct database users, wasted space from null columns (though many databases compress empty columns efficiently), large table may cause locking/indexing problems, single namespace for fields requires careful naming to avoid collision. The pattern can be mixed with Concrete Table Inheritance for subclasses with substantial unique fields.

Class Table Inheritance creates one table per class in the inheritance hierarchy, with superclass and subclass fields stored in corresponding tables. Rows are linked either through a common primary key (same ID in both tables) or through foreign keys from subclass to superclass table. The key implementation challenge is efficiently retrieving data from multiple tables—joins across many tables are slow, outer joins are nonstandard and slow, and multiple queries increase database round trips. When querying for a superclass (e.g., all Players), you must either join multiple tables or determine types and query separately. Strengths: clear relationship to domain model, no wasted space (all columns relevant), fields moved up/down hierarchy require only moving between tables. Weaknesses: multiple table accesses required, joins can be slow, refactoring up/down hierarchy requires schema changes, superclass tables may become bottlenecks from frequent access.

Concrete Table Inheritance creates one table per concrete (non-abstract) class, with each table containing columns for its own fields plus all inherited fields (duplicated across tables). Each concrete object maps to a single row in a single table. Strengths: self-contained tables with no irrelevant fields; good for multi-application access; no joins for concrete mapper reads; access load spreads across tables. Weaknesses: primary keys difficult to handle (must be unique across all tables, not just table-unique); can't enforce database relationships to abstract classes; field refactoring requires altering multiple tables; superclass field changes require updating all tables containing that field; querying for abstract classes (e.g., all Players) requires checking all concrete tables (multiple queries or awkward outer join). The pattern uses compound keys (table identifier + primary key) to ensure uniqueness across tables when necessary. Referential integrity problems arise when linking to abstract classes—foreign keys can't point to a union of tables.

Inheritance Mappers organizes mapping code for inheritance hierarchies through a parallel class hierarchy. Each domain class has a corresponding mapper class. Concrete mappers (BowlerMapper, CricketerMapper, FootballerMapper) implement find, load, insert, update, and delete for their specific classes. Load methods chain through the hierarchy with each class loading its own fields and calling the superclass load method. Save methods similarly chain through the hierarchy. For abstract class operations (finding/inserting/updating a generic Player), a separate abstract player mapper handles player-specific persistence logic (abstract, used only by concrete mappers), and a concrete player mapper class acts as the interface providing find and delegating insert/update/delete to the appropriate concrete mapper based on the object type. This dual-class approach avoids awkward generalization-composition combinations. The pattern structure applies across Single Table, Class Table, and Concrete Table Inheritance, though details vary by scheme.

## Chapter 13. Object-Relational Metadata Mapping Patterns

Metadata Mapping addresses the tedious, repetitive nature of hand-coded object-relational mapping by storing mapping information in metadata that drives generic code for reading, inserting, and updating data. The two main implementation approaches are code generation (generating mapper classes during the build process from metadata input, with generated classes deployed with server code) and reflective programming (reading field and method names from a metadata file and using reflection to invoke methods dynamically). Code generation is less dynamic—changes require recompilation and redeployment—but provides better debuggability and explicit code visibility, making it preferable for less sophisticated developers. Reflective programming is more flexible, allowing runtime metadata changes, but typically slower and harder to debug. The metadata itself can be held in external files (typically XML for hierarchic structuring and ease of parsing), directly in source code (avoiding parsing but making edits harder), or in the database itself (keeping mapping information together with data). When deciding on metadata complexity, most projects benefit from starting simple (handling core cases) and adding capabilities incrementally. A key technique for handling special cases is to override specific methods through subclasses rather than building metadata complex enough for all edge cases.

Query Object applies the Interpreter pattern to represent database queries as object structures that can generate appropriate SQL strings. This allows developers to form queries using domain object and field names rather than table and column names, making queries independent of database schema details and localizing schema changes to a single place. Query Objects work alongside Metadata Mapping to translate field-based queries into column-based SQL. For multiple databases, Query Objects can generate different SQL depending on the target database or use different mappings for the same classes stored in different schemas. A sophisticated optimization involves detecting redundant queries—either exact duplicates or queries that are specific cases of earlier queries (like the same query with an additional AND clause)—and selecting objects from the Identity Map to avoid database trips. Query Objects typically start minimal, handling only needed functionality (like single-table queries with AND-linked predicates), and evolve as requirements grow. A variation allows query-by-example: a domain object with some fields set and others null acts as a query specification that returns matching records, though this approach breaks down for complex queries.

Repository provides an abstraction layer over Data Mapper that presents a collection-like interface to domain logic, receiving declaratively-constructed criteria objects and returning matching domain objects without exposing the underlying query mechanics or data source. It combines Metadata Mapping with Query Object to automatically generate SQL from criteria, acting conceptually like a small object-oriented database. The Repository interface shields domain code from data source awareness: clients never construct SQL or even think in database terms, constructing criteria objects (implementing the Specification pattern) and submitting them to Repository for satisfaction. Common methods include matching(criteria) for flexible queries, soleMatch(criteria) for queries expecting single results, byObjectId(id) for ID-based lookups, and all() for retrieving entire collections. Repository's power becomes apparent with multiple data sources—it readily supports strategy objects for replacement of the data-mapping component, making it especially useful during testing (in-memory strategies) or with multiple database schemas. A declarative flavor distinguishes Repository from direct Query Object use: with Repository there's no notion of query "execution," but rather "selection" of objects through "satisfaction" of a query specification. This abstraction improves code readability, particularly in systems with extensive querying like browser-based query pages where HttpRequest objects convert directly to criteria submitted to appropriate Repository instances. Repository implementations use strategy objects—RelationalStrategy constructs a Query Object from criteria and executes against the database, while InMemoryStrategy iterates domain objects asking criteria if they're satisfied, using reflection to interrogate field values.

## Chapter 14. Web Presentation Patterns

Model View Controller separates concerns into three roles: the model (domain logic and data), the view (display representation), and the controller (user input handler). The fundamental principle is unidirectional dependency: presentation depends on model, but model remains unaware of presentation. This separation enables testing of domain logic without GUI machinery, supports multiple UI presentations of identical models, and allows independent evolution of presentation and domain. The view/controller separation is less critical; MVC's primary value lies in presentation/model separation. Page Controller assigns one input controller object per logical page or action. It can be implemented as a server page directly handling requests or as a separate script delegating to helper objects. Key responsibilities include decoding URL parameters and form data, invoking model objects to process data, and forwarding to appropriate view pages. Helper objects reduce scriptlet-laden pages. Server pages work well for simple controller logic; scripts handle complex logic. Front Controller consolidates all request handling through a single Web handler that dispatches to command objects specific to each action. The handler is nearly always a class (not a server page) and decides which command to invoke either statically (parsing URL, using conditional logic) or dynamically (instantiating classes by name). Key advantage: only one component needs Web-server-specific configuration. Multiple command instantiation per request eliminates thread-safety concerns for command classes. Intercepting Filter (a decorator wrapping the handler) enables filter chains for authentication, logging, and locale identification at configuration time. Template View embeds markers into static HTML pages; upon request, markers are replaced with computation results. Markers use HTML-like tags (compatible with WYSIWYG editors and XML tools) or special text syntax. Server pages (ASP, JSP, PHP) extend Template View by allowing arbitrary scriptlets, but Fowler emphasizes this is problematic: scriptlets eliminate nonprogrammer editability, compromise modularity, and encourage domain logic leakage into pages. Helper objects are essential, containing all real programming logic while pages focus purely on display. Conditional display is handled by moving conditions into helpers (returning empty strings if false) or using focused conditional tags rather than generic IF tags. Iteration naturally supports collection tags. Template View tests are difficult without running Web servers. Transform View processes domain data element-by-element, writing appropriate HTML for each element type—a fundamentally different organization than Template View. XSLT is the dominant implementation; its functional programming style differs from imperative approaches. Transform requires XML input (either from domain objects directly or via Data Transfer Objects) passed to an XSLT engine with a stylesheet. Trade-offs: fewer tools for XSLT development, more difficult learning curve due to functional style, but better portability across J2EE and .NET, natural fit for XML documents, cleaner separation from rendering logic, and easier testing without a Web server. Two Step View splits transformation into two stages: first stage constructs a presentation-oriented logical screen (fields, headers, tables, etc.) from domain data without HTML; second stage renders that logical structure into HTML. Implemented via two-step XSLT (first stylesheet produces presentation-oriented XML, second produces HTML), classes (presentation-oriented class hierarchy instantiated in first stage, rendered in second), or Template View with logical tags. Advantage for single-appearance applications: changes to appearance require modifying only the second stage. For multiappearance applications (same functionality, multiple designs), using M screens and A appearances, single-stage requires M×A modules; Two Step View requires M first-stage plus A second-stage modules. Drawback: design-heavy sites with unique layouts resist the pattern; presentation-oriented structure constrains what pages can show. Tools require programmer involvement in design changes. Application Controller centralizes screen navigation logic and application flow, avoiding duplication when multiple input controllers need identical sequencing or conditional view selection. It maintains two structured collections: references to domain commands (keyed by command name and application state) and references to views. Stateless Application Controllers can be pooled; stateful implementations manage session state. It's optimal to separate the Application Controller from UI machinery (accessing HTTP sessions directly, invoking server pages) so it can be tested independently and reused across multiple presentations. Useful for state machines where events trigger different responses based on object state, often represented via metadata. Multiple Application Controllers can exist, split by UI area. Domain logic boundary becomes murky—conditional screen display (e.g., showing extra questions for smoker applicants) may belong here or in domain model depending on scope.

This fragment is the opening of Chapter 15 ("Distribution Patterns") and covers the Remote Facade pattern in full.

Intent: "Provides a coarse-grained facade on fine-grained objects to improve efficiency over a network."

Motivating problem: Good OO design favors small objects with small methods (fine-grained), which enables control, substitution, and intention-revealing naming. This works well within a single address space but fails across process boundaries. Remote calls are far more expensive than in-process calls — data marshaling, security checks, network routing, and even speed-of-light latency for globally distributed machines add overhead. Fowler states this cost applies "even if both processes are on the same machine," so it can't be dismissed even by proponents of lazy optimization. The implication: any object exposed remotely needs a coarse-grained interface minimizing call count — e.g., fetching/updating an order and its order lines in one call rather than many small ones. This trade-off sacrifices the clarity and fine control of small objects/methods, making programming harder and slowing productivity when applied universally.

The pattern: A Remote Facade is a coarse-grained facade (citing Gang of Four's Facade pattern) wrapping a web of fine-grained objects. Key architectural rule: none of the underlying fine-grained objects are remote themselves, and the Remote Facade contains no domain logic — it only translates coarse-grained calls into calls on the fine-grained object model.

How It Works:
- Complex logic stays in fine-grained objects designed for in-process collaboration; a separate facade object provides the remote interface — a "thin skin."
- Simple case (Figure 15.1): an address object's many individual getters/setters are replaced by a single bulk accessor getter and setter on the facade. Calling the bulk setter causes the facade to read data from the incoming structure and invoke the underlying individual accessor methods on the real address object — "one call to the facade causes several calls from the facade to the domain object." This keeps validation/computation logic on the address object itself, reusable by other fine-grained objects.
- Complex case: a single Remote Facade (e.g., an order facade) fronts many fine-grained objects — order, order lines, customer data — combining access/update across all of them.
- Data transfer format: if fine-grained classes exist on both sides and are serializable, you can copy them directly across the wire (e.g., getAddressData returns a copy; setAddressData receives an object and copies its fields into the real object rather than replacing it, preserving identity). More often, you can't/don't want to duplicate domain classes across processes or serialize complex relationship graphs — here a Data Transfer Object (DTO) should back the transfer instead.
- Realistic granularity: a Remote Facade rarely maps 1:1 to a single domain object. More typically it's a class like CustomerService with many methods (getAddressData/setAddressData, getPurchasingHistory, updateCreditData) spanning multiple underlying objects.

Granularity trade-off (opinionated): Fowler notes some prefer many small Remote Facades (e.g., one per use case); he prefers few, coarse-grained ones — even a large application might need only half a dozen. Each ends up with many methods, but since the methods themselves are small, he doesn't consider this a problem.

Design driven by client needs: Typically designed around a UI's needs — one Remote Facade per family of screens, with bulk accessor methods per screen and command methods for actions (e.g., changing an order's status via a button). Multiple facade methods may map to the same underlying command — reasonable, because the facade serves external client convenience, not internal implementation purity.

Statefulness: Can be stateless (poolable — good for resource efficiency, especially B2C) or stateful. Stateless facades needing cross-call state must externalize it via Client Session State, Database Session State, or an implementation of Server Session State. Stateful facades hold their own state (simple Server Session State implementation) but risk performance problems under thousands of concurrent users.

Additional responsibilities naturally layered onto Remote Facade methods: security (access control lists per method) and transactional control (each facade call begins and commits its own transaction — transactions shouldn't stay open across a client round-trip since they aren't efficient for long-running spans).

Central admonition: "Remote Facade has no domain logic" (repeated for emphasis). Workflow/coordination logic belongs either in the fine-grained objects or in a separate, non-remotable Transaction Script. Corollary design test: you should be able to run the whole application locally without the Remote Facades and without duplicating any code.

Remote Facade vs. Session Facade (J2EE community pattern, Alur et al.): Fowler initially conflated these but now distinguishes them. Remote Facade = thin remote skin, no domain logic. Session Facade as commonly described does include logic, usually workflow, largely because J2EE session beans wrap entity beans and entity beans aren't re-entrant, so coordination logic needs a home elsewhere — the session bean. He characterizes Session Facade as "putting several Transaction Scripts in a remote interface" — reasonable, but not the same pattern, and he argues it arguably shouldn't be called a "facade" at all since it contains domain logic.

Remote Facade vs. Service Layer: A Service Layer need not be remote, so it isn't restricted to fine-grained methods only for network reasons — its coarser methods (when present) exist for clarity, not network efficiency. Service Layer also has no inherent need for DTOs; it can return real domain objects directly. If a Domain Model is used both in-process and remotely, you can layer a separate Remote Facade on top of a Service Layer. If the process is remote-only, and the Service Layer contains no application logic, it's simpler to fold the Service Layer into the Remote Facade; but if there is real application logic, keep the Remote Facade separate.

When to Use It: Use whenever remote access to a fine-grained object model is needed — gets coarse-grained interface benefits while preserving fine-grained internal design. Most common at the boundary between a presentation layer and a Domain Model running in a different process (e.g., Swing UI talking to a server domain model, or a servlet talking to a server object model across separate app/web servers). Important nuance: the expense of inter-process calls is high enough to warrant this pattern even between processes on the same machine — it's not purely a network-distance concern.

Explicitly not needed when everything is in a single process: not between a client Domain Model and its own presentation, and not between a CGI script and an in-process Domain Model in one web server. Also not typically paired with Transaction Script, since Transaction Script is already inherently coarse-grained.

Scope limitation noted: Remote Facades imply synchronous, RPC-style distribution. Fowler acknowledges asynchronous, message-based communication often improves responsiveness significantly and has compelling advantages, but explicitly states discussion of asynchronous patterns is out of scope for the book.

This second part of the chapter closes out the Remote Facade example, then presents the full "Data Transfer Object" pattern.

Data Transfer Object definition: An object that carries data between processes to reduce the number of method calls.

Motivation/How It Works: Because each call to a Remote Facade is expensive, calls must be minimized, which means each call must carry more data. Using many parameters is awkward and often impossible in languages like Java that return only one value. The DTO solves this by bundling all needed data into one serializable object. Fowler notes this is one of those objects "our mothers told us never to write" — little more than fields plus getters/setters — but valuable precisely because it lets you move many pieces of information in a single network call.

Key design points:
- A DTO usually carries more data than immediately requested, anticipating what the remote caller will need "for a while," because it's better to over-send than to incur additional round trips.
- A single DTO typically aggregates data from multiple server-side domain objects (e.g., an "album" DTO might include artist, tracks, and performer data), not just one.
- You generally cannot serialize Domain Model objects directly — their object graphs are too complex/interconnected to serialize, and you don't want domain classes replicated on the client. DTOs hold simplified, flattened data: primitives, strings, dates, or other DTOs, structured as simple hierarchies rather than the complex webs seen in a Domain Model. DTO classes (and anything they reference) must exist on both sides of the connection.
- DTOs are often designed around a specific client need (a web page, a GUI screen), so multiple DTOs may exist for the same conceptual entity depending on the screen.
- Design trade-off — one DTO per interaction vs. one per request: separate DTOs per call make it clearer what data moves where but multiply classes; a single shared DTO is less work but obscures per-call data flow. Fowler leans toward one DTO when there's a lot of data commonality, but freely mixes in dedicated DTOs for particular requests.
- Whether to share one DTO for request and response, or use two: no blanket rule — use one if the data is similar, two if very different.
- Mutable vs. immutable DTOs: some prefer immutability (receive one DTO, return a different one even of the same class); Fowler prefers mutable DTOs since it's easier to populate data incrementally, though he notes immutability advocates partly want to avoid confusion with the separate Value Object pattern.
- A common concrete form of DTO is a Record Set (a tabular set of records resembling raw SQL output) — "the DTO for a SQL database." A Domain Model can generate a Record Set for transfer, which the client treats as if from SQL directly — useful when the client has tools that bind to Record Set structures; pairs naturally with Table Module.
- Another form is a generic collection (dictionary/map) rather than an explicit class. Fowler discourages arrays (indices obscure meaning) but says a dictionary is workable for ad hoc cases without a generator — easier to hand-manipulate than writing an explicit object by hand, though it sacrifices explicit interfaces and strong typing. With a code generator available, he prefers explicit typed interfaces.

Serializing the DTO: depends on both endpoints' capabilities. Built-in mechanisms (Java binary serialization, .NET binary/XML serialization) are preferred when available since DTOs are simple structures. Absent built-in support, code generators can produce record classes plus read/write serialization code. Reflective programming can serialize generically via a superclass, at a possible performance cost. When both ends aren't controlled by the same team, a connector/adapter can be built at the foreign end while keeping a simple DTO locally.

Text vs. binary trade-off: text (especially XML) is easier to inspect/tool but costs more bandwidth and often incurs real performance penalties.

Synchronization fragility: if server and client DTO definitions drift, pure binary serialization tends to break outright on any structural change (even adding an optional field), causing deserialization errors. More tolerant approaches — XML serialization written to tolerate change, or dictionary-based binary serialization — degrade more gracefully, though Fowler still dislikes dictionaries as the primary DTO representation, just as a tolerant serialization technique.

Assembling a DTO from Domain Objects: A DTO should know nothing about how to connect to domain objects, since it must be deployable unchanged on both sides of the connection; conversely, domain objects shouldn't depend on the DTO structure. Fowler's solution: a separate assembler object responsible for building a DTO from the domain model and for updating the domain model from a DTO (Figure 15.4). The assembler is characterized as an instance of the Mapper pattern. Multiple assemblers can share one DTO class when different update semantics apply to the same underlying data in different scenarios. DTOs can often be generated automatically from a simple data description, while assemblers usually cannot be auto-generated and require hand-written mapping logic.

When to Use It: whenever multiple data items must move between two processes in one method call. Alternatives Fowler dislikes: (1) pass-by-reference/many-argument setters/getters instead of an object — awkward since languages like Java permit only a single return value; (2) using a raw string representation directly without an encapsulating object — couples all callers to the specific string format, whereas hiding the representation behind an explicit interface lets you swap string/binary formats later without ripple effects. DTOs are specifically worth creating when communicating via XML, since manipulating the XML DOM directly is painful. Another common purpose: a DTO as a shared data carrier passed through and mutated by successive layers/components — Record Set in COM/.NET again cited as the exemplar. Fowler also flags an asynchronous use case: return a DTO normally for synchronous calls, but for an asynchronous call return a Lazy Load wrapping the DTO; the caller only blocks when it actually accesses the DTO's data.

Further Reading: terminology collision — Alur et al. (J2EE-community "Value Object" pattern) discuss the identical concept under the name "Value Object," which Fowler explicitly distinguishes from his own separate Value Object pattern; he deliberately uses "Data Transfer Object" to avoid this confusion. He credits Alur et al.'s "Value Object Assembler" as the source discussion for the assembler concept, keeping the "assembler" name rather than naming it after Mapper. Marinescu is cited for DTO implementation variants; Riehle et al. for flexible serialization approaches.

Example 1 — Transferring Information About Albums (Java): domain model of Artist/Album/Track objects (Figure 15.5) mapped to a simplified DTO structure (Figure 15.6) with AlbumDTO and TrackDTO. The album's artist data is collapsed directly into the album DTO; track performers become a simple string array — the typical "flattening" from a richer domain graph to a DTO. No separate Artist DTO is needed since all needed artist data already appears within Album/Track DTOs.

AlbumAssembler class: writeDTO(Album subject) builds an AlbumDTO from a domain Album, delegating to private writeTracks and writePerformers helpers that flatten tracks and performer names into arrays. createAlbum(String id, AlbumDTO source) creates a brand-new domain Album from an incoming DTO, looking up the artist via a Registry pattern and throwing a RuntimeException if the artist isn't found (a deliberate design decision — a different creation policy could instead create missing artists on the fly), then builds tracks/performers via createTracks/createPerformers. updateAlbum(String id, AlbumDTO source) is a distinct method for updating an existing Album (looked up via Registry), illustrating whether to mutate an existing domain object in place versus discard-and-replace — Fowler updates in place (since other objects reference them) but simply replaces track title/performers wholesale rather than diffing. Fowler notes such a routine handles most cases well, though primitives need extra handling.

He closes with a wry aside that he'll be glad when JAXB makes this manual boilerplate unnecessary.

## Chapter 16. Offline Concurrency Patterns

## Optimistic Offline Lock

Optimistic Offline Lock solves the inconsistent read and lost update problems by validating pre-commit that no session has altered records since another session loaded them. This pattern assumes conflict probability is low, allowing high concurrency. The implementation associates a version number with each record. When a session loads a record it stores the version; commitment validates that the version matches the current database version before applying changes. The comparison happens via SQL criteria: UPDATE and DELETE statements include the version in their WHERE clause, and a row count of 1 indicates lock acquisition success; 0 means conflict requiring rollback.

Beyond simple version numbers, developers should store modification timestamps and the user who last modified each record for conflict reporting. An alternative uses a complete field-value comparison in the WHERE clause rather than version numbers, but this complicates SQL and may impact performance. A critical oversight is addressing inconsistent reads—where one transaction depends on data it doesn't modify. For example, a billing session creates charges based on a customer's address that another session modifies. The charge session must check the address version despite not modifying it, either by adding it to the change set or maintaining a separate version-check list. Condition-based checks (verifying specific field presence rather than full version increment) can improve liveness when transactions depend only on particular record properties. The pattern can detect dynamic query result changes by comparing query results at commit versus initial execution.

The pattern alone doesn't solve all concurrency issues—domain expertise determines whether a conflict truly exists. Source code management systems exemplify powerful Optimistic Offline Lock usage through sophisticated merge strategies. Business object merging is possible but complex. A `checkCurrent()` method allows early conflict detection without guaranteeing commit success, useful for long-running processes.

Optimistic Offline Lock is appropriate when conflict likelihood is low. If high conflict causes users to abandon the system (work repeatedly thrown away at commit), Pessimistic Offline Lock becomes necessary. However, optimistic locking is easier to implement and less prone to deadlock than pessimistic schemes, making it the default approach. The correct concurrency strategy maximizes concurrent data access while minimizing conflicts.

## Pessimistic Offline Lock

Pessimistic Offline Lock prevents conflicts by acquiring locks before data access, ensuring users complete transactions without conflict-induced failure. It requires three phases: determining lock types, building a lock manager, and defining lock acquisition/release protocols.

Three lock types exist: exclusive write lock (required only to edit, minimal contention), exclusive read lock (required to read or edit, restrictive but ensures current data), and read/write lock (concurrent reads allowed, write-locked records block all access, maximizing concurrency with implementation complexity). Lock type choice balances business needs, system concurrency, and developer comprehension—locking is as much a domain modeling problem as a technical one. Ineffective strategies that prevent concurrent access or fail to prevent early conflicts remain problematic regardless of implementation quality.

The lock manager maps locks to owners. It may use in-memory hashtables (for single-server) or database tables (for clustered environments). Access to the lock table must be serialized; database-stored locks benefit from uniqueness constraints (exclusive locks) or serializable isolation levels (read/write locks). Database deadlock is avoided by immediately throwing exceptions on unavailable locks rather than waiting—lock manager exceptions signal failure immediately rather than timeouts. Lost session handling through timeouts (implemented via HTTP session listeners or timestamp-based expiration) prevents locks from being permanently held.

Lock acquisition protocols must specify what to lock and when. Generally, acquire locks before loading data to guarantee freshness. What gets locked is typically the primary key/ID, enabling lock acquisition before object instantiation. Locks release when the business transaction completes; early release requires careful design matching lock type and continued object usage. When locks unavailable, the simplest course is aborting—Pessimistic Offline Lock's early failure should make re-attempt acceptable. Serialize lock table access carefully; with large lock sets, Coarse-Grained Lock addresses contention.

Pessimistic Offline Lock fits high-conflict scenarios or when conflict costs are unacceptable. However, overuse (locking every entity) degrades concurrency severely. It complements Optimistic Offline Lock well—use pessimistic locks only where truly required. Long transactions sometimes offer similar benefits with easier programming, though load testing should guide the choice. Don't use offline pessimistic locking if business transactions fit within single system transactions—leverage database "SELECT FOR UPDATE" or entity EJBs instead.

## Coarse-Grained Lock

Coarse-Grained Lock addresses the complexity of locking sets of related objects (e.g., customer and addresses) individually. It provides a single lock covering many objects, eliminating the need to load all group members to lock them.

For Optimistic Offline Lock, a shared version object across the group creates the single contention point. All group members reference the same version instance (not equal versions). Incrementing this shared version locks the entire group. For Pessimistic Offline Lock, group members share a lockable token; a shared version object makes an excellent token.

Eric Evans' aggregate concept defines a cluster of associated objects treated as a unit for data changes. An aggregate has a root (sole access point) and boundary (membership definition). Root lock implementation locks all aggregate members by navigating each object to its root and locking it. Navigation can use direct references or parent-chain traversal. Direct references are efficient; parent-chain navigation requires loading intermediate objects potentially impacting performance. Lazy Load patterns prevent loading before needed and avoid infinite bidirectional relationship loops, but lazy-loading across system transactions risks inconsistent aggregate construction.

Shared and root lock implementations trade off database join overhead (shared locks require joins to version tables) against navigation cost (root locks must traverse object graphs, possibly reloading objects for freshness). Legacy systems impose additional constraints. Choosing correct lock type requires balancing concurrency, meeting business needs, and managing code complexity.

Use Coarse-Grained Lock when business requirements mandate group locking (one user edits lease while another edits its asset—both should lock the entire group), or when acquisition/release is significantly cheaper than individual locks. However, avoid creating unnatural object relationships purely for performance-based locking.

## Implicit Lock

Implicit Lock prevents developer errors by having framework or Layer Supertype code automatically handle mandatory locking mechanics. Without implicit handling, forgetting a single lock acquisition can render entire locking schemes useless—if an item *might* be locked anywhere, it *must* be locked everywhere. Offline concurrency management's difficulty in testing allows such errors to slip past test suites.

Implementing Implicit Lock requires assembling mandatory locking tasks and factoring code so frameworks execute them. For Optimistic Offline Lock mandatory tasks include: storing version counts per record, including version in update SQL criteria, and storing incremented versions on changes. For Pessimistic Offline Lock: acquiring read locks before data access and releasing all locks on transaction completion.

For Pessimistic Offline Lock write locks (exclusive write or read/write lock's write portion) cannot be implicitly acquired—points like Unit of Work registration offer no guarantee the transaction aborts immediately if write locks are unavailable, violating Pessimistic Offline Lock's intent that users avoid wasted work. Instead, frameworks should ensure write locks were acquired before commit time, throwing concurrency exceptions (not assertions) if missing. This avoids silently allowing inconsistent data in production where assertions may be disabled.

Implicit Lock requires care—developers using pessimistic schemes with waiting locks must still consider deadlock. The danger is unexpected failure modes once developers stop thinking about locking mechanics. Making locking work requires determining how frameworks best carry out the mechanics implicitly.

Implicit Lock should be used in all but simplest applications lacking framework infrastructure. The risk of forgotten locks is too great.

## Chapter 17. Session State Patterns

Client Session State stores all session data on the client, requiring the client to send the complete session state with each request and the server to return updated state with each response. This enables stateless servers with maximal clustering and failover resilience. For HTML interfaces, three mechanisms exist: URL parameters (easiest for small data but limited by URL length constraints, problematic for bookmarks), hidden fields sent with HTML forms (serialized to XML or text-based encoding, visible via page source inspection), and cookies (automatically transmitted but size-limited, disabled by some users, vulnerable to prying eyes like fingers altering data). The pattern works well only with minimal session data; as data volume grows, transfer overhead becomes prohibitive. Security concerns are paramount: any data sent to the client is vulnerable to inspection and alteration, requiring encryption to defend against both automated and manual attacks. All returned data must be completely revalidated. Session identification itself nearly always uses Client Session State (usually just a single session ID), but random generation or hashing reduces session-stealing risks. The pattern excels for stateless server designs and supports load balancing automatically, but fails when session data becomes substantial.

Server Session State keeps session objects in memory on an application server, keyed by session ID in a map structure; the client sends only the session ID and the server retrieves the session object to process requests. In its simplest form this assumes a single application server, sufficient memory capacity, and acceptable loss of all session work if the server fails. Real implementations require variations: passivating (serializing) session state to persistent storage using a memento pattern to free memory resources. Persistence form choice weighs binary serialization (easier, more compact, faster activation) against textual form like XML (human-readable, versioning-problematic). Binary serialization's major drawbacks are unreadability and versioning issues when classes change post-serialization; multiple application servers at different versions or 24/7 environments rarely experience session spanning across upgrades. Storage location determines scaling: application server file system or local database provides simple in-memory implementation but sacrifices clustering and failover; shared server or database storage enables clustering and failover but costs activation time, partly recoverable through caching. The arbitrary boundary between Server Session State and Database Session State falls where serialized objects convert to tabular form. Session cleanup in databases requires daemon processes or partitioning strategies (rotating twelve database segments every two hours, deleting oldest segment, directing inserts to emptied segment, tolerating rare twenty-four-hour session losses). Java implementations use HTTP session (simple, causes server affinity, no failover) or stateful session beans (EJB container handles persistence and passivation automatically, avoiding manual coding). EJB stateful beans' failover capabilities vary by vendor; IBM WebSphere serializes into BLOB in DB2 for multi-server access. Entity beans can store Serialized LOB session data simply. The claim that stateless beans always outperform stateful beans is false—load testing typically shows no significant performance difference at typical concurrent user loads (couple hundred), making stateful beans reasonable when simpler. .NET provides built-in Server Session State storing data in-process by default or in a separate state service (local or remote), changeable via configuration without code modification. Cluster and failover support remove the main programming burden from developers as application servers increasingly provide automatic capabilities.

Database Session State has the server pull required data from the database when a request arrives, perform work, then save all modified data back to the database. The client stores minimal information (typically session ID and keys to find appropriate database data). Session data separates into categories: local to current interaction only versus committed data relevant across all interactions. A core design problem emerges: session data typically shouldn't affect the broader system until the entire session commits, yet saving intermediate state to persistence is required. Pending versus committed data requires explicit separation. Simple approaches add a Boolean isPending field to each row; better approaches store a session ID as a pending field, making it easy to find all session data for a particular session. This session ID approach is invasive because all applications touching record data must know to exclude it via sessionID IS NOT NULL clauses, though database views can filter invisibly at some cost. Separate pending tables offer less invasiveness: create pending_orders and pending_order_lines mirroring existing tables, route session saves to pending tables and commit-time saves to real tables. Pending tables allow dropping record data integrity rules and validation requirements that don't apply to in-progress work. Tables must be exact clones with identical field names plus a session ID field for efficient lookup. Session cleanup requires finding and deleting all data by session ID; abandoned sessions need timeout mechanisms and a session tracking table recording last interaction time. Rollback becomes complex with updates: disallow rollback (making updates part of record data immediately, matching user expectations), or use pending tables copying modified data for session-scoped changes, remembering that pending fields require session ID as part of key to store old and new values simultaneously (messy). If pending tables are only read by session objects, tabularizing data provides little benefit—Serialized LOB becomes a better choice, crossing into Server Session State territory. Avoiding pending data entirely by designing all data as record data simplifies the pattern dramatically, though often impractical or awkward. Database Session State gains from stateless server objects enabling pooling and easy clustering, but pays database round-trip costs per request. Caching server objects reduces read costs but not write costs. Programming effort concentrates on session state handling; when no session state exists and all data saves as record data per request, this pattern becomes obvious. Between Database and Server Session State, clustering and failover support availability in the specific application server platform becomes decisive; Database Session State clustering and failover are usually more straightforward through regular database mechanisms.

## Chapter 18. Base Patterns

Gateway encapsulates access to awkward external system APIs (relational databases, CICS transactions, XML structures) behind a simple object interface, translating method calls to specialized APIs. It's primarily used as a testing point for Service Stub and should remain minimal, focusing on adaptation and testability rather than complex business logic. The pattern differs from Facade (which simplifies complex APIs for general use) and Adapter (which maps existing interfaces) in that Gateway is client-specific and may copy the wrapped interface entirely for substitution purposes.

Mapper creates an insulating layer between two independent subsystems that must communicate without knowing about each other, handling data shuffling and invocation logic. Unlike Gateway which wraps external resources, Mapper decouples two parts of your own system. It's more complex than Gateway and mostly used for database interactions (Data Mapper pattern).

Layer Supertype provides a common superclass for all objects in a layer, consolidating shared features like Identity Field handling and common methods. This reduces duplication when multiple classes need identical behavior across a layer.

Separated Interface defines an interface in one package but implements it in another, breaking architectural dependencies. Implementation dependencies point to the interface, not vice versa. This enables Plugins and allows client packages to define the interface their dependencies must satisfy, particularly useful when domain code shouldn't reference lower layers.

Registry functions as a well-known global object providing lookup access to common objects and services. Static methods hide singleton implementation details from callers. Critical consideration: Registry data scope varies—process-scoped (static fields, singletons), thread-scoped (using ThreadLocal), or session-scoped. Singletons are problematic for multi-threaded mutable data due to thread synchronization complexity. Registry should be used as a last resort, with passing parameters or constructor injection preferred for testability. The Pattern differs from Service Locator in implementation specificity—explicit class methods maintain type safety and self-documenting APIs better than generic map-based lookups.

Value Object represents small objects like money or dates whose equality is based on field values, not identity. Unlike reference objects using identity-based equality (primary keys, object references), Value Objects should be immutable to prevent aliasing bugs. They're passed by value, not reference. Persistence uses Embedded Value or Serialized LOB. .NET provides first-class struct support for Value Objects with automatic value semantics.

Money specializes Value Object for monetary calculations, critical because mainstream languages lack native money types. Amount should use integral types (long in Java) or fixed decimals—absolutely avoid floating-point to prevent rounding errors. Key challenges: currency-aware arithmetic operations that detect mismatched currency additions; complex rounding behavior during multiplication and especially allocation across multiple targets. Solutions include: ignoring minor discrepancies (accountants hate this); subtracting last allocation to avoid losing pennies; parameterized rounding schemes; or dedicated allocator functions that distribute remainder pennies pseudo-randomly. Conversion operations need dedicated convertor objects to handle non-simple rounding rules. Comparison operations require currency awareness. Storage uses Embedded Value with currency stored either on each money object or once on the containing entity.

Special Case provides special behavior for particular cases (nulls, infinity, unknown entities) as subclass instances rather than returning null. This eliminates scattered null-checking code. Different special cases can represent different null meanings (missing vs. unknown customer). Special Cases often return other Special Cases (unknown customer returns unknown bill). Null Object is a specific case of Special Case for null handling.

Plugin links class implementations to interfaces at runtime via configuration rather than compile-time, solving configuration management without scattered factory conditionals or rebuilds. Configuration uses text files mapping interface names to implementation class names. Reflection enables dynamic instantiation without compile-time dependencies. Separate factory package enables build-time dependency checking. Used with Separated Interface, Plugin defers implementation selection to runtime configuration, essential for multi-environment deployment (test vs. production with different generators, services, or database backends).

Service Stub replaces problematic remote services with local, fast, in-memory implementations during testing. Eliminates development delays from unavailable external services and enables test execution. Implemented as Separated Interface with real and stub implementations loaded via Plugin. Stub simplicity is crucial—complexity defeats the purpose. Start with flat rates for services like tax calculation, add conditional exemptions for specific test cases, or dynamic registration methods. Stub interfaces should match the real service Gateway, with assertion failures in unused test methods on production implementations.

Record Set provides in-memory tabular data representation mimicking SQL query results, enabling two-tier tools (rapid UI development on relational data) without mixing business logic in databases or UI code. It disconnects from data source for network transmission and acts as Data Transfer Object when serialized. Platforms provide Record Sets (ADO.NET DataSet, JDBC RowSet), but you can build them as lists of maps. Implicit interfaces (generic `aReservation["passenger"]`) sacrifice type safety and self-documentation for flexibility; explicit interfaces (typed property accessors) are superior for maintainability, especially in statically typed languages where code generation can produce strongly typed DataSets from XSD definitions.
