## Preface

The authors open by questioning whether architecture remains relevant given the proliferation of cloud infrastructures, microservices, frameworks, and reference architectures. They observed that some might assume architects today only need to select tools, instantiate them, and configuration will follow. To challenge their own potential bias, they consulted working architects across diverse domains (healthcare, automotive, social media, aviation, defense, finance, e-commerce) who confirmed architecture's continued importance.

Four key reasons justify this relevance. First, the accelerating rate of new requirements—driven by customer needs and competitive pressures—means that without careful attention to system modularity, systems become difficult to understand, change, debug, and modify. Microservices alone are not a panacea; modularity remains a fundamental architectural concern. Second, while abstraction levels in systems are rising (enabling sophisticated services to be used without understanding implementation), the overall complexity of systems being created is increasing at least as rapidly, making architecture's traditional role of taming complexity as critical as ever.

Third, the introduction of Model-Based Systems Engineering (MBSE) as a transformational enabler in systems engineering is reshaping how systems are designed. MBSE represents a shift from document-centric to model-centric engineering, where graphical, mathematical, and physical models are applied consistently to build systems more effectively. The authors note that what is being modeled is fundamentally architecture, and architects are the ones building these models. Fourth, high employee turnover in information systems means no single person understands everything in real systems—hence a guiding hand is needed to make architectural decisions consistently.

Finally, despite automation tools like Kubernetes that handle orchestration, deployment, and management, architects must still understand the quality attribute properties of systems and the emergent properties when systems are composed together. Quality attributes like performance, security, availability, and safety are subject to "weakest link" problems that only emerge during composition; without an architect's oversight, composition failures are likely.

## Chapter 1. What Is Software Architecture?

The authors establish that software architecture is fundamental to successful system development, defining it as the set of structures needed to reason about the system, with structures comprising software elements, relations among them, and their properties. This definition deliberately avoids framing architecture as simply "early" or "major" decisions, recognizing that in Agile and spiral projects, architectural decisions are not always made early and early decisions are not always architectural. A structure is architectural if it supports reasoning about properties important to stakeholders—functionality, fault tolerance, modifiability, and responsiveness—rather than based on timing.

The chapter establishes several implications: First, architecture is a set of software structures; since no single structure constitutes "the" architecture, structures are categorized into three useful groups for design, documentation, and analysis. Second, architecture is an abstraction intentionally omitting information not useful for reasoning, focusing on public element interfaces while suppressing private implementation details—this abstraction is essential for managing complexity so architecture understanding requires orders of magnitude less effort than grasping all implementation details. Third, architecture differs from design: architecture defines certain aspects and structures, but many design decisions remain unbound and depend on downstream designers and implementers. Fourth, every software system has an architecture, though it may not be documented or known. Finally, not all architectures are good; they either support or hinder achieving important requirements, making architecture design and evaluation crucial practices.

Module structures partition systems into implementation units (modules like classes, packages, or layers) with specific computational responsibilities, representing a static view for construction or procurement. Decomposition structure shows hierarchical "is-a-submodule-of" relations, determining modifiability by encapsulating changes within small modules and forming the basis for project organization and team structure. Uses structure captures the specialized dependency relation where one module requires a correctly functioning version of another, enabling subsetting and incremental development while measuring social debt by identifying which teams must communicate. Layer structure organizes modules as abstract "virtual machines" providing cohesive services through managed interfaces, with strict layering allowing each layer to use only one other layer below it, enhancing portability through ability to change the underlying virtual machine. Class (or generalization) structure groups related behaviors through "is-a" or "is-an-instance-of" inheritance relations, supporting reuse and incremental functionality addition. Data model shows entities and many-to-many relationships for consistent global data structures, with attributes, types, and relationship cardinalities indicating functional dependencies between entities.

Component-and-connector (C&C) structures focus on runtime behavior, showing how elements interact at execution to carry out system functions. Components are principal units of computation (services, peers, clients, servers, filters) and connectors are communication vehicles (call-return, process synchronization, pipes). Service structure shows services interoperating through coordination mechanisms like messaging, useful for independently developed components with different origins. Concurrency structure identifies opportunities for parallelism and resource contention by arranging components into "logical threads"—sequences of computation allocatable to separate physical threads later in design.

Allocation structures map software structures to nonsoftware structures such as hardware, organizational units, and file systems. Deployment structure shows software assignment to hardware processing and communication elements using "allocated-to" and "migrates-to" relations, enabling reasoning about performance, security, and availability in distributed systems, particularly for achieving deployability quality attributes. Implementation structure maps modules to file structures in development, integration, and test environments, supporting configuration control and build management. Work assignment structure assigns module implementation responsibility to development teams; Amazon's decision to assign a single team to each microservice exemplifies how work assignment has architectural implications for team communication patterns, expertise distribution, and organizational structure.

The chapter emphasizes structures are not independent: elements from one structure map to others with often many-to-many correspondences. A single module might manifest as one, part of, or several components at runtime; a code unit compiled into a replicated service creates many runtime instances from one module. Projects often designate one structure dominant—frequently module decomposition as it mirrors team structure—though some prefer dominant C&C structures emphasizing runtime behavior and quality attributes.

Smaller systems may omit trivial structures: single-process systems need not document a process structure, single-processor systems need not detail deployment structure. Architects should select structures based on leverage on the system's most important quality attributes, choosing only those returning positive investment in development or maintenance costs.

For good architecture, the authors recommend five process practices: (1) single architect or small lead group ensuring conceptual integrity with strong development team connection avoiding ivory-tower designs; (2) basing architecture on prioritized, well-specified quality attribute requirements rather than functionality, informing necessary tradeoffs; (3) documenting using views (representations of architectural structures) addressing key stakeholder concerns on the project timeline, with possible minimal initial documentation elaborated later; (4) evaluating architecture early for quality attribute delivery when returns are highest, repeated iteratively to ensure changes have not rendered design obsolete; (5) enabling incremental implementation through skeletal systems exercising communication paths before full functionality, avoiding large-bang integration and discovering problems early.

Structural recommendations include: (1) well-defined modules using information hiding and separation of concerns, encapsulating likely-to-change aspects and providing well-defined interfaces enabling team independence; (2) using well-known architectural patterns and tactics for quality attributes rather than unprecedented approaches; (3) avoiding dependence on specific tool or product versions, or structuring transitions to be straightforward and inexpensive; (4) separating data-producing modules from data-consuming modules for staged, incremental upgrades; (5) avoiding one-to-one module-component correspondence—concurrency creates multiple component instances from one module, and multiple threads may use services from components built from different modules; (6) ensuring processor assignment for each process is easily changeable, perhaps at runtime, supporting virtualization and cloud deployment; (7) employing a small number of simple, consistent component interaction patterns so systems do the same things in the same way throughout; (8) identifying specific resource contention areas with clearly specified resolutions, such as network utilization guidelines or performance time budgets.

The chapter contextualizes software architecture within system and enterprise architecture. System architecture maps functionality onto hardware and software while considering human interaction, defining processor allocation and network types affecting software structure. Enterprise architecture describes organizational processes, information flow, and how software systems support business goals. Both provide constraints within which software architects operate; while all share qualities—designability, evaluability, stakeholder alignment, structural composition—each has specialized vocabulary, and software architecture shares commonalities with both insofar as they involve structures, patterns, and design practices.

## Chapter 2. Why Is Software Architecture Important?

Architecture's primary importance lies in its direct determination of system quality attributes. High performance requires managing time-based behavior of elements, their use of shared resources, and frequency and volume of interelement communication. Modifiability demands assigning responsibilities and limiting coupling so changes affect minimal elements. Security requires managing interelement communication and introducing specialized elements like authorization mechanisms. Scalability needs localization of resource use and avoidance of hard-coded assumptions. Reusability requires restricted coupling so elements extract cleanly. However, architecture alone cannot guarantee quality—poor downstream design or implementation decisions undermine adequate architectural design.

Architecture partitions possible changes into three categories. Local changes modify a single element, such as adding a business rule to a pricing module. Nonlocal changes require multiple element modifications but preserve the underlying architectural approach—for example, adding a pricing rule, updating database fields, and revealing results in the user interface. Architectural changes affect fundamental element interactions and ripple throughout the system, such as converting from single-threaded to multi-threaded execution. An effective architecture makes the most common changes local and easy, with nonlocal changes staged over time in an orderly manner. Architects must reason about architectural relationships, performance, and behaviors to decide which changes are essential, assess consequences, and arbitrate change sequences and priorities, preventing accumulation of architecture debt.

Architecture enables predictable quality attributes. Instead of making random design decisions, building, and hoping for performance or security, architects can make specific decisions known to produce quality attributes and confidently expect those outcomes. Upon examining an architecture, one can determine whether necessary decisions were made and predict associated qualities. Understanding how architecture determines qualities and knowing how to produce those qualities provides profound leverage for system development.

Architecture serves as a simplified abstraction and common language for stakeholders—users, customers, project managers, coders, testers—each with different concerns. Users care about speed, reliability, and availability. Customers care about meeting schedule and budget. Managers want independent team work with controlled interaction. The architect addresses all these goals. Architecture documentation provides intellectual manageability even for large, complex systems, enabling early decisions that influence both quality and usefulness. Different stakeholders can understand the architecture to the extent needed—nontechnical people grasp the abstraction with coaching, while it refines into rich technical specifications for implementation, integration, testing, and deployment.

Architecture embodies the earliest design decisions—whether the system runs on one or multiple processors; whether software is layered and how; whether components communicate synchronously or asynchronously, transferring control or data; whether information is encrypted; which operating system; which communication protocol. These early decisions carry enormous weight because they influence and constrain subsequent decisions. Changing them causes ripple effects requiring additional changes throughout the system. Like an artist's initial choices about canvas material and media, early architectural choices disproportionately influence all that follows.

Architecture defines constraints on implementation. Elements must conform to prescribed structure, interactions, and responsibilities. Architects assign performance budgets, for example—if each unit stays within its budget, the overall transaction meets its performance requirement. Implementers of individual pieces need only know their own budget, not the overall budget. This allows architects to establish tradeoffs without requiring implementers to understand them, while architects need not be experts in all algorithm design or programming language intricacies, though they must know enough to avoid designing something difficult to build.

Architecture dictates organizational structure through the work-breakdown structure. Large projects divide labor by assigning different groups different system portions. The architecture provides the broadest system decomposition and forms the basis for work-breakdown structure, which dictates planning units, scheduling, budgets, interteam communication channels, configuration control, file-system organization, integration and test plans, and even organizational minutiae. Teams communicate via interface specifications for their elements. Maintenance activities also reflect software structure with teams formed to maintain specific architectural elements. However, once architecture is agreed upon and formalized in contractual relationships, changing responsibilities becomes expensive or litigious, freezing some architectural aspects and making significant modifications costly for managerial and business reasons.

Architecture enables incremental development. After architecture definition, the first increment can be a skeletal system with infrastructure present—how elements initialize, communicate, share data, access resources, report errors, log activity—but without much application functionality. Infrastructure and application functionality can be built hand in hand: design and build little infrastructure supporting little end-to-end functionality; repeat until done. Many systems are built as skeletal systems extended via plug-ins, packages, or extensions—examples include the R language, Visual Studio Code, and most web browsers. Extensions provide additional functionality over the skeleton. This approach aids development by ensuring executable systems early in the product lifecycle. Fidelity increases as extensions are added or early low-fidelity versions or surrogates—which consume and produce data at appropriate rates but do little else—are replaced by more complete versions. This identifies potential performance and other problems early. Alistair Cockburn's "walking skeleton" concept from the early 2000s emphasizes this approach; more recently, MVP (minimum viable product) strategy adopts it for risk reduction.

Architecture enables accurate cost and schedule estimation. The project manager uses these estimates to acquire resources and monitor progress. Architects help create estimates early in the project lifecycle. While top-down estimates set goals and apportion budgets, bottom-up estimates based on understanding system pieces are typically more accurate. The organizational and work-breakdown structure is almost always based on architecture, so teams responsible for work items make more accurate estimates for their piece than project managers can, and feel more ownership. The best estimates emerge from consensus between top-down estimates created by architect and project manager and bottom-up estimates created by developers. This discussion and negotiation creates far more accurate estimates than either approach alone. Validated requirements improve estimate accuracy.

Architecture creates transferable, reusable models forming product-line hearts. Early-lifecycle reuse provides greater benefit than later reuse. While code reuse offers benefit, architectural reuse across systems with similar requirements provides tremendous leverage. Reusing architectural decisions transfers all early-decision consequences to those systems. A product line or family is a set of systems built using shared assets—components, requirements documents, test cases—with architecture designed to handle the entire family's needs. Product-line architects choose architecture (or closely related architectures) serving all envisioned product-line members. Architecture defines what is fixed and variable for all members. Product lines show order-of-magnitude payoffs in time to market, cost, productivity, and quality, with architecture's power at the paradigm's heart. Architectures for product lines become the developing organization's shared assets similar to capital investments.

Architecture-based development shifts focus from programming as the prime activity to composing and assembling independently developed elements. The architecture defines which elements can be incorporated and constrains possible replacements and additions according to interaction with environment, control reception and relinquishment, data consumption and production, data access, and communication and resource-sharing protocols. Commercial off-the-shelf components, open-source software, publicly available apps, and networked services are independent elements. Integrating many independently developed elements spawned an industry of software tools—Apache Ant, Apache Maven, MSBuild, Jenkins. Payoffs include decreased time to market, increased reliability, lower cost, and flexibility. Avoiding vendor lock-in where a single vendor provides an element at premium price is critical. Open systems define standards for software elements—behavior, interaction, data sharing—enabling many suppliers to produce elements. Open systems architecture defines elements and their interactions, exemplified by AUTOSAR developed for the automotive industry.

Architecture restricts the vocabulary of design alternatives to proven solutions, minimizing system design complexity. Software engineering emphasizes discipline, which comes partly from restricting alternative vocabulary to proven solutions. Examples include tactics and patterns and reusing off-the-shelf elements. Restricting design vocabulary to proven solutions yields enhanced reuse, more regular and simpler designs that are understood and communicated more easily with predictable outcomes, easier analysis with greater confidence, shorter selection time, and greater interoperability. Unprecedented designs are risky; proven designs are proven. Innovative solutions should be sought when existing solutions are insufficient, not for novelty's sake. Properties of software follow from architectural tactics or patterns choice. Desirable tactics or patterns for particular problems improve resulting design solutions, make arbitrating conflicting design constraints easier, increase insights into poorly understood design contexts, and surface requirements inconsistencies.

Architecture serves as the first introduction to the system for new project members, reinforcing its role in stakeholder communication and serving as a common reference point. Module views show project structure: who does what, which teams are assigned to which system parts. Component-and-connector views explain how the system works and accomplishes its job. Allocation views show where assigned parts fit into development or deployment environment.

## Chapter 3. Understanding Quality Attributes

The chapter establishes that functionality does not determine architecture—multiple architectures can satisfy identical functional requirements. Functionality is achieved by assigning responsibilities to architectural elements through module decomposition, but the structuring of these modules is driven by quality attributes, not by function alone. A monolithic design could satisfy all functional requirements; structural decisions exist to support modifiability, maintainability, scalability, security, performance, and other quality concerns.

The authors identify three critical problems in traditional quality attribute discussions: (1) definitions are not testable—a system may be "modifiable" with respect to one set of changes but not others; (2) overlapping ownership and categorization issues obscure architectural solutions (e.g., a denial-of-service attack is claimed by availability, performance, security, and usability communities); (3) each quality attribute community uses different vocabulary for the same phenomena (performance speaks of "events," security of "attacks," availability of "faults," usability of "user input").

Quality attribute scenarios solve these problems by providing a common, testable form for specifying all quality attributes. Each scenario has six components: stimulus (the motivating event or request), stimulus source (the origin of the stimulus), response (the activity the system or developers undertake in response), response measure (how success is measured—latency, throughput, labor hours, etc.), environment (the system state and operational context—normal operation, overload, degraded mode, development phase), and artifact (the target of the stimulus—a component, subsystem, or the whole system). The distinction between general scenarios (system-independent, reusable templates) and concrete scenarios (tailored to a specific system) enables stakeholders to elicit quality requirements more effectively by refining generic patterns rather than generating requirements from scratch.

Quality attributes fall into two categories: those describing runtime properties (availability, performance, usability) and those describing development-time properties (modifiability, testability, deployability). All quality attributes are interdependent; achieving portability through isolation of system dependencies introduces process or procedure boundaries that hurt performance. The architect's task is to understand these trade-offs and design accordingly.

Tactics are design decisions that directly influence achievement of a single quality attribute response; they control system behavior in response to stimuli. Patterns are recurring solutions to design problems and typically comprise multiple tactics, thus bundling several design decisions and frequently forcing trade-offs among quality attributes. Because no single pattern may fully solve a design problem, tactics provide a systematic means to augment or modify patterns for particular contexts. Refinement is necessary—Schedule resources must become a specific scheduling strategy (shortest-job-first, round-robin); Use an intermediary must be realized as a specific intermediary type (layer, broker, proxy, tier).

The chapter acknowledges that architectures often emerge through accumulated small decisions influenced by business forces and development pressures, leading to gradual degradation of quality attributes—a form of technical debt called architecture debt. Refactoring may address specific quality concerns: improving security by separating modules based on security properties, improving performance by eliminating bottlenecks, improving modifiability by factoring out duplicate functionality. Process-related decisions (employee security training, password policies) complement architecture decisions in achieving quality attributes.

Tactics-based questionnaires provide a practical analysis tool used at any stage of architectural maturity. For each tactic, analysts record: whether the tactic is supported by the architecture (Y/N), specific design decisions and code locations implementing the tactic, risk assessment (High/Medium/Low), and rationale explaining the decision and its implications for cost, schedule, and evolution. This lightweight technique typically requires 30–90 minutes per quality attribute and forces architects to consider the broader picture of how quality attribute decisions interact.

## Chapter 4. Availability

The chapter begins by establishing foundational terminology: a failure is the externally observable deviation of a system from its specification, while a fault is the cause of that failure, with errors being intermediate states between fault occurrence and failure manifestation. This distinction allows architects to discuss repair strategies—if a fault is detected and the system recovers without observable deviation from specified behavior, no failure occurs.

Availability is measured using the steady-state formula MTBF/(MTBF + MTTR), adapted from hardware reliability engineering but reinterpreted for software to emphasize thinking about failure modes, failure likelihood, and repair time. Scheduled downtimes are excluded from availability calculations per SLA specifications, which formalize availability expectations—for example, Amazon's EC2 commitment of 99.99% monthly uptime percentage—allowing calculation of acceptable outage windows (ranging from 8 seconds at 99.9999% availability to 3.6+ days at 99%).

The General Availability Scenario specifies six components: Source (where faults originate—internal/external from people, hardware, software, infrastructure, or environment), Stimulus (the fault itself, including omission, crash, incorrect timing, or incorrect response types), Artifact (system portions responsible for and affected by the fault), Environment (operational context from normal operation to degraded or overloaded modes), Response (desired system actions including prevention, detection with logging/notification, recovery, source disabling, temporary unavailability during repair, masking/containment, or degraded operation), and Response Measure (quantified success metrics like availability percentage, fault detection time, repair time, or degradation duration).

Availability tactics organize into three categories. Fault detection tactics include: Monitor (a component supervising system health across processors, processes, I/O, memory, and shared resources, orchestrating other detection tactics and capable of initiating self-tests or detecting missed heartbeats via watchdog counters); Ping/echo (asynchronous request/response pairs determining reachability and round-trip delay, with time thresholds defining failure timeout); Heartbeat (periodic message exchange between monitor and monitored process, with watchdog timer reset capability and piggybacking support for scalability); Timestamp (detecting incorrect event sequences in distributed message-passing systems using local clock state or sequence numbers); Condition monitoring (checking process/device conditions or design assumptions through checksums and related techniques); Sanity checking (validating reasonableness of operations or component outputs, typically at interfaces); Voting (comparing multiple sources' computational results with three schemes—Replication using exact clones effective against random hardware failures but not design/implementation errors; Functional redundancy adding design diversity to handle common-mode failures; Analytic redundancy allowing diverse inputs and outputs including multiple sensors like barometric pressure, radar altimeter, and geometric calculations in avionics); Exception detection (capturing conditions altering normal execution flow including system exceptions like divide-by-zero, parameter fence patterns using 0xDEADBEEF markers, parameter typing via type-length-value formatted base classes, and timeout mechanisms); and Self-test (procedures initiated by component or monitor using condition monitoring techniques like checksums).

Fault recovery tactics subdivide into preparation/repair and reintroduction categories. Preparation/repair tactics include: Redundant spare (duplicate components stepping in upon primary failure, foundational to hot spare, warm spare, and cold spare patterns); Rollback (reverting to previous known good state or "rollback line" upon failure detection, depending on checkpoints stored at fixed intervals or significant processing moments); Exception handling (responding to detected exceptions beyond crashing, using function return codes or exception classes containing name, origin, and cause information for fault masking or repair); Software upgrade (achieving non-service-affecting in-service code updates via Function patch using incremental linker/loader storing updated functions in pre-allocated memory, Class patch adding member data/functions via runtime backdoor mechanisms in OOP, or Hitless ISSU leveraging redundant spare tactics); Retry (assuming transient faults and retrying operations with retry limits, common in networks and server farms); Ignore faulty behavior (discarding messages from sources determined spurious); Graceful degradation (maintaining critical functions while dropping less critical ones); and Reconfiguration (reassigning responsibilities to remaining functioning resources).

Reintroduction tactics reintegrate repaired components: Shadow (operating failed or upgraded components in shadow mode for predefined duration, monitoring behavior and incrementally repopulating state); State resynchronization (sending state information from active to standby components organically with active redundancy or periodically with passive redundancy via checkpointing, comparing states through checksums or cryptographic hashes); Escalating restart (varying restart granularity across four levels—Level 0 killing/recreating child threads with warm spare impact, Level 1 freeing unprotected memory, Level 2 freeing all memory forcing application reload, Level 3 completely reloading executable images); and Nonstop forwarding (originating from router design, splitting functionality into control plane managing routing information and data plane forwarding packets, allowing continued packet forwarding during routing protocol recovery).

Fault prevention tactics include: Removal from service (temporarily placing components out-of-service to scrub latent faults like memory leaks or soft errors before accumulation reaches service-affecting levels, called software rejuvenation or therapeutic reboot); Transactions (leveraging ACID—atomic, consistent, isolated, durable—properties through two-phase commit (2PC) protocol preventing race conditions in distributed asynchronous message exchanges); Predictive model (combining monitoring with state-of-health supervision ensuring nominal operating parameters, monitoring metrics like session establishment rates, threshold crossings on constrained shared resources, process state statistics, and queue lengths to predict fault onset triggering corrective action); Exception prevention (employing error-correcting code, smart pointers doing bounds checking and automatic resource deallocation preventing dangling pointers and leaks, and wrappers preventing semaphore violations); and Increase competence set (designing components to handle more cases—more fault states—as normal operation, expanding states in which components are "competent," exemplified by waiting for access versus throwing exceptions).

The chapter presents a Tactics-Based Questionnaire (Table 4.3) with rows for each tactic group's questions regarding support, risk, design decisions, location, and rationale, enabling systematic availability architecture assessment.

Availability patterns complement tactics. Three redundancy patterns differ in backup state synchronization: Active redundancy (hot spare) configures all nodes in protection groups receiving and processing identical inputs in parallel, maintaining synchronous spare state enabling millisecond takeover (one-plus-one redundancy as simplest case), highest cost but fastest recovery; Passive redundancy (warm spare) where only active members process traffic with periodic state updates to loosely-coupled spares balancing active redundancy's cost against cold spare's simplicity; Spare (cold spare) keeping spares out-of-service until failover triggers power-on-reset procedures, poorly suited to high-availability requirements due to poor recovery performance and high MTTR. Triple modular redundancy (TMR) widely implements voting using three identical components each receiving identical inputs with voting logic detecting inconsistencies and deciding output through majority rule or averaging, simple to understand and implement though tradeoffs exist between replication level, cost, and availability—three components represent the sweet spot. Circuit breaker prevents endless retry cycles by breaking retry loops when system faults are detected, returning immediately without passing service requests until reset, removing retry policy burden from individual components and preventing cascading failures in distributed systems where multiple callers could collectively fail. Additional patterns include Process pairs employing checkpointing and rollback with backups ready for takeover, and Forward error recovery moving systems to desirable states using error-correction capabilities like data redundancy, avoiding fallback to previous states or retry.

## Chapter 5. Deployability

The chapter opens by distinguishing between continuous deployment (fully automated, no human intervention) and continuous delivery (automated up to the final production step, which requires human approval, often for regulatory or policy reasons). The deployment pipeline is the sequence starting from code commit to version control through automated integration, testing, and eventual production deployment, with each stage occurring in isolated environments designed for specific testing concerns.

The four primary environments are development (where single modules undergo unit testing), integration (where a continuous integration server compiles the service with compatible versions of other components and performs unit and integration tests), staging (where system-wide testing including performance, security, and license compliance occur, plus user testing and field testing for embedded systems), and production (where the service runs with close monitoring until confidence is established).

Three critical measures of pipeline quality are cycle time (pace of progress enabling hundreds of deployments per day), traceability (ability to recover all artifacts, code versions, dependencies, test cases, and tools involved in an element—typically maintained in an artifact database), and repeatability (achieving identical results from identical actions and artifacts, complicated by dynamic dependency fetching and test state management).

Architectural choices profoundly affect deployability. Microservice architecture allows independent technology choices per team, eliminating integration-time incompatibilities and enabling independent deployments. Continuous deployment mindset necessitates designing testing infrastructure early and making architectural decisions about feature toggles and interface backward compatibility. Virtualization and cloud computing have transformed the physical environments into virtual ones with environment parity, where all environments share fundamental structure and hardware types despite differing in scale.

The chapter identifies deployability as a property enabling allocation to an environment within predictable, acceptable time and effort, with rollback capability equally important. Key deployment concerns include mechanism of arrival (push vs pull), integration while executing (in-service vs offline), delivery medium, packaging format, integration approach, execution efficiency, and process controllability.

Architects must ensure deployments are granular (deployable in whole or selective parts to reduce risk), controllable (at varying levels with monitoring and rollback capability), and efficient (rapid deployment and rollback with reasonable effort).

Six core tactics emerge from the deployment pipeline framework: Scale rollouts deploys new versions gradually to controlled user subsets using routing mechanisms (the old version continues serving other users), minimizing impact of flawed deployments. Rollback reverses deployments if defects or unmet expectations are discovered, requiring tracking of multiple coordinated service and data updates. Script deployment commands automates complex, multi-step deployments treated as versioned, reviewed, tested code. Manage service interactions mediates simultaneous execution of multiple service versions, preventing version incompatibilities while avoiding complete resource replication. Package dependencies bundles elements with libraries, OS versions, and utility containers (sidecars, service mesh) so versions remain consistent across development-to-production transitions. Feature toggle implements runtime kill switches disabling problematic new features without redeployment, providing control without the cost and risk of rolling back entire services.

Microservice architecture patterns services as independently deployable, message-communicating units with no shared memory or interprocess communication. Services are typically stateless, relatively small, and developed by single small teams; service dependencies are acyclic; discovery services route messages appropriately. Benefits include reduced time to market (independent deployments without team coordination), technology choice freedom per team (reducing integration errors), improved scaling (dynamic instance addition matches supply to demand). Tradeoffs include network overhead from message-based communication (mitigated somewhat by service mesh colocation), difficulty with complex transactions across distributed systems, organizational burden of maintaining multiple technologies, difficulty maintaining intellectual control of the total system, and the significant design challenge of granular service decomposition.

Complete replacement strategies maintain N running instances during service replacement. Blue/green deployment creates N new instances (green), installs the new service version, switches DNS or discovery service routing, and only removes original instances (blue) after confirming satisfactory operation, allowing simple rollback before cutoff. Rolling upgrade replaces instances one at a time: allocate resources, install and register the new version, begin directing requests to it, allow the old instance to complete processing and destroy it, repeat until all old instances are replaced. Netflix implements this via Asgard on AWS EC2. Blue/green benefits from completely replacing versions (no temporal inconsistency), while rolling upgrade uses fewer peak resources (N+1 vs 2N instances). Tradeoffs for blue/green include higher resource requirements and potential longer rollback time if errors are discovered after original instances are deleted. Rolling upgrade introduces temporal inconsistency (requests to the same service routed to different versions, producing inconsistent results unless managed by service interaction mediation) and interface mismatch (incompatible interface changes breaking unadjusted clients unless prevented by extending but not modifying interfaces with mediator pattern translation).

Partial replacement patterns support quality control and marketing tests. Canary testing routed a small user set (power users, preview-stream users, or internal testers like Google employees) to new releases before general rollout, allowing real-world testing in production with minimal exposure if defects emerge. Dark launch variant focuses on feature acceptance. A/B testing routes different user groups to different versions (minor UI changes like font size through major feature variants), monitoring results to determine which alternative yields best business outcomes. HomeAway varied website format and content across regions to track rental generation; Google famously tested 41 shades of blue for search results. Both leverage DNS and discovery-service configuration for routing and monitoring.

## Chapter 6. Energy Efficiency

Energy has transitioned from a free and unlimited resource to a critical architectural concern due to the energy demands of data centers (now consuming an estimated 3-10 percent of global energy), the battery constraints of mobile devices as primary computing platforms, and the proliferation of IoT devices in industrial and governmental applications. The authors identify three fundamental reasons why architectural approaches are necessary: first, system-wide monitoring and management mechanisms are essential to prevent ad-hoc, unmaintainable solutions; second, energy efficiency is rarely taught in computer science curricula, leaving developers and architects without proper understanding; third, design concepts—models, patterns, and tactics—for energy efficiency are still in their infancy with no established catalogs.

The chapter formalizes energy efficiency through a general scenario framework with defined portions: Source (who initiates energy requests: end user, manager, administrator, automated agent), Stimulus (the energy conservation request: total usage, maximum instantaneous, average, etc.), Artifacts (what is managed: devices, servers, VMs, clusters), Environment (runtime conditions: connected, battery-powered, low-battery mode, power-conservation mode), Response (actions taken: disable services, deallocate runtime services, change service allocation, reduce consumption mode, allocate/deallocate servers, change service levels, change scheduling), and Response measure (energy saved in kilowatt loads, kilowatt hours, or powered-on duration while maintaining required functionality).

Energy efficiency tactics are organized into three broad categories focused on effective resource utilization:

Monitor Resources comprises three tactics: Metering collects actual energy consumption data via sensor infrastructure in near real time, from coarse-grained data center measurement via power meters to fine-grained individual device measurement via external amp or watt-hour meters, metered rack PDUs (power distribution units), ASICs (application-specific integrated circuits), or battery management systems in battery-operated devices. Static classification estimates energy consumption when real-time data collection is infeasible (such as in off-premises cloud scenarios) by cataloging computing resources and their known energy characteristics from benchmarks or manufacturer specifications. Dynamic classification addresses cases where static models are inadequate by estimating energy consumption based on transient conditions like workload, employing simple table lookups, regression models from prior execution data, or simulation.

Allocate Resources encompasses three tactics: Reduce usage decreases energy at the device level through device-specific activities such as reducing display refresh rates or darkening backgrounds; at the system level by removing or deactivating resources when no longer needed (spinning down hard drives, turning off CPUs, running CPUs at slower clock rates, shutting down unused processor blocks, or consolidating VMs onto minimum physical servers with idle shutdown); or at the application level by offloading computation from energy-constrained mobile devices to cloud back-ends when communication energy is lower than local computation energy. Discovery services, used in microservice and cloud architectures, traditionally match service requests to providers based on API descriptions; they can be extended to annotate requests with energy information, enabling requestors to select service providers based on dynamic energy characteristics, populating a "green service directory" with information from resource monitoring tactics. Schedule resources allocates computational tasks to resources while respecting task constraints and priorities, using data collected from monitoring tactics to dynamically switch tasks among service providers (or among CPU cores in multi-core systems) to select the ones offering better energy efficiency or lower energy costs; for example, a lightly-loaded provider can adapt its energy usage and consume less energy per unit of work.

Reduce Resource Demand tactics (detailed in Chapter 9) complement the allocation tactics by directly reducing work rather than reallocating existing demand: managing event arrival, limiting event response, prioritizing events (potentially letting low-priority events go unserviced), reducing computational overhead, bounding execution times, and increasing resource usage efficiency.

The chapter presents three architectural patterns for energy efficiency: Sensor Fusion uses low-power sensors to infer whether higher-power sensors need activation, exemplified by accelerometer data determining if GPS updates are required; it benefits from intelligent minimization of energy-intensive device usage but trades off added complexity and risks higher overall energy if inference frequently triggers the expensive sensor. Kill Abnormal Tasks monitors the energy consumption of applications with unknown power characteristics and interrupts or kills exceptionally power-hungry operations when they persist without user response (such as audible alerts and vibration continuing beyond a timeout period); it provides a fail-safe for managing rogue apps but adds monitoring overhead and raises usability concerns about unexpected task termination. Power Monitor automatically disables devices and interfaces not actively used by the application, minimizing time devices are active through techniques long used in integrated circuit design to shut down inactive blocks; it enables intelligent power savings with minimal user impact but introduces startup latency and energy costs when devices are reactivated, and requires system knowledge of each device's energy characteristics, adding design complexity.

## Chapter 7. Integrability

Integrability encompasses more than merely making separately developed components cooperate; it requires managing the costs and technical risks of anticipated and unanticipated future integration tasks. The authors present a formal problem statement: integrating a component C or set of components {C₁, C₂, …, Cₙ} into a system S, where S may be a platform or an existing system, and the level of control over {Cᵢ} may vary (they may be supplied by external vendors).

The chapter identifies integration difficulty as a function of size (number of potential dependencies between {Cᵢ} and S) and distance (the difficulty of resolving differences at each dependency). Dependencies extend beyond syntactic relations (calls, inheritance, usage) to include temporal coupling (components sharing and competing for finite resources like memory, bandwidth, CPU) and semantic coupling (shared knowledge of protocols, file formats, units of measure, metadata). The critical distinction is that temporal and semantic dependencies are often implicit and poorly documented, creating significant risks for long-lived projects.

The authors introduce five specific dimensions of "distance" that characterize integration challenges:

Syntactic distance occurs when cooperating elements disagree on the number and type of data elements being shared (e.g., integer vs. floating point, or misinterpreted bit masks). Such mismatches are typically observable and predictable via compiler checks, though subtle bit-level discrepancies require code scrutiny.

Data semantic distance arises when two elements share the same data type but interpret values differently (e.g., altitude in meters vs. feet). These mismatches are difficult to observe without metadata or documentation and are typically discovered through interface documentation comparison or code review.

Behavioral semantic distance occurs when cooperating elements disagree on system behavior, particularly regarding states and modes (startup, shutdown, recovery) or control assumptions (which component initiates interactions). Some behavioral protocols are explicitly captured; others remain implicit.

Temporal distance refers to disagreements about timing assumptions—elements operating at different rates (e.g., 10 Hz vs. 60 Hz) or making different latency assumptions (e.g., event A must follow event B within 50 ms). This is treated separately from behavioral semantics due to its importance and subtlety.

Resource distance involves misalignment on shared resource assumptions—exclusive vs. shared device access, memory availability, or communication channel capacity (e.g., three elements producing 3 Mbps each on a channel with 5 Mbps capacity).

The chapter emphasizes that integrability is a form of planning for modifiability: architectures must be designed to accommodate anticipated integrations, reducing their cost and risk.

The General Scenario for Integrability (Table 7.1) specifies the structure for describing integration concerns: Source (mission/system stakeholder, component marketplace, vendor), Stimulus (add new component, integrate new version, integrate existing components in new way), Artifact (entire system, specific components, metadata, configuration), Environment (development, integration, deployment, runtime), Response (changes completed/integrated/tested/deployed, components exchanging information correctly, collaborating, not violating resource limits), and Response measure (cost in components changed, code percentage, lines of code, effort, money, calendar time; or effects on other quality attributes).

The Integrability Tactics are organized into three categories with specific mechanisms:

Limit Dependencies tactics reduce the number of potential dependencies. Encapsulation introduces explicit interfaces and ensures all access passes through them, reducing the probability that changes propagate to other elements by limiting both the number of dependencies and their distances. Encapsulation may hide irrelevant interfaces (e.g., internal libraries) and reduce syntactic, data, and behavior semantic distances. Use an Intermediary breaks dependencies between {Cᵢ} and S through mechanisms like publish-subscribe buses, shared data repositories, dynamic service discovery, data transformers, and protocol translators. Intermediaries reduce the number of dependencies by decoupling producers and consumers and resolve syntactic and data semantic distance. Restrict Communication Paths controls which elements can communicate by limiting visibility (making interfaces unseen) and authorization (restricting access). This tactic appears in Service-Oriented Architectures (SOAs) through enterprise service buses that force all requests through a central routing mechanism for consistent preprocessing. Adhere to Standards leverages standards from recognized organizations (IEEE, ISO, OMG) or organizational conventions to prescribe syntax, data semantics, behavioral/temporal semantics, and protocols. Standards reduce the number of potential dependencies and may address syntactic, data semantic, behavioral semantic, and temporal dimensions of distance. Abstract Common Services hides similar but non-identical services behind a common abstraction (either a common interface or an intermediary translator), allowing future components to integrate with a single abstraction rather than separately integrating with each specific element. When combined with intermediaries like wrappers or adapters, this tactic normalizes syntactic and semantic variations (e.g., different sensor manufacturers with different drivers and timing properties behind a common interface).

Adapt tactics reduce the expected distance between components. Discover provides a catalog service for translating addresses, dynamically binding targets, or handling multiple targets. Registration may be static (DNS) or dynamic (upon service instantiation); entries may be de-registered statically or dynamically via health checks. A discovery service may contain nested discovery services and include attributes for query refinement (e.g., cost of forecast). The tactic reduces dependencies by enabling services to function without knowledge of each other, providing flexibility in binding timing. Tailor Interface adds or hides capabilities in an existing interface without changing the API or implementation—techniques include translation, buffering, data smoothing, function hiding (for untrusted users), intercepting filters (SQL injection prevention, data format translation), and aspect-oriented programming. This tactic is typically applied during integration and resolves syntactic and data semantic distance; it may also address behavioral semantic distance but is then more accurately categorized as introducing an intermediary. Configure Behavior allows components to be configurable at build time (recompile with flags), initialization time (configuration files, databases), or runtime (protocol version in requests). Supporting multiple standard versions increases the probability that assumptions of S and future C components will match. This tactic can address syntactic, data semantic, behavioral semantic, and temporal dimensions of distance.

Coordinate tactics manage the integration of multiple components. Orchestrate uses a control mechanism to coordinate service invocation while keeping services unaware of each other, reducing the number of dependencies between S and new {Cᵢ} and eliminating explicit dependencies among {Cᵢ} by centralizing them at the orchestration mechanism. Workflow engines use this tactic through organized activities that order and coordinate components to complete business processes; these may be nested and aggregated. Business Process Management (BPM) treats processes as competitive assets. Complex orchestration can be specified in BPEL (Business Process Execution Language). Manage Resources is a specific intermediary that governs access to computing resources (threads, memory blocks); components request resources from the manager rather than accessing directly. Resource managers allocate access while preserving invariants (avoiding exhaustion, concurrent use) and enforcing fair access policies. Examples include operating systems, database transaction mechanisms, thread pools, and the ARINC 653 standard for space/time partitioning in safety-critical systems. This tactic reduces resource distance by exposing requirements and managing their common use.

The Service-Oriented Architecture (SOA) pattern describes distributed components providing and/or consuming services, where service providers and consumers can use different implementation languages and platforms. Services are largely standalone; providers and consumers are usually deployed independently and may belong to different organizations or systems. Components have interfaces describing requested and provided services; service quality is specified and guaranteed via Service Level Agreements (SLAs). Communication typically uses WSDL (Web Services Description Language) or SOAP (Simple Object Access Protocol). SOA differs from microservices in that SOAs provide reusable components assumed to be heterogeneous and managed by distinct organizations, whereas microservices compose a single system managed by one organization. Benefits include genericity (services designed for variety of clients), independence (accessible only through interfaces and network messages), and heterogeneous implementation. Tradeoffs include added complexity and overhead from interoperability features like WSDL and SOAP.

Patterns related to the tailor interface tactic include Wrappers (encasing a component in an alternative abstraction, transforming data/control for it while serving as the single access point); Bridges (independently translating "requires" assumptions of one component to "provides" assumptions of another, explicitly invoked by external agents, typically transient, addressing narrower interface translation ranges); and Mediators (combining properties of bridges and wrappers, incorporating a planning function for runtime-determined translation and acting as explicit first-class architectural components with semantic complexity and runtime persistence). All three patterns allow access to elements without forcing changes to the element or interface, though they require up-front development work and introduce performance overhead.

The Dynamic Discovery pattern applies the discovery tactic to enable service provider discovery at runtime, enabling runtime binding between consumer and concrete service. The system must clearly advertise available services and minimal information per service, typically including mechanically searchable data (interface standard versions). Benefits include flexibility in binding services based on pricing or availability at startup or runtime. Tradeoffs require automated registration/de-registration and tool acquisition or generation.

## Chapter 8. Modifiability

Modifiability is fundamentally about change, with the goal of lowering the cost and risk of making changes. The authors establish four critical planning questions: (1) What can change addresses functions, platform, environment, qualities, and capacity; (2) What is the likelihood of the change recognizes that not all potential changes can be economically planned for; (3) When is the change made and who makes it distinguishes between source code changes by developers and runtime changes by end users, administrators, or even the system itself through learning mechanisms; (4) What is the cost of the change involves two components—the cost of introducing modification mechanisms and the cost of exercising them. The chapter presents a justification equation: N * Cost of making change without mechanism ≤ Cost of creating mechanism + (N * cost of making change using mechanism), where N is the anticipated number of modifications. This equation acknowledges that building sophisticated change-handling mechanisms can lead to technical debt if the predicted number of changes does not materialize, and involves opportunity costs that must be weighed against other investments.

The chapter defines several important flavors of modifiability: Scalability (horizontal/scaling out through adding logical units like servers, and vertical/scaling-up through adding resources to single machines; elasticity in cloud environments); Variability (ability to support production of system variants in product lines by sharing engineering assets); Portability (ease of moving software between platforms by minimizing platform dependencies, isolating them, or running on virtual machines); Location Independence (handling distributed systems where service location is unknown or changes at runtime, requiring dynamic discovery mechanisms).

The chapter establishes four fundamental parameters affecting modifiability costs: module size (larger modules are more difficult and costly to change), cohesion (how strongly a module's responsibilities are related; high cohesion is desirable as it reduces the probability that a change affects multiple modules), coupling (the overlap and interdependency between modules; high coupling is an enemy of modifiability, measured as the probability a modification to one module will propagate to another), and binding time of modification (the phase in the software development lifecycle when values are bound; later binding is preferred but requires preparation costs).

The tactics for modifiability address these parameters: Increase Cohesion tactics include Split module (refactoring non-cohesive modules into more cohesive submodules) and Redistribute responsibilities (gathering similar responsibilities sprinkled across modules and placing them together or creating new modules). Reduce Coupling tactics include Encapsulate, Use an intermediary (placing intermediaries between tightly coupled modules), Abstract common services (used for portability and environmental variations), and Restrict dependencies (implemented through visibility constraints and authorization, seen in layered architectures and wrapper patterns). Defer Binding tactics allow binding values at different lifecycle phases: compile-time and build-time binding through Component replacement, Compile-time parameterization, and Aspects; deployment/startup/initialization-time binding through Configuration-time binding and Resource files; and runtime binding through Discovery, Interpret parameters, Shared repositories, and Polymorphism. The concept of externalizing changes separates the building of modification mechanisms from their use, allowing different stakeholders to participate at different lifecycle phases.

Four major architectural patterns support modifiability: Client-Server establishes dynamic connection between multiple distributed clients and servers with low coupling, scalability to additional clients, independent evolution, shared services, and isolation of user interaction to clients; however, network-based communication introduces performance degradation and security/integrity concerns. Plug-in (Microkernel) pattern consists of core functionality with fixed-interface plug-ins for extension, enabling controlled extension, separate development by different teams/organizations, and independent evolution of plug-ins and microkernel, but introduces security vulnerabilities from third-party plug-ins. Layers pattern partitions software into layers with cohesive service sets and unidirectional allowed-to-use relationships, preventing upward usage and sometimes allowing layer bridging; benefits include reusability of lower layers, reduced interface complexity, and change isolation, but introduces performance penalties through traversal of multiple layers and may hinder portability if layer bridging occurs excessively. Publish-Subscribe pattern uses asynchronous event-based communication through publishers, subscribers, and an event bus, achieving loose coupling where publishers and subscribers are independent, system behavior changes through event/topic modifications, and events can be logged for record and playback; tradeoffs include performance/latency impacts, difficulty in reasoning about performance and resource management, reduced determinism compared to synchronous systems, testability challenges when event bus associations change, and limited end-to-end encryption mechanisms since publishers and subscribers do not know each other's identity.

## Chapter 9. Performance

The chapter opens by establishing that performance is fundamentally about time—operations take predictable durations (CPUs in nanoseconds, disk access in milliseconds, network access in microseconds to hundreds of milliseconds)—and systems must respond to events within required time windows. The general performance scenario framework has six components: Source (user, external system, internal component, timer), Stimulus (periodic, sporadic, or stochastic event arrivals), Artifact (whole system or component), Environment (operating mode: normal, emergency, error correction, peak load, overload, degraded, etc.), Response (system returns result, error, no response, ignores on overload, changes service level, prioritizes higher-priority events, consumes resources), and Response Measure (latency, throughput, jitter, unsatisfied request percentage, resource utilization). The authors emphasize concurrency as a critical but complex concern: concurrent operations exploit parallelism to improve performance by allowing one thread's delays to be masked by processor work on other threads, but race conditions arise when shared state exists, requiring careful management through locks or state partitioning. A concrete example establishes the scenario: "Five hundred users initiate 2,000 requests in a 30-second interval, under normal operations. The system processes all of the requests with an average latency of two seconds." Response time consists of two components: processing time (when the system actively consumes resources on computation) and blocked time (when processing is suspended due to resource contention, unavailability, or dependency on other computations). The response time to an event is partitioned based on how the system's resources behave under saturation—CPUs degrade steadily as load increases, while memory performance crashes suddenly when page swapping becomes overwhelming. The tactic framework divides into two categories. Control Resource Demand tactics reduce the work entering or processed by the system: Manage Work Requests includes manage event arrival (via Service Level Agreements specifying maximum event arrival rates and response times), manage sampling rate (reducing input frequency from sensors or video streams to maintain predictable latency, accepting reduced fidelity), limit event response (processing events only up to a maximum rate and queueing or discarding excess), and prioritize events (servicing high-importance events before low-priority ones, ignoring low-priority events when resources are insufficient). Reduce Computational Overhead includes reducing indirection (removing intermediaries, though this trades modifiability for performance; some brokers allow direct communication after initial connection), co-locating communicating resources (avoiding network communication delays, context switching costs, and subroutine call expenses by placing components on the same processor or runtime), and periodic cleaning (recalculating and reinitializing hash tables and virtual memory maps to prevent efficiency degradation). Additional control tactics include bound execution times (limiting iterations in data-dependent algorithms to sacrifice computation accuracy for predictability) and increase efficiency of resource usage (improving algorithm performance in critical code paths). Manage Resources tactics enhance how available resources handle demand: Increase Resources (faster processors, additional processors, memory, networks—often the cheapest immediate solution), Introduce Concurrency (processing different event streams or activities on different threads to reduce blocked time and exploit parallelism), Maintain Multiple Copies of Computations (load balancers distribute requests across replicated services using policies like round-robin or least-busy-server assignment, reducing contention), and Maintain Multiple Copies of Data (data replication reduces simultaneous access contention with synchronization responsibilities, while caching keeps frequently accessed data on faster storage with different memory hierarchies or local versus remote access speeds; caches can be reactive—storing recent requests—or predictive—prefetching based on behavioral patterns). Bound Queue Sizes limits queued arrivals and resource consumption, paired with overflow policies. Schedule Resources manages contention through scheduling policies: First-in/first-out (FIFO) treats all requests equally but may suffer head-of-line blocking; Fixed-Priority Scheduling assigns static priorities by semantic importance, deadline monotonic (shorter deadlines receive higher priority), or rate monotonic (shorter periods receive higher priority for periodic streams); Dynamic Priority Scheduling includes round-robin (cyclic assignment), earliest-deadline-first (optimal for single processor with preemptible processes), and least-slack-first (prioritizes jobs with smallest remaining slack time). Static scheduling (cyclic executive) determines preemption points offline, eliminating scheduler overhead. Patterns instantiate these tactics: Service Mesh uses sidecars—proxy processes accompanying each microservice—to handle interservice communication, monitoring, and cross-cutting concerns (authentication, service discovery, load balancing, encryption, observability), separating business logic from infrastructure concerns; benefits include separation of concerns and reduced latency from co-location, while tradeoffs include sidecar processing overhead and functions included that may not be needed. Load Balancer acts as a single contact point (IP address) distributing requests across a pool of providers using scheduling algorithms from simple round-robin to load-aware assignment, isolating client failures, reducing predictable latency, and allowing easy resource addition; tradeoffs include algorithm performance requirements and the load balancer itself becoming a bottleneck or single point of failure (thus often replicated). Throttling packages manage work requests as an intermediary throttler that monitors and controls service access, gracefully handling demand variations to prevent overload; tradeoffs include throttler performance criticality, potential for large buffers or request loss if demand exceeds capacity, and difficulty adding to tightly coupled legacy systems. Map-Reduce addresses distributed analysis of massive datasets through parallel processing: the infrastructure allocates work to hardware nodes and handles sorting; the programmer writes map (hashing data into buckets by key and filtering records, executed in parallel by many instances on partitioned input) and reduce (performing analysis on bucketed data from shuffled map output, emitting aggregated results) functions. Map instances are stateless with no intercommunication; only communication is \<key, value\> pair output from map to reduce. Benefits include exploitation of parallelism for unsorted large datasets and fault tolerance (single instance failure affects only its partition). Tradeoffs include overhead unjustified for small datasets, performance loss if data cannot be evenly partitioned, and complexity orchestrating multi-reduce operations.

## Chapter 10. Safety

Safety concerns systems avoiding entry into states causing damage, injury, or death. Unsafe states arise from omissions (events that fail to occur), commission (spurious undesirable events), timing failures (early or late occurrences), incorrect system values (coarse, detectable errors versus subtle, undetectable ones), sequence omissions/commissions (missing or unexpected events in sequences), and out-of-sequence events. The authors cite foundational disasters: the 2009 Shushenskaya turbine accident, Therac 25 radiation overdoses, Ariane 5 explosion, Stuxnet, the 1983 Soviet false-alarm missile incident, Air France 447 (where ice-clogged airspeed sensors combined with confusing stall-warning software led pilots to perform recovery actions that worsened the stall), and Boeing 737 MAX crashes caused by the MCAS software relying on a single sensor without testing under failure conditions. These examples establish that the connection between software and the physical world—via actuators—is the bridge where 0s and 1s translate into mechanical action that can kill.

Once an unsafe state is detected, the system must either continue operations after recovering, place itself in a safe mode, or shut down (fail safe). Alternatively, the system may transition to manual operation. The unsafe state must be reported immediately and logged.

Safety tactics are categorized into four groups: unsafe state avoidance, unsafe state detection, containment, and recovery. Unsafe state avoidance includes substitution (replacing dangerous software design features with safer, often hardware-based mechanisms like watchdogs, monitors, and interlocks, beneficial only when the function replaced is relatively simple) and predictive model (monitoring system health and processes to predict problems and provide early warning, exemplified by automotive cruise control systems calculating closing rates to warn drivers before collision risk becomes critical). Unsafe state avoidance relies on the assumption that hazards have been identified beforehand through rigorous analysis.

Unsafe state detection employs multiple tactics. Timeout detects whether component operations meet timing constraints by raising exceptions if constraints are violated; it particularly suits real-time and embedded systems. Timestamp detects incorrect event sequences in distributed message-passing systems by assigning local clock values to events. Condition monitoring checks process and device conditions or validates design assumptions (ideally via simple, provable assertions) to identify hazardous states; the monitor must remain simple to avoid introducing new errors. Sanity checking validates the reasonableness of operation results, inputs, and outputs, typically deployed at interfaces. Comparison detects unsafe states by comparing outputs from synchronized or replicated elements; with three or more replicants, comparison can identify which component caused the unsafe state, potentially replacing voting with simple shutdown if outputs diverge.

Containment tactics limit harm from unsafe states already entered. Redundancy subcategory includes replication (clones protecting against random hardware failures but vulnerable to design/implementation errors since no diversity exists), functional redundancy (implementing design diversity to address common-mode failures where replicas share the same implementation and thus fail identically), and analytic redundancy (enabling higher-level diversity at input/output level, often partitioning the system into high-assurance-simple and high-performance-complex portions, allowing tolerance of specification errors through separate requirement specifications). Limit consequences subcategory includes abort (stopping operations determined unsafe before damage), degradation (maintaining critical functions while dropping or replacing others in controlled, planned ways—e.g., a GPS-dependent car navigation system switching to dead reckoning in tunnels), and masking (comparing redundant component results and voting when outputs differ; the voter must be simple and reliable). Barrier subcategory includes firewall (limiting access to resources like processors, memory, and network connections) and interlock (protecting against incorrect event sequencing by controlling all access to protected components).

Recovery tactics place systems into safe states. Rollback reverts to a saved previous known-good state upon failure detection, often combined with checkpointing and transactions; after reaching the good state, execution continues using tactics like retry or degradation. Repair state repairs erroneous states (like lane-keep assist returning a drifting vehicle between lane lines) and continues execution, though it is inappropriate for unanticipated faults. Reconfiguration recovers from component failures by remapping logical architecture onto remaining functioning resources, potentially maintaining full or partial functionality via combination with degradation.

The Safety General Scenario (Table 10.1) defines sources (sensors, software components, communication channels, clocks, user actions), stimuli (omissions, commissions, incorrect data/timing), environments (normal, degraded, manual, recovery modes), artifacts (safety-critical system portions), and responses (avoid unsafe states, recover, continue in degraded/safe mode, shut down, switch to manual/backup, notify entities, log events). A concrete example: a patient monitoring system detects a sensor failure to report a life-critical value after 100 ms, logs the failure, illuminates a warning light, and engages a backup lower-fidelity sensor within 300 ms total. Response measures quantify percentage of avoided unsafe entries, percentage of recoverable unsafe states, risk exposure (loss magnitude × loss probability), time in degraded/safe modes, shutdown time, and recovery elapsed time.

Safety patterns leverage availability patterns while addressing safety-specific concerns. Redundant sensors replicate critical sensor data with independent software monitoring each sensor (redundant spare tactic applied to safety-critical hardware), guarding against single-sensor failure at the cost of complexity and processing overhead. Monitor-actuator separates actuator controller computations from monitoring/validation of those values before sending to physical actuators, creating redundant checking but requiring development and maintenance effort; the tradeoff allows flexibility in monitor sophistication. Separated safety divides systems into safety-critical and non-safety-critical portions to reduce certification costs and focus effort, though the separation itself requires expensive work (e.g., dual networks for partitioned messages) and must be certified as influencing nothing between portions.

The text introduces Design Assurance Levels (DAL) from DO-178C, categorizing software functions by failure impact: A (Catastrophic—deaths, loss of aircraft), B (Hazardous—large safety impact or serious injuries), C (Major—significant safety margin reduction), D (Minor—slight safety margin reduction), E (No effect). DAL guides allocation of limited validation and testing resources; critical flight-control software receives far more testing than entertainment systems. Similar frameworks include Safety Integrity Levels (SILs) under IEC 61508 (four levels, SIL 4 most dependable), domain instantiations like IEC 62279 for railways, and emerging standards such as ANSI/UL 4600 for autonomous vehicles.

## Chapter 11. Security

Security is defined as the system's ability to protect data and information from unauthorized access while providing access to authorized users and systems. Attacks take multiple forms: unauthorized access attempts, data modification, service denial, and behavior alteration. The CIA model forms the conceptual foundation: confidentiality protects data from unauthorized viewing (e.g., preventing access to tax returns), integrity ensures data remains unmanipulated (e.g., grades unchanged), and availability maintains service access for legitimate users (e.g., preventing DoS attacks from blocking online purchasing).

The chapter introduces attack trees as threat modeling technique, similar to fault trees, with successful attacks as roots and possible causes as decomposable nodes. Attack leaves represent stimuli in security scenarios; responses aim to preserve CIA or deter attackers through monitoring. Privacy, addressed in a sidebar, relates to personally identifiable information (PII)—defined by NIST as any information distinguishing or traceable to individuals or linked to them through medical, educational, financial, or employment records. Architects may verify PII is hidden from team members lacking access needs.

Security General Scenario formalizes evaluation with five components: Source (human or system, internal/external, identified or unknown); Stimulus (unauthorized attempts to display, capture, change/delete data, access services, change behavior, or reduce availability); Artifact (system services, internal data, components, or produced/consumed data); Environment (online/offline, connected/disconnected, firewalled/open, fully/partially/non-operational); Response (maintaining CIA, protecting from unauthorized access, preventing unauthorized manipulation, identifying parties with assurance, preventing repudiation, ensuring availability, tracking activities through logging and notifications); Response Measure (frequency of successful attacks, detection time, recovery duration, data vulnerability scope).

Security tactics organize into four categories inspired by physical security parallels:

Detect Attacks includes four tactics. Detect intrusion compares network traffic and service request patterns against signature databases or known malicious behavior patterns based on protocol characteristics, request signatures, payload sizes, applications, source/destination addresses, or port numbers. Detect service denial compares incoming traffic patterns to historical profiles of known denial-of-service attacks. Verify message integrity employs checksums or hash values—checksums maintain redundant file/message information for verification, while hash values generate unique strings from hashing functions where slight input changes produce significant output variations. Detect message delivery anomalies identifies man-in-the-middle attacks by checking delivery times; stable timing baselines allow detection of suspicious behavior, and abnormal connection/disconnection patterns indicate potential attacks.

Resist Attacks encompasses nine tactics. Identify actors determines the source of external input through user IDs for users or access codes, IP addresses, protocols, and ports for systems. Authenticate actors ensures subjects are who they claim using passwords, one-time passwords, digital certificates, two-factor authentication, biometric identification, CAPTCHA (Completely Automated Public Turing test), and periodic reauthentication like smartphone auto-locking. Authorize actors ensures authenticated subjects possess rights for data/service access and modification via per-actor, per-class, or per-role access control mechanisms. Limit access restricts access points and traffic types, minimizing attack surface; demilitarized zones (DMZ) illustrate this, positioned between Internet and intranet, protected by paired firewalls where internal firewall serves as single access point controlling intranet traffic. Limit exposure minimizes damage effects through passive defense by reducing accessible data/services per access point, preventing single-attack compromise of all resources. Encrypt data achieves confidentiality by applying encryption to persistent data and communication; symmetric encryption uses shared keys between readers/writers while asymmetric encryption uses paired public/private keys; communication links lacking authorization rely on encryption alone. Separate entities reduces attack scope through physical separation on different servers/networks, virtual machines, or air gaps (no electronic connection); sensitive data separation from non-sensitive data limits attack vectors from users with only non-sensitive access. Validate input defends early by cleaning and checking received input through security frameworks or validation classes performing filtering, canonicalization, and sanitization; this defends against SQL injection (malicious code in SQL statements) and cross-site scripting/XSS (malicious server code executing on clients). Change credential settings forces users to change default system security settings preventing attacker access through publicly available defaults; many systems also enforce periodic password changes.

React to Attacks includes three tactics. Revoke access severely limits sensitive resource access if attack is detected, even for legitimate users (e.g., limiting desktop access after virus compromise). Restrict login limits access from computers with repeated failed attempts, with lockout durations that may increase exponentially after each failure, though legitimate users may experience temporary restrictions. Inform actors notifies operators, personnel, or cooperating systems when attacks are detected.

Recover from Attacks includes audit and nonrepudiation tactics plus availability tactics for service restoration. Audit maintains records of user and system actions and effects, enabling attacker action tracing, identification, and future defense improvement or prosecution. Nonrepudiation guarantees senders cannot deny message transmission and recipients cannot deny reception, achieved through digital signatures and trusted third-party authentication combinations. Availability tactics for service restoration (covered in Chapter 4) reserve additional servers or network connections for such purposes.

Section 11.3 provides a Tactics-Based Questionnaire with columns for tactic support (Y/N), risk assessment, design decisions/locations, and rationale/assumptions, enabling architects to survey architectural choices systematically.

Security patterns complement tactics. Intercepting Validator inserts wrapper elements between message sources and destinations, particularly important for external message sources; primarily implements verify message integrity but incorporates detect intrusion, detect service denial, and detect message delivery anomalies. Benefits include comprehensive "detect attack" tactic coverage in single package; tradeoffs include performance costs from intermediary introduction and maintenance obligations as intrusion patterns evolve. Intrusion Prevention System (IPS) standalone elements identify and analyze suspicious activity, allowing acceptable activity and preventing/reporting suspicious patterns beyond anomalous messages alone. IPS benefits encompass most "detect attacks" and "react to attacks" tactics; tradeoffs include constantly updating pattern databases, system performance costs, and commercial off-the-shelf components potentially not suited to specific applications. Compartmentalization and distributed responsibility patterns combine limit access and limit exposure tactics. The chapter references availability patterns (Chapter 4) as applicable to security for countering availability-targeting attacks.

## Chapter 12. Testability

Testability is formally defined as the probability, given that a fault exists, that the software will fail on its next test execution. It reflects the intuition that a testable system readily "gives up" its faults during testing. Beyond fault revelation, architecture can also facilitate bug diagnosis by making it easier to replicate faults and identify root causes. The authors emphasize that achieving testability is as much about supporting the testing process—finding and fixing bugs—as it is about initial detection.

The foundational model of testing involves four elements: input fed to a program, output produced by the program, internal state that can be examined, and an oracle—a human or computational agent—that judges whether the output and internal state are correct by comparing them to expected results. This oracle function is central because determining test success requires comparing actual system behavior to expected behavior. The test harness is a critical infrastructure component: specialized software or hardware designed to exercise the system under test, potentially including record-and-playback capabilities, environment simulators, or production-grade monitoring systems.

Testability scenarios characterize testing requirements across multiple dimensions. The source identifies who performs testing (unit, integration, system, acceptance testers, or end users), and whether testing is manual or automated. The stimulus describes what tests accomplish: validating system functions, validating quality attributes, or discovering emerging threats to quality. The environment contextualizes when testing occurs—at major lifecycle milestones such as code unit completion, subsystem integration, full system implementation, production deployment, or customer delivery. The artifacts being tested range from individual code units through components, services, subsystems, to the entire system, plus the test infrastructure itself. The response measures how the system supports testing: executing test suites, capturing activity that led to faults, and controlling/monitoring system state. Response measures quantify testing effectiveness through effort to find faults, state space coverage achieved, probability of fault revelation, test execution time, effort to detect faults, time to prepare test infrastructure, effort to set specific states, and risk exposure reduction.

The tactics for testability fall into two primary categories. The first, Control and Observe System State, acknowledges that simple input-output testing is insufficient for complex systems; testers need fine-grained access to internal mechanisms. Specialized interfaces provide dedicated testing methods beyond normal functional APIs: set/get methods for important variables and modes, report methods returning full object state, reset methods restoring objects to specified states, and methods enabling verbose output, event logging at multiple levels, performance instrumentation, and resource monitoring. These interfaces should be clearly marked or separated from production APIs, though performance-critical and safety-critical systems face a dilemma—removing test code changes timing behavior relative to what was tested.

Record/playback addresses the difficulty of re-creating system states that expose bugs. Recording captures state crossing interfaces; playback uses that recorded state as input to reproduce the fault. Localizing state storage concentrates state in a single location, making it convenient to initialize arbitrary test states; when state is distributed, this becomes difficult or impossible. State granularity ranges from bit-level to coarse operational modes, chosen based on test requirements. State machines provide a convenient externalization mechanism for state tracking and reporting.

Abstract data sources applies similar control principles to input data. By abstracting the interfaces through which data is obtained, testers can substitute test databases or files for production databases without changing functional code. Sandboxing isolates the system from the real world, enabling risk-free experimentation with reversible consequences or complete rollback. Virtualizing resources is a common sandboxing approach—system time can be virtualized to test day-boundary conditions without waiting, memory and network behavior can be simulated, and battery states in embedded systems can be emulated. Stubs, mocks, and dependency injection are simple but effective virtualization forms.

Executable assertions embed test oracles into code by placing hand-coded assertions at key locations, checking whether data values satisfy specified constraints. Assertions capture pre-conditions, post-conditions, and class invariants, increasing observability when they fail. When systematically placed wherever data changes, they essentially create extended types—annotations with automatic checking. If assertions are correctly implemented and cover test cases, they embed the oracle logic in the code itself.

The second category, Limit Complexity, recognizes that complex software has large operating state spaces, making fault reproduction difficult and testing inefficient. Limiting structural complexity involves avoiding cyclic component dependencies, isolating external environment dependencies, reducing general component coupling (typically through lower coupling metrics), simplifying inheritance hierarchies by limiting derivation depth and width, and controlling polymorphism and dynamic calls. The response metric of a class—counting the methods of the class plus methods it calls from other classes—correlates empirically with testability and should be kept low. Architecture-level coupling metrics like propagation cost and decoupling level measure overall coupling. High cohesion, loose coupling, and separation of concerns (all modifiability tactics) help by giving elements focused responsibilities with limited interactions, reducing state space size and improving controllability and observability. Layered architectural patterns support testability by allowing lower layers to be tested first, then higher layers tested with confidence in lower-layer correctness.

Limiting nondeterminism addresses behavioral complexity. Nondeterministic systems are inherently harder to test than deterministic ones because exact state reproduction becomes difficult. The tactic involves identifying and eliminating nondeterminism sources: unconstrained parallelism, unpredictable concurrency, and uncontrolled async behavior. Some nondeterminism is unavoidable in multi-threaded event-responsive systems, but record/playback and other tactics can manage this complexity.

Patterns for testability decouple test-specific code from functional code. Dependency injection separates client dependencies from behavior through inversion of control, allowing test instances to be injected in place of production instances; clients operate without knowledge of concrete implementations, with specifics injected at runtime. This enables modern testing frameworks but introduces runtime performance unpredictability and initial complexity overhead. The strategy pattern changes class behavior dynamically, using context to select concrete implementations of abstract methods. This simplifies classes by separating concerns and algorithms but adds complexity when few runtime choices exist. The intercepting filter pattern injects pre- and post-processing between clients and services; any number of filters can be defined and applied in arbitrary order, which simplifies classes and promotes reuse but can introduce latency if large data volumes pass through multiple filters.

## Chapter 13. Usability

Usability is defined as concerned with the ease of accomplishing desired tasks and the quality of user support provided. The chapter identifies five areas that characterize usability concerns: learning system features (providing help and guidance for unfamiliar aspects), using systems efficiently (enabling user redirection after command issuance, suspending and resuming tasks), minimizing user error impact (supporting cancel and undo operations), adapting the system to user needs (automatic URL completion based on history), and increasing confidence and satisfaction (progress indicators with completion percentages for long-running operations).

The general usability scenario framework specifies source (the end user or external events), stimulus (desires to use efficiently, learn, minimize errors, adapt, or configure), environment (runtime or configuration time), artifacts (GUI, command-line, voice, touch interfaces), response (providing needed features, anticipating needs, appropriate feedback), and response measures (task time, error count, learning time, user satisfaction, successful operations ratio, time/data lost on errors).

Usability tactics are organized into two dimensions. Support User Initiative tactics address user-directed actions: Cancel requires a constant non-blocking listener, termination of the canceled activity, resource freeing, and notification of collaborating components. Undo requires maintaining sufficient state information through snapshots or reversible operations, recognizing that some operations (like find-and-replace with character conflicts) require elaborate records while others (unshipping packages, unfiring missiles) cannot be undone at all; undo variants range from single-undo to multi-step history. Pause/Resume allows users to temporarily suspend long-running operations to reallocate resources to other tasks. Aggregate enables grouping lower-level objects and applying operations to the group, reducing tedious repetitive actions and error potential.

Support System Initiative tactics rely on system models to predict behavior or user intention. Maintain task model determines context of user intent (exemplified by search-engine type-ahead and mail-client spell-correction). Maintain user model explicitly represents user knowledge, expected response times, and user-specific behavior (as in language-learning apps monitoring mistakes and providing corrective exercises, with customization as a special case). Maintain system model provides explicit self-representation to determine expected behavior and deliver appropriate feedback (manifest in progress bars predicting completion time).

The architecture must support user initiative by enumerating and allocating system responsibilities for user commands. The chapter emphasizes that usability and modifiability are strongly connected because UI design consists of iterative generation and testing—architects should design systems expecting to iterate the interface design repeatedly as users provide feedback and improvements are discovered. This iterative design pattern requires architectural support for painless modification.

Three key usability patterns are discussed: Model-View-Controller (MVC) and its variants (MVP: model-view-presenter, MVVM: model-view-viewmodel, MVA: model-view-adapter) separate underlying business logic from UI realization. In original MVC, the model sends updates to views the user sees and interacts with; user interactions (key presses, button clicks, mouse motions) are transmitted to the controller, which interprets them as model operations and sends them to the model, which updates state; the controller can also send updates back to the view. Updates are sent via the observer pattern in single-process MVC or publish-subscribe pattern when distributed across processes. MVC benefits include isolation of UI layout changes from model/controller logic and enables parallel independent development and testing of model, view, and controller aspects; a single model can support multiple views or views can serve multiple models. MVC trade-offs include complexity for intricate UIs when information scatters across components (requiring multiple component changes for single model modifications), unnecessary up-front complexity for simple UIs, and small added latency potentially problematic for very-low-latency applications.

Observer pattern links functionality with one or more views through subject-observer relationships where observers register with subjects; when subject state changes, observers are notified. Benefits include separating underlying functionality from presentation concerns and enabling runtime binding changes. Trade-offs include being overkill for single-view scenarios, requiring proper registration/deregistration (neglected deregistration causes memory leaks and continued invocation of obsolete observers), and potential "impedance mismatch" where fine-granularity state changes (temperature sensor reporting 1/100th degree fluctuations) conflict with coarse-granularity view updates (full-degree increments), wasting processing resources.

Memento pattern implements the undo tactic through originator (processing event streams that change state), caretaker (sending state-changing events to originator), and memento (snapshot of existing state). The caretaker requests mementos before state changes and passes them back to the originator for state restoration, keeping the caretaker unaware of state management details. Benefits include delegating complicated undo implementation and state-preservation logic to the state-creating class, preserving originator abstraction and hiding details from the rest of the system. Trade-offs include potentially arbitrarily large memory consumption (cutting and pasting many large document sections then undoing all operations noticeably slows text processors) and difficulty enforcing memento as opaque abstraction in some programming languages.

## Chapter 14. Working with Other Quality Attributes

The chapter begins by noting that while Chapters 4-13 provided comprehensive coverage of ten important quality attributes, they represent only the beginning of the spectrum of QAs that real systems may require. The authors argue that existing QA lists, while sometimes useful as checklists, often prove problematic in practice and should not constrain architectural thinking.

The chapter identifies several categories of quality attributes beyond the primary ten. Quality attributes of the architecture itself include buildability (measuring how efficiently the architecture enables rapid development by minimizing cost and time to implement), conceptual integrity (ensuring consistent design across the system—same communication mechanisms, error handling, logging, user interaction patterns—to enhance understandability and reduce confusion), and marketability (recognizing that architectural style can carry independent value regardless of other technical qualities, as evidenced by the emphasis on cloud-based and microservice architectures).

Development distributability, relevant for globally distributed teams, emphasizes that the system must be designed to minimize coordination needs through low coupling at both code and data model levels. Architectural structure must align reasonably with the organizational structure of development teams. System quality attributes address constraints of embedded systems, where software impacts physical properties like weight, power consumption, battery life, and pollution output. The chapter stresses that architects of such systems must collaborate with system engineers to ensure software decisions contribute positively to system-level goals.

Regarding standard QA lists, the chapter critically analyzes ISO/IEC FCD 25010, which organizes quality characteristics into functional suitability, performance efficiency, compatibility, usability, reliability, security, maintainability, and portability, each with multiple sub-characteristics. The authors identify three fundamental problems. First, no list is ever complete—organizations may need domain-specific qualities like "Iowability" (attracting talent and deploying state-of-the-art technology) or "manageability" (ease of system administration). Second, standard lists generate unproductive controversy about taxonomic placement rather than fostering understanding. Third, QAs resist neat classification; denial-of-service, for instance, concerns security, availability, performance, and usability simultaneously. The fundamental lesson is that QA names alone are useless; scenarios provide the precise way to specify what is actually meant by a quality attribute.

The chapter presents a systematic three-step approach for handling new quality attributes. First, capture scenarios through stakeholder interviews, decompose the new QA into refined subattributes (development distributability decomposed into software segmentation, composition, and team coordination), create specific scenarios, and generalize them into a general scenario capturing stimulus, response, and response measures.

Second, build a conceptual model by identifying parameters to which the QA is sensitive and the architectural characteristics influencing those parameters. The authors illustrate this with a queuing model for performance identifying seven critical parameters: arrival rate, queuing discipline, scheduling algorithm, service time, topology, network bandwidth, and routing algorithm. Each parameter can be affected by specific architectural decisions (fixed or dynamic routing, scheduling algorithm choice, dynamic server topology adjustment), making the model a powerful tool for understanding which design levers affect the QA.

Third, assemble design approaches by enumerating model parameters and, for each, identifying architectural characteristics and mechanisms that affect it. This is accomplished by revisiting familiar bodies of mechanisms, searching for successful existing designs addressing the QA, researching publications and blogs, and interviewing domain experts. The result is a finite, reasonably small list of mechanisms because both the number of model parameters and architectural decisions per parameter are bounded, making the design problem tractable.

## Chapter 15. Software Interfaces

Chapter 15 opens with the Mars Climate Orbiter failure—a system lost because of a fundamental interface mismatch between metric and English measurement systems—illustrating how interface errors cascade catastrophically. The authors define an interface as a boundary across which elements meet and interact, emphasizing that interaction encompasses control transfer, data transfer, or both. Elements possess actors (the other elements, users, or systems with which they interact) and an environment (the collection of all such actors). The chapter distinguishes between direct interactions through resources (procedures, data streams, shared memory, message passing) and indirect interactions (state implications observable by other elements).

Three core principles establish interface philosophy: (1) all elements have interfaces since they interact with some actors; (2) interfaces are two-way, comprising both provided resources and required resources from the environment; and (3) a single interface can mediate interactions with multiple actors simultaneously.

Interface structure breaks down into multiple components. Elements can expose multiple interfaces, each serving different classes of actors with separation of concerns—enabling role-based access control, security policies, debugging vs. production functionality, and read-only vs. modifiable access. Resources have both syntax (signature, argument names, data types) and semantics (return values, state changes, exceptions, side effects, future behavior implications, observable effects).

Interfaces consist of three interaction mechanism types: operations (invoked to transfer control and data, returning results and potentially failing), events (asynchronous notifications about state changes or message arrival), and properties (metadata such as access rights, units of measure, formatting assumptions that influence operation behavior). Complex interfaces combine all three, particularly in stateful, active elements.

Interface evolution employs three techniques: deprecation (removing resources with advance notice and error codes), versioning (maintaining old and new interfaces simultaneously), and extension (adding new resources while preserving backward compatibility, using internal interfaces and mediators when incompatibilities arise—illustrated through the apartment number address parsing example).

Interface design follows four core principles. The principle of least surprise demands consistent behavior matching actor expectations, with aptly-named resources providing clear hints about purpose. The small interfaces principle prescribes minimal information exchange between interacting elements. The uniform access principle requires hiding implementation details—actors should access resources identically regardless of whether values come from cache, computation, or fresh external fetch. The don't-repeat-yourself principle advocates for composable primitives rather than redundant ways achieving the same goal. Consistency in naming conventions, parameter ordering, and error handling across all elements minimizes development errors from misunderstanding.

Successful interaction requires agreement on four aspects: interface scope (which resources are directly available and access constraints), interaction style, data representation and structure, and error handling. Interface scope often uses gateway elements—message gateways that translate actor requests into target element requests, handling granularity mismatches, access restrictions, and providing stable interfaces despite underlying resource changes.

Interaction styles vary by deployment and coordination needs. Remote Procedure Call (RPC), modeled on imperative procedure calls but executing on remote networks, has evolved from synchronous text-based 1980s versions to gRPC—supporting binary parameter transfer, asynchronous operation, authentication, bidirectional streaming, flow control, and cancellation/timeouts, using HTTP 2.0 for transport. Representational State Transfer (REST), a web services protocol standardized around six constraints: uniform interface (same interaction form, typically HTTP, using URIs with consistent naming following least-surprise principles), client-server architecture, stateless interactions (clients encode authorization tokens in requests rather than assuming server state retention), cacheability of resources, tiered system architecture (servers decomposable into independently deployable layers), and optional code-on-demand capability. HTTP, the standard REST transport, uses command-URI structure with five key commands mapped to CRUD operations: post (create), get (read), put (update/replace), patch (update/modify), and delete (delete).

Data representation and exchange involves serialization decisions balancing expressiveness (arbitrary data structures, trees, multilingual text), interoperability (whether actors know how to parse the representation), performance (bandwidth efficiency, parsing algorithmic complexity), implicit coupling (shared assumptions between actors and elements risking errors on decoding), and transparency (message interception and observation capabilities, trading debuggability against security). Textual formats include XML and JSON; binary formats include Protocol Buffers. XML, standardized by W3C in 1998, uses tags and attributes to structure data and define custom languages via XML schemas, enabling rich information structures and validation against malformation. However, XML's verbosity and expensive parsing/validation create performance and bandwidth tradeoffs, requiring complete document reads and multiple passes for unmarshaling. JSON, a textual name/value pair and array format standardized in 2013 (originally from JavaScript), reduces verbosity by using single field names and enables stream parsing. Its data types resemble modern programming languages, making serialization/deserialization more efficient than XML, and it originated for lightweight data transfer between browsers and servers. Protocol Buffers, originated at Google and released as open source in 2008, employ binary formatting for extreme compactness and memory/bandwidth efficiency. Like XML, they support schemas defining valid structures with required, optional, and nested elements; like JSON, they use programming-language-close data types for efficient serialization. Protocol Buffers provide code generators for multiple languages through proto file compilation, and are mostly deployed as part of gRPC. Because proto files are compiled by language-specific compilers, specifications are necessary for correct behavior and serve as searchable interface documentation tracking value propagation through elements.

Error handling strategies must address nominal and failure cases. Actors need to determine whether elements function correctly and whether interactions succeeded. Common error-handling techniques include exceptions on failed operations, status indicator codes requiring testing, properties storing success/failure or error state flags, error events like timeouts for asynchronous failures, and error logs accessible via dedicated output data streams. Common error sources the interface must handle gracefully include incorrect/invalid/illegal information (null parameters where disallowed), element state incorrectness for handling the request, hardware/software errors preventing execution (processor failures, network unavailability, memory allocation failure), and misconfiguration. Error source indication helps systems choose appropriate correction and recovery: temporary errors with idempotent operations succeed via wait-and-retry; invalid input errors require request fixes and resending; missing dependencies need reinstallation; implementation bugs require test case additions preventing regressions.

Interface documentation distinguishes between actual observable behavior (how long operations take) and published contract (what actors depend on). Different stakeholders require different information: element developers need contract awareness and testability; maintainers make changes minimizing disruption; consuming-element developers understand contracts and use cases; systems integrators need complete resource and functionality details; analysts (e.g., performance analysts) need SLA guarantees; architects reusing assets need capabilities, quality attributes, and variability. Documentation exposes only what actors need to know, treating undocumented observable behaviors as subject to change and used at actors' own risk. Hyrum's law warns that with sufficient users, all observable behaviors become depended-upon regardless of contract promises, but undocumented dependents assume their own risk.

## Chapter 16. Virtualization

Virtualization enables multiple applications to share a single physical machine's resources while remaining isolated from one another. The chapter begins by explaining how the four primary resources are shared and isolated: CPU scheduling through a thread-scheduling mechanism that controls which thread has processor access; memory management through virtual memory technology that pages between physical RAM and secondary storage; disk access through controllers ensuring sequential data delivery and OS-level tags (user ID, group) for access control; and network isolation through IP addresses and port numbers that route traffic to the correct destination. These mechanisms form the foundation for higher-level virtualization.

Virtual machines (VMs) are multiple simulated computers executing on a single physical host, managed by a hypervisor—an operating system for VMs. Two types exist: bare-metal (Type 1) hypervisors run directly on physical hardware and are used in data centers and cloud environments; hosted (Type 2) hypervisors run atop a host operating system and enable developers to test applications incompatible with their development OS (e.g., Linux on Windows). Hypervisors require that guest VMs use the same instruction set as the underlying CPU—they do not translate instructions. Related technology, emulators (such as QEMU), simulate cross-processor execution and can emulate entire systems including BIOS and I/O devices, at the cost of performance.

The hypervisor performs two main functions: (1) it intercepts code that accesses virtualized disks or networks, executes those operations on behalf of the VM, tags external requests so responses route correctly, and forwards asynchronous I/O interrupts to the correct VM; (2) it manages VMs by creating and destroying them based on infrastructure instructions, monitoring health and resource usage, and enforcing resource limits (CPU utilization, memory, disk and network bandwidth). From the VM's perspective, it appears to execute on bare-metal hardware. Virtualization incurs performance overhead; Microsoft's Hyper-V reports approximately 10% overhead, though it varies by workload and virtualization technology. VMs are booted identically to physical machines: the hypervisor establishes disk connections, the boot loader loads the OS from disk into memory, and execution transfers to the OS. Two major implications for architects are: virtualization incurs performance cost (Type 1 modest, Type 2 potentially larger), and it enables treating runtime resources as commodities, deferring provisioning decisions to infrastructure teams.

VM images are disk snapshots containing the OS, services, boot loader, and all software bits. Three creation approaches exist: snapshot copying an already-running machine; adding software to an existing base image; or creating from scratch using OS installation media. Repositories provide minimal base images with just OS kernels or complete applications. Issues arise when using pre-built images: uncontrolled OS and software versions, potential vulnerabilities or malware, very large transfer sizes (8 GB images take 3+ minutes on 1 Gbps networks operating at 35% efficiency), and bundled dependencies. Best practice creates minimal OS-only images, then adds services post-boot during configuration, avoiding proliferation of service-specific images.

Containers maintain virtualization's isolation and resource-sharing benefits while reducing image size and transfer time. They operate under a container runtime engine (standardized implementations from Docker, containerd, Mesos) that acts as a virtualized OS, unlike VMs that virtualize physical hardware. All containers on a host share the same OS kernel through the runtime engine. Containers are allocated by finding a runtime engine with spare capacity; this may require creating new VMs to support additional runtime engines, allowing containers to run on bare-metal, VMs, or hybrid stacks.

Containers employ "layers" during image construction. The text illustrates this with LAMP stack (Linux, Apache, MySQL, PHP) image building: (1) Create a Linux container image and instantiate it; (2) load Apache using Linux features, exit, and inform the container management system to save this as a second image; (3) execute that image, load MySQL, exit, and save as a third image; (4) repeat for PHP to create the fourth image containing the complete stack. Container layers enable efficient updates: when only PHP changes, the management system moves only the PHP layer (saving transfer time and effort), whereas updating MySQL requires rebuilding from step 3 onward. VM loading takes minutes; container layer updates take microseconds to milliseconds. Container image creation scripts can be stored in version-controlled files (tool-specific, such as Dockerfile) and treated as code—designed, tested, reviewed, documented, and shared.

VMs and containers present distinct trade-offs. VMs virtualize physical hardware, allow any OS and multiple services per instance, support legacy and purchased software, and enable efficient interservice communication for tightly coupled services sharing large data sets. The hypervisor starts, monitors, and restarts the OS if it crashes. Containers share an OS (must be compatible with runtime engine), typically run one service per instance to keep images small, and are limited to Linux, Windows, or IOS. The container runtime engine starts, monitors, and restarts the single service; if that program completes, the container ends. Services within VMs are managed through OS functions; container services through runtime engine functions. VMs persist after services terminate; containers do not. Port usage restrictions exist for containers but not VMs. Single-service containers prevent image bloat and maintain fast startup times; multiple-service containers reduce efficiency. Some container runtime engines allow containers within a group (Pods) to share memory and IPC mechanisms like semaphores.

The Open Container Initiative has standardized the interface between container runtime engines and containers, enabling containers created by one vendor (Docker) to execute on another's runtime engine (containerd), and allowing development containers to deploy to production with simplified resource configuration.

Kubernetes is orchestration software for deploying, managing, and scaling containers. It introduces Pods—groups of related containers. In the Kubernetes hierarchy, nodes (hardware or VMs) contain Pods, and Pods contain containers. Containers in a Pod share an IP address and port space, communicate via IPC mechanisms (semaphores, shared memory), share ephemeral storage volumes with Pod lifetime, and have identical lifetimes (allocated and deallocated together). The purpose of Pods is to reduce communication costs between closely related containers; co-location on the same VM enables faster communication mechanisms than message passing. Service meshes are often packaged as Pods.

Serverless architecture carries container fast allocation further. Since container allocation takes seconds for cold start and milliseconds for reallocation (versus VM allocation taking minutes), a new container instance can be allocated per request instead of maintaining persistent containers. When a service completes request processing, it exits and the container is deallocated. The servers hosting container runtime engines are allocated dynamically, managed by the cloud provider, not the developer. Cloud providers offer this as function-as-a-service (FaaS). A critical consequence is that serverless containers must be stateless—any required state must be stored in infrastructure services (provided by the cloud) or passed as parameters. FaaS imposes practical limitations: limited base container image selection restricts programming language and library options; cold start times (first allocation and load) can be several seconds, with cached images handling subsequent requests nearly instantaneously; execution time is limited by the provider's timeout (enforced for economic and resource-pool fairness). Architects designing serverless systems may work around these limits by prestarting services to avoid cold-start latency, making dummy requests to keep services cached, or chaining requests across services to extend effective execution time.

## Chapter 17. The Cloud and Distributed Computing

Cloud computing provides on-demand, elastic computing resources accessible over the Internet with pay-per-use economics and self-provisioning capabilities. The chapter focuses on infrastructure-as-a-service public clouds organized into regions and availability zones to reduce latency and meet regulatory constraints like GDPR. Cloud data centers contain tens of thousands of physical computers organized in racks with high-speed network switches. Access occurs through two main gateways: the management gateway (for allocating, deallocating, and monitoring VMs) and the message gateway (for service communication).

The management gateway identifies a hypervisor with sufficient unallocated CPU and memory capacity to host a new VM instance, instructs it to create the VM, and returns the IP address and hostname registered in the cloud's DNS. The gateway also handles VM monitoring, billing collection, and destruction. VM allocation requires three essential parameters: cloud region, instance type, and VM image ID. Because cloud providers ensure sufficient physical hardware, allocation requests never fail due to resource insufficiency.

Failure is inevitable in cloud systems. Amazon reports that in a data center with approximately 64,000 computers with two spinning disk drives, roughly 5 computers and 17 disks fail daily. Network switches can fail, data centers can overheat, and natural disasters can cause total outages. Architects must design for availability and recognize that the specific physical computer running a service may fail at any time.

Timeouts are a critical availability tactic used to detect failures in distributed systems, but they introduce complications. A timeout cannot distinguish between actual failure (software crash, hardware failure, broken network) and slow responses exceeding the timeout period, causing some slow responses to be falsely labeled failures. Timeouts cannot isolate the failure location—software, virtual machine, physical machine, or network connection. Recovery actions carry latency penalties, such as starting a new VM (taking minutes) or establishing new sessions. Cloud systems show considerable response time variations, making timeout tuning critical. Typical systems employ parameterized timeout mechanisms with two parameters: timeout interval (e.g., 200 milliseconds) and failure recovery trigger threshold (e.g., triggering recovery after 3 missed responses over a 1-second window). Single data center systems can use aggressive timeout settings due to minimal network delays; systems operating over wide area networks, cellular radio networks, or satellite links require relaxed parameters to avoid unnecessary recovery actions.

Long tail latency is the phenomenon where a small percentage of requests experience dramatically longer completion times than average. In the AWS example of 1,000 "launch instance" requests, the distribution peaks at 22 seconds, the median is 23 seconds, the mean is 28 seconds, but the 95th percentile is 57 seconds—meaning 5 percent of requests take 2 to 10 times longer than average. Long tail latencies result from congestion (server queues, hypervisor scheduling) or failures anywhere in the request path, factors outside the service developer's control. Two techniques address long tail latencies: hedged requests (issuing more requests than needed and canceling extras once sufficient responses arrive) and alternative requests (issuing initial requests, then issuing additional requests after a threshold, canceling remaining requests once enough responses are received).

Vertical scaling (scaling up) handles overload by running services on larger VM instance types with more resources. However, maximum VM sizes limit vertical scaling effectiveness. Horizontal scaling or scaling out runs multiple copies of the same service behind a load balancer, implementing the maintain multiple copies of computations tactic and load balancer pattern.

Load balancers solve the central distribution problem: distributing requests arriving at a single service instance to multiple instances. A simple round-robin algorithm alternates messages between instances, balancing load uniformly when each request consumes roughly equal resources. The load balancer sits in the path of every message from client to service and must be extremely efficient. From the client perspective, the service's IP address is the load balancer's address registered in DNS; clients remain resilient to changes in the number of service instances or their addresses—an example of intermediary principle use. Multiple clients send messages to the load balancer, which distributes them as they arrive without caring about message source. Load balancers themselves can become overloaded, requiring global load balancing (a hierarchy of load balancers) as a solution.

Return messages from service instances go directly to clients (via the IP header's "from" field), bypassing the load balancer, so the load balancer receives no information about message processing success or duration. Health checks (a fault detection availability tactic) allow the load balancer to determine if instances are functioning. The load balancer periodically checks instance health through pings, TCP connection opening, or sending messages for processing. If an instance fails to respond, it is marked unhealthy and receives no further messages. Instances can transition from healthy to unhealthy to healthy again; the load balancer checks multiple times before moving instances to unhealthy lists and periodically re-checks unhealthy instances. Instances may crash and restart, re-registering with the load balancer, or replacement instances may be started. Load balancers with health checking improve availability by hiding instance failures from clients. However, instances might start processing requests but never return responses; clients must be designed to resend requests on timeout, and services must handle identical requests repeatedly.

State refers to information internal to a service affecting response computation. When services process multiple concurrent client requests (through multi-threading, multiple instances, or both), state management becomes critical. Three options exist: maintaining history in each service instance (stateful services), maintaining history in each client (stateless services), or maintaining history in external storage like databases (stateless services). Common practice favors stateless services because they lose history if they fail, recovery is difficult, and stateless services allow new instances to produce identical responses. When stateless design is difficult or inefficient, sticky sessions or session affinity approaches route subsequent client requests to the same service instance. Direct sessions bypass the load balancer for subsequent requests after the first one; sticky messages configure load balancers to send subsequent requests from a client to the same instance handling the previous request. Both approaches should be used only under special circumstances because instance failure breaks the affinity and overloaded instances remain overloaded.

Service instances frequently need to share small amounts of information across all instances (state or configuration data like load balancer IP addresses). Distributed coordination algorithms solve this problem, managing shared state reliably despite computer and network failures. Leslie Lamport's Paxos algorithm uses consensus mechanisms allowing participants to reach agreement despite failures. These algorithms are notoriously complicated to implement correctly; distributed coordination should never be solved from scratch. Existing solutions like Apache Zookeeper, Consul, and etcd provide proven, reliable implementations. Service instances store shared information in these coordination services, ensuring all instances see identical values.

Time coordination in distributed systems is non-trivial. Hardware clocks gain or lose approximately one second every 12 days. Multiple networked devices will have different clock readings. The Network Time Protocol (NTP) synchronizes time across devices by exchanging messages with time servers to estimate network latency and adjust client clocks. NTP achieves approximately 1 millisecond accuracy on local area networks and around 10 milliseconds on public networks; congestion can cause 100+ millisecond errors. Cloud providers use atomic clocks for extremely accurate time references, but the timing of receiving that answer remains uncertain. Most distributed systems assume some clock error exists between devices and avoid requiring time synchronization for correct operation. Device time can trigger periodic actions and timestamp log entries where precise coordination is unnecessary. Many systems prioritize event order over absolute timing (vector clocks trace actions propagating through services rather than comparing times), ensuring correct action sequencing. Critical coordination often uses vector clocks instead of wall-clock times. Architects should determine whether actual clock times are necessary or whether correct event sequencing suffices.

Autoscaling automatically creates new instances when needed and releases surplus instances when no longer needed, operating in conjunction with load balancing. Unlike traditional data centers requiring capacity for peak workloads with idle capacity during lower demand, cloud systems with pay-per-use pricing and elasticity allow on-demand capacity matching. Some systems with stable workloads might adjust resources monthly or quarterly; systems with dynamic workloads need automated addition and release of instances.

Autoscaling VMs works by monitoring service instance utilization. When utilization exceeds configured thresholds, the autoscaler creates new instances from the same VM image and notifies the load balancer of new IP addresses. Clients remain unaware of autoscaling activities; the load balancer and autoscaler handle scaling transparently. Architects configure autoscaler behavior through rules specifying: the VM image and configuration parameters; CPU utilization thresholds (above which new instances launch); CPU thresholds (below which instances shut down); network I/O bandwidth thresholds for creation and deletion; minimum and maximum instance counts. Autoscalers do not act on instantaneous metric values but on averages over time intervals because VM allocation takes minutes (loading the image, networking setup, OS boot). Typical rules follow the pattern "Create a new VM when CPU utilization is above 80 percent for 5 minutes." In addition to utilization-based rules, architects can specify time-schedule-based allocations reflecting historical usage patterns (e.g., more VMs during work hours, fewer after).

When removing instances, autoscalers cannot simply shut down VMs. The process called "draining" involves notifying the load balancer to stop sending requests to the instance, then notifying the instance to terminate activities and shut down after completing in-flight requests. Service developers must implement appropriate interfaces to receive and handle draining instructions.

Autoscaling containers differs from autoscaling VMs because containers run on runtime engines hosted on VMs, creating a two-level decision process. First, the autoscaler decides whether additional containers or Pods are needed. Second, it determines whether new containers can be allocated on existing runtime engine instances or whether new VM instances must be allocated. This two-level scaling is independent of VM scaling, allowing container scaling portability across cloud providers. However, integrated evolution of container and VM scaling could create provider dependencies.

The architecture supports availability through multiple mechanisms working in concert: load balancers distribute work, health checks detect failures, timeouts trigger recovery, hedged and alternative requests compensate for long tail latencies, and autoscaling maintains capacity. State management decisions (favoring stateless services with external coordination services) ensure new instances can immediately serve clients. Time coordination techniques and distributed algorithms ensure correct operation despite timing uncertainties and failures.

## Chapter 18. Mobile Systems

Mobile systems span smartphones, tablets, automobiles, aircraft, and any device without permanent access to abundant power. Five characteristics distinguish them from fixed systems: energy limitations, network connectivity challenges, sensor and actuator dependence, severe resource constraints, and distinctive lifecycle issues.

Energy management requires three architectural tactics. First, monitoring the power source through a battery manager component that periodically queries a smart battery management system (BMS) to determine remaining capacity. Applications register with this battery manager to receive shutdown notifications when power becomes critically low. The battery manager can also profile per-application energy consumption and detect capacity degradation as batteries age, allowing graceful degradation within changing power envelopes. Second, throttling energy usage through the tactic of reducing or terminating energy-consuming subsystems—dimming displays, reducing processor core counts and clock rates, lowering sensor sampling frequencies (e.g., GPS requests every minute instead of every few seconds), or selecting fewer location sources. Third, tolerating loss of power through architecture that survives sudden shutdown without data corruption, restarts robustly when power returns, and resumes operation within specified timeframes (e.g., 30 seconds). This requires hardware that tolerates power cuts, an OS that starts reliably, and software that maintains state consistency in permanent storage with application-level strategies for handling data received during inoperativity.

Network connectivity architecture must balance multiple concerns across rapidly evolving wireless standards. Protocols are categorized by transmission distance: Near Field Communication (NFC) for 4-centimeter ranges; IEEE 802.15 protocols (Bluetooth, Zigbee) for 10 meters; IEEE 802.11 (Wi-Fi) for 100 meters; IEEE 802.16 (WiMAX) for kilometers; and cellular or satellite for greater distances. Architects must minimize the number of communication interfaces to reduce power consumption, heat, and physical space, yet account for seamless transitions when devices move between protocol coverage areas—a video streaming over Wi-Fi should transparently switch to cellular as the device moves. Dynamic protocol selection based on cost, bandwidth, and power consumption is essential. Systems must be designed for modifiability since protocol standards evolve constantly. When multiple protocols are available, bandwidth requirements must be analyzed against distance, volume, and latency characteristics to make appropriate choices. The architecture must handle intermittent or absent connectivity by maintaining data integrity, resuming computation without consistency loss, and supporting graceful degraded modes. Security against spoofing, eavesdropping, and man-in-the-middle attacks is particularly critical for wireless mobile systems.

Sensor architecture transforms raw environmental data into meaningful system representations through a sensor stack—a confederation of devices and software drivers. The stack performs several functions: reading raw data through device drivers that query sensors either directly or through a sensor hub (a coprocessor that offloads sensor processing from the main CPU, reducing battery consumption); smoothing noisy data using techniques like moving averages or Kalman filters; converting sensor outputs to common formats (different sensors report temperature in different units, altitude in feet or meters, etc.); and sensor fusion, combining inputs from multiple sensors to build more accurate or complete representations than any single sensor allows. The architecture must handle sensor degradation—for example, GPS readings become unavailable in tunnels, so systems employ dead reckoning techniques to maintain position estimates. A concrete example is automobile pedestrian detection, which requires fusing thermal imagery, radar, lidar, and camera inputs to recognize pedestrians day or night in all weather conditions—no single sensor suffices.

Resource architecture requires explicit tradeoffs between a resource's functional contribution and its volume, weight, and cost. Physical packaging and weight constraints are fundamental—devices must fit human hands, match weight limits for vehicle payloads or track widths, or meet marketing specifications. Cost becomes critical when mobile systems are manufactured in millions of units; a small per-unit processor price difference translates to millions in total cost. Architects assign functional subsystems to different Electronic Control Units (ECUs) based on fit (matching ECU capability to function; graphics functions require graphics processors), criticality (critical functions get more powerful, reliable ECUs), location (aircraft cabin class differences), connectivity (ECUs communicating intensively should be co-located), and cost minimization (minimize ECU count). Cloud offloading shifts computation-intensive tasks like route planning or pattern recognition to cloud resources while keeping sensors on the mobile device, but requires evaluation of local processing power, connectivity adequacy, and performance impact when functions split between device and cloud. Mode operations allow subsystems to scale down when unused—sports cars switch to "race mode," disabling comfortable suspension calculation and activating torque distribution and braking power functions. Display strategy depends on resolution constraints; a 320x320 pixel display requires significant effort to minimize information density, while 1280x720 pixels allow richer displays. Using patterns like Model-View-Controller (MVC) allows the view to swap based on specific display characteristics.

Lifecycle architecture addresses four distinctive issues. Hardware is typically chosen before software is designed, imposing constraints that software must accommodate; architects should actively drive early hardware discussions, emphasizing tradeoffs rather than passively accepting outcomes. Testing presents unique challenges: verifying display layout correctness across diverse screen sizes and aspect ratios; testing operational edge cases like battery exhaustion and asynchronous UI behavior where problem recreation is timing-dependent; testing battery resource usage (simulators cannot accurately simulate battery consumption); and testing network transitions as the device moves between protocol networks seamlessly. Testing for transportation systems follows four levels: software component level (unit and end-to-end testing of individual components like lane detection), function level (testing components together in simulation), device level (testing integrated functions on target ECUs with simulated inputs), and system level (full-size configurations in labs and prototypes). Test traceability is critical—if an issue emerges at system level, it must be reproducible and traceable through all four levels since fixes must pass all levels again. Deploying updates must maintain data consistency (upgrades are typically automatic and one-way with no rollback), handle safety (updating engine control software while driving is dangerous; safe states must be defined), support partial deployment (frequently-changing components should update independently), and ensure extendability for long-lived systems (retrofitting cars, trains, aircraft with newer technology without dissecting legacy components). Finally, logs must be critical for incident investigation and analysis, offloaded to accessible locations independent of mobile system availability.

## Chapter 19. Architecturally Significant Requirements

An Architecturally Significant Requirement is one that will have a profound effect on the architecture—the system would likely be dramatically different in its absence. ASRs frequently take the form of quality attribute requirements such as performance, security, modifiability, availability, and usability, though they may also emerge from business goals or organizational constraints.

Requirements documents provide an important but incomplete source. The fundamental problem is twofold: (1) most information in specifications addresses features and functionality, which shape architecture the least, while QA requirements—which actually drive architectural decisions—are rarely captured adequately or testably. Vague statements like "The system shall be modular" or "The system shall exhibit high usability" are not useful because they are unfalsifiable. (2) Much of what matters to architects never appears in requirements documents. Business goals, developmental qualities like teaming assumptions, and organizational factors often drive architecture but do not manifest as observable system properties. Architects should systematically search requirements documents for categories including: usage patterns (user roles, internationalization), time concerns (timeliness, coordination), external elements (systems, protocols, sensors, middleware), networking properties, orchestration (processing flows), security properties, data persistence, resource constraints (time, concurrency, memory, devices, energy, scalability), project management factors, hardware choices, portability requirements, and named technologies. Evolution of any category should also be considered architecturally significant.

The Quality Attribute Workshop (QAW) is a structured stakeholder-focused method to generate, prioritize, and refine QA scenarios. The process includes: business/mission presentation where a business representative spends one hour presenting business context, broad requirements, constraints, and known QAs; architectural plan presentation where the architect shares current thinking; identification of architectural drivers where facilitators compile a distilled list and seek consensus; scenario brainstorming where stakeholders express concerns as scenarios with explicit stimulus and response; scenario consolidation merging similar scenarios when stakeholders agree; scenario prioritization allocating each stakeholder votes equal to 30 percent of total scenarios, distributable across any scenarios; and scenario refinement elaborating top-priority scenarios in six-part form (source–stimulus–artifact–environment–response–response measure).

A key insight is that stakeholders often cannot articulate precise QA requirements initially but possess domain knowledge that can be elicited through probing. By asking exploratory questions—"Would 24 hours be acceptable? How about 10 seconds?"—architects extract actionable ranges even when exact requirements are unknown. Different response times (24 hours versus 10 minutes versus 100 milliseconds) demand different architectural approaches. Architects contribute by explaining cost implications of requirements, enabling informed trade-offs, and suggesting architecturally superior alternatives.

Business goals drive system development and lead to ASRs through three relationships: (1) business goals often induce quality attribute requirements—desire to differentiate from competitors might demand unusually fast response times; understanding the business goal behind a stringent requirement allows architects to question it meaningfully; (2) business goals may affect architecture without inducing a QA requirement—an organization might mandate a database component because the database unit has staff needing assignment, a constraint never appearing in specifications yet architecturally consequential; (3) not all business goals lead to architectural implications. The PALM method conducts workshops with architects and business stakeholders in two core steps: business goals elicitation using research-based categories to guide discussion, capturing goals as business goal scenarios (structured seven-part expressions), consolidating nearly-alike goals, and prioritizing them; and identifying potential QAs from business goals where participants describe QAs and response measures helping achieve each goal. Candidate business goal categories include: growth and continuity of organization, meeting financial/personal/employee/society/state/shareholder objectives, managing market position, improving business processes, managing product quality and reputation, and managing environmental change.

A utility tree is a top-down hierarchical representation of QA-related ASRs critical to system success, beginning with "Utility" as the root node expressing overall system "goodness." The root elaborates into major QAs (performance, security, usability, modifiability, availability), with each QA decomposed into specific refinements—for example, performance decomposing into "data latency" and "transaction throughput" or "user wait time" and "web page refresh time." Under each refinement, specific ASRs are recorded as QA scenarios. Once scenarios populate the tree's leaves, evaluate them against two criteria: business value (H/M/L where high designates must-have, medium identifies important but non-critical, low describes nice-to-have) and technical risk (high means keeping you awake, medium means concerning but manageable, low means confident ability to meet). Scenarios rated (H, H) demand priority attention; many (H, H) scenarios suggest the system may not be achievable. A completed utility tree enables validation: QAs without scenarios may indicate unrecorded scenarios worth investigating; the tree provides a single document for design justification and stakeholder communication.

Requirements constantly change, demanding architects maintain open channels with key stakeholders to track evolving ASRs. Proactive architects stay ahead of change by designing for anticipated changes, sharing cost implications with stakeholders, and suggesting alternatives meeting goals within budget.

## Chapter 20. Designing an Architecture

ADD operates through a cycle of design rounds, each containing one or more iterations. Within each iteration, architects execute seven key steps: reviewing architectural drivers (design purpose, primary functional requirements, quality attribute scenarios, constraints, and concerns); establishing iteration goals by selecting specific drivers to address; choosing elements of the system to refine (decomposition for greenfield systems or refinement of existing elements in brownfield development); selecting design concepts—tactics, patterns, reference architectures, and externally developed components—to satisfy the chosen drivers; instantiating architectural elements and allocating responsibilities to them; creating preliminary documentation through sketches and decision records; and analyzing the design to verify iteration goals are achieved.

The identification and selection of design concepts represents one of the most challenging decisions. Rather than inventing novel solutions, architects should leverage existing design concepts through three primary sources: established catalogs and best practices (benefits include access to documented knowledge but require significant time investment); personal experience and prior successful designs (rapid identification but risk of applying solutions inappropriately); and collaborative brainstorming with peers to cross-pollinate diverse experiences. Selection involves trade-off analysis using techniques like SWOT (strengths, weaknesses, opportunities, threats) analysis and creating comparison tables of pros and cons for each alternative. Critical constraints and incompatibilities with prior design decisions must constrain option selection.

Instantiation of design concepts differs significantly by type. Reference architectures require customization through addition or removal of elements; patterns require adaptation of their generic structures to specific problems (e.g., client-server pattern requires deciding number and functionality of clients/servers); tactics may be realized through custom-coded solutions, patterns, or frameworks; and externally developed components may require inheritance-based extension or configuration parameter specification. When instantiating elements, architects must ensure high internal cohesion, narrow responsibility sets, and low external coupling. Element properties—configuration options, statefulness, resource management, priority, hardware characteristics—must be identified. Relationships between elements govern which elements communicate, via which protocols, synchronously or asynchronously, and the information exchange rates. Interfaces establish contractual specifications: external interfaces with other systems form constraints, while internal interfaces enable element collaboration and information exchange, typically identified through behavioral analysis using sequence diagrams and statecharts.

Preliminary documentation should be captured during design, not deferred to later. Sketches—whether on whiteboards, flipcharts, or in modeling tools—serve as initial documentation, complemented by responsibility tables. The three documentation purposes—analysis, construction, and education—guide what must be documented at this stage. Design decisions and their rationale should be recorded, particularly trade-off decisions. Progress tracking uses an architectural backlog (populated initially with drivers, then expanded with prototype needs, reverse-engineering tasks, and discovered concerns) and a Kanban board organizing backlog items into "Not Yet Addressed," "Partially Addressed," and "Completely Addressed" columns. The design round completes when high-priority drivers have been addressed, balanced against time and resource constraints. Risk should guide iteration termination criteria: architects must address drivers with highest priority and achieve certainty or "good enough" confidence in critical driver satisfaction.

## Chapter 21. Evaluating an Architecture

Architecture evaluation is the process of determining whether an architecture is fit for its intended purpose. Every architecture carries risks—events with both probability and impact; the estimated cost of a risk is the product of its probability and impact cost. Evaluation acts like insurance, with the amount needed depending on exposure to unsuitable architecture and organizational risk tolerance. Evaluations can occur at any development phase and build on core concepts: systems satisfy business goals exemplified by quality attribute scenarios, and these goals are achieved through tactics and patterns.

All evaluations share four essential steps: First, reviewers individually ensure they understand the current architecture state through documentation or architect presentation. Second, reviewers determine drivers—typically high-priority quality attribute scenarios rather than purely functional use cases. Third, for each scenario, reviewers determine if it is satisfied by posing two types of questions: whether the scenario is in fact satisfied (via architecture walkthrough or documentation), and whether other scenarios will not be satisfied due to decisions made in the portion being reviewed. Reviewers may propose alternatives to risky aspects, which should undergo the same analysis. Fourth, reviewers capture potential problems, which form the basis for follow-up; real problems either must be fixed or be explicitly accepted as risks by designers and project managers. The amount of analysis depends on decision importance, number of alternatives, and the principle of "good enough as opposed to perfect"—often, two alternatives do not differ dramatically enough to justify extensive evaluation time.

Three evaluation approaches exist: architect self-evaluation (implicit or explicit at each key design decision), peer review (conducted by internal peers typically within several hours to half a day), and outside evaluation (by external evaluators who bring objectivity and specialized expertise, though at considerable cost). Outside evaluators are often chosen for domain knowledge about important quality attributes, skill with particular technologies, or long experience in architecture evaluation.

Contextual factors for peer and outside evaluations include: availability of artifacts describing the architecture (some evaluations occur after system operation, requiring architecture recovery tools); visibility of results (full stakeholder knowledge or private evaluation); stakeholder participation (identifying which individuals must be present to articulate goals and concerns); and explicit capture and prioritization of business goals. Two formalized processes guide evaluation: the Architecture Tradeoff Analysis Method (ATAM) for outside evaluators, and Lightweight Architecture Evaluation (LAE) for peer review.

The ATAM, developed and refined over two decades, works for large systems across automotive, financial, and defense domains. It requires no prior architect familiarity and does not require a constructed system. The evaluation team consists of three to five external people filling specific roles: Team Leader (sets up evaluation, coordinates with client, ensures final report delivery), Evaluation Leader (runs evaluation, facilitates scenario elicitation and prioritization), Scenario Scribe (writes scenarios in sharable form during elicitation), E-Scribe (captures proceedings electronically), and Questioner (asks probing quality attribute questions). Project decision makers include the project manager, customer representative, and architect (whose willing participation is cardinal). Stakeholders—developers, testers, integrators, maintainers, performance engineers, users, and builders of interacting systems—articulate specific quality attribute goals; a rule of thumb suggests 10 to 25 stakeholders for large enterprise-critical architectures.

ATAM outputs include: a concise one-hour-or-less architecture presentation; articulation of business goals (frequently seeing light for the first time by some participants); prioritized quality attribute requirements as scenarios in the form described in Chapter 3; a set of risks and non-risks (architectural risks are decisions that may lead to undesirable consequences; non-risks are decisions deemed safe upon analysis); a set of risk themes—overarching patterns identifying systemic weaknesses in architecture or process that, if untreated, threaten business goals; mapping of architectural decisions to quality requirements (rationales for decisions); and identified sensitivity points and tradeoff points (sensitivity points are architectural decisions with marked effect on a quality attribute response; tradeoffs occur when two or more quality attributes respond differently to the same architectural decision). ATAM also produces intangible benefits: sense of community among stakeholders, open communication channels between architect and stakeholders, and shared understanding of architecture strengths and weaknesses.

The ATAM spans four phases spread over approximately three to four weeks. Phase 0 (Partnership and Preparation): evaluation team leadership and key decision makers work out exercise details, discuss project context, agree on logistics, negotiate stakeholder list and report delivery timeline, handle formalities like nondisclosure agreements, examine architecture documentation, and prepare presentations. Phase 1 (Evaluation, typically 1-2 days): evaluation team meets with decision makers for information gathering and analysis; the team analyzes highest-ranked scenarios one at a time, asking architects to explain how the architecture supports each scenario, probing for architectural approaches used, and documenting relevant decisions, risks, non-risks, and tradeoffs. A hiatus of about one week follows phase 1, during which the team summarizes findings and may analyze additional scenarios or clarify answers. Phase 2 (Evaluation continued, 2 days): expanded stakeholder participation; the method is re-presented, results of phase 1 are recapped, then steps 7-9 are carried out. Phase 3 (Follow-up, 1 week): evaluation team produces and delivers the final report, first circulated to stakeholders to ensure accuracy before delivery to the client.

The nine ATAM steps consist of: Step 1 (Present the ATAM): explain the process, answer questions, set context and expectations. Step 2 (Present Business Goals): a decision maker provides system overview from business perspective, describing system's important functions, technical/managerial/economic/political constraints, business goals and context, major stakeholders, and architectural drivers emphasizing ASRs. Step 3 (Present Architecture): architect presents architecture at appropriate detail level, covering technical constraints (operating system, platforms, interacting systems) and architectural approaches (patterns or tactics) used to meet requirements; architectural views (context, component-and-connector, module decomposition or layered, deployment) are primary vehicles for conveying architecture. Step 4 (Identify Architectural Approaches): analysis focuses on understanding architectural patterns and tactics and their known effects on quality attributes; for example, layered patterns promote portability and maintainability possibly at performance expense, publish-subscribe patterns are scalable in producers/consumers, active redundancy promotes high availability. Step 5 (Generate Quality Attribute Utility Tree): quality attribute goals are articulated via a utility tree (introduced in Section 19.4), making requirements concrete by defining precise quality attribute requirements architects worked to provide; the architect and decision makers determine scenario importance by rating technical difficulty/risk (H/M/L scale) and business importance. Step 6 (Analyze Architectural Approaches): the evaluation team examines highest-ranked scenarios one at a time; the architect explains how the architecture supports each scenario; evaluators probe for architectural approaches used, document relevant decisions, identify and catalog risks, non-risks, and tradeoffs; for well-known approaches, evaluators ask how the architect overcame known weaknesses or gained assurance the approach sufficed. Analysis is not meant to be comprehensive but to elicit sufficient information to establish links between architectural decisions and quality attribute requirements. A template (Figure 21.1) captures the analysis, identifying risks, non-risks, sensitivity points, and tradeoffs. Step 7 (Brainstorm and Prioritize Scenarios): stakeholders brainstorm quality attribute scenarios operationally meaningful to their roles (maintainers propose modifiability scenarios, users propose ease-of-use scenarios, QA proposes testing scenarios); scenarios are then merged where they represent the same concern and stakeholders vote, with each allocated votes equal to 30 percent of the scenario count rounded up (e.g., 40 scenarios yields 12 votes per stakeholder); the prioritized list is compared with the utility tree—agreement indicates alignment between architect and stakeholders; additional driving scenarios discovered may indicate a risk if discrepancy is large. Step 8 (Analyze Architectural Approaches with stakeholders): the architect explains how architectural decisions contribute to realizing highest-ranked newly generated scenarios; evaluators perform the same analysis as Step 6, typically covering the top five to ten scenarios as time permits. Step 9 (Present Results): the evaluation team groups risks into risk themes based on common underlying concern or systemic deficiency (e.g., documentation inadequacy, insufficient backup capability); for each risk theme, the team identifies affected business goals, relating final results to initial presentation and elevating risks to management attention. The presentation summarizes architectural approaches documented, prioritized scenarios from brainstorming, the utility tree, discovered risks and non-risks, sensitivity and tradeoff points, and risk themes with threatened business goals.

Lightweight Architecture Evaluation (LAE) is designed for project-internal peer review on a regular basis, using the same ATAM concepts but with omitted or shortened steps. An LAE may focus on changes since the prior review (in architecture or architecture drivers) or examine previously unexamined architecture portions. Duration depends on generated and examined quality attribute scenarios and system importance; LAE exercises range from a couple of hours to a full day, conducted entirely by internal organization members. Because participants are internal and fewer in number, achieving shared understanding takes much less time; being lightweight, LAE can be done regularly. The typical LAE agenda includes: Step 1 (Present Method Steps): may be omitted if participants are familiar with the process; Step 2 (Review Business Goals): a brief review ensures goals and priorities are fresh and contain no surprises; Step 3 (Review Architecture): brief overview using module and component-and-connector views, highlighting changes since the last review, tracing one or two scenarios through views; Step 4 (Review Architectural Approaches): architect highlights approaches used for specific quality attribute concerns, typically as part of Step 3; Step 5 (Review Quality Attribute Utility Tree): the team reviews an existing tree and updates it with new scenarios, response goals, or revised priorities and risk assessments; Step 6 (Brainstorm and Prioritize Scenarios): brief brainstorming establishes whether new scenarios merit analysis; Step 7 (Analyze Architectural Approaches): mapping highest-ranked scenarios onto architecture, consuming bulk of time and focusing on recent changes or previously unanalyzed architecture portions; if architecture has changed, high-priority scenarios should be reanalyzed in light of changes; Step 8 (Capture Results): the team reviews existing and newly discovered risks, non-risks, sensitivities, tradeoffs, and discusses new risk themes. No formal report is produced, but a scribe captures results shareable as basis for risk remediation. LAE is inexpensive, easily convened, and involves low ceremony, enabling rapid deployment for architecture quality assurance sanity checks; however, being internal, the evaluation team is typically less objective than external teams, potentially compromising results through fewer new ideas and dissenting opinions. Even lighter evaluation uses tactics-based questionnaires, which focus on a single quality attribute at a time, can be used by architects for reflection or to structure question-and-answer sessions (typically one hour per quality attribute), and can reveal design decisions taken and not taken plus buried risks. The example given describes using a security tactics questionnaire to uncover that a healthcare data system XOR'd (not truly encrypted) sensitive data before network transmission—technically meeting the "no data in the clear" requirement but using a crackable algorithm.

## Chapter 22. Documenting an Architecture

Architecture documentation fundamentally serves four distinct purposes: education (introducing new team members and stakeholders), communication among stakeholders (especially for future architects), analysis and construction guidance, and forensics for incident investigation. Documentation must be kept current and present information in ways that let different stakeholders quickly access what they need.

Architects should choose among three categories of notations by formality: informal notations (box-and-line diagrams and general visual conventions) that are easy to create but cannot be formally analyzed; semiformal notations such as UML and SysML that provide standardized elements and rules allowing rudimentary syntactic analysis; and formal notations (architecture description languages or ADLs) with precise mathematical semantics enabling full analysis and automation, though rarely used in practice. More formal notations cost more effort but reduce ambiguity and enable deeper analysis; informal notations are faster to create and understand but provide fewer guarantees.

The core documentation strategy is organizing architecture into distinct views, each representing a particular type of system element and its relationships. Module views show how software decomposes into manageable implementation units (classes, layers, aspects). Their elements are modules providing coherent responsibilities; their relations include is-part-of (hierarchical decomposition), depends-on (module dependencies), and is-a (generalization). Module view properties include name, responsibilities, and implementation information (source file mappings, test information, management data, implementation constraints, revision history). Module views support blueprint-for-construction, impact analysis of changes, incremental development planning, and communication of code structure to developers. However, they cannot express runtime behavior or performance characteristics.

Component-and-connector views show elements with runtime presence—components (processing units and data stores) and connectors (interaction pathways including service invocation, asynchronous message queues, event multicast, and data pipes). Connectors embody protocols governing interaction order, locus of control, and error handling. Attachments indicate which connectors link which components. C&C view properties include reliability (component/connector failure likelihood), performance (response time, bandwidth, latency, jitter), resource requirements (processing, storage, energy), functionality, security (encryption, audit trails, authentication), concurrency (separate process or thread), and runtime extensibility. C&C views enable showing how systems work through end-to-end activity traces and reasoning about runtime quality attributes like performance and availability.

Allocation views map software elements to environmental elements (processors, disk farms, file systems, development teams). The allocated-to relation shows which software elements execute or are developed within which environmental contexts. Software elements have required properties; environmental elements provide properties. Allocation can be static (fixed resource assignment) or dynamic (changing during execution based on conditions like load balancing). Dynamic allocations document the conditions, triggers, and mechanisms causing allocation changes.

Quality views—security views, communications views, exception-handling views, reliability views, performance views—are tailored views extracting relevant pieces from structural views and packaging them together to address specific stakeholder concerns, following ISO/IEC/IEEE standard 42010:2011 philosophy of creating views driven by architecture stakeholders' concerns.

Views can be combined through overlays when relationships between two views are tight. C&C views combine naturally with each other; deployment views combine well with C&C views showing processes; decomposition views combine with work assignment, implementation, uses, and layered views.

Behavior documentation complements structural views by describing how elements interact. Trace-oriented notations document sequences of activities responding to specific stimuli: use cases (describing how actors accomplish goals via primary/secondary actors and event flows), UML sequence diagrams (showing interaction sequences among element instances with vertical time axis and horizontal instance axis, using filled arrowheads for synchronous messages, open arrowheads for asynchronous messages, dashed arrows for returns, and execution occurrence bars indicating processing or waiting), UML communication diagrams (showing graphs of interacting elements annotated with sequence numbers, useful for verifying functional requirements), and UML activity diagrams (showing business processes as action sequences with fork nodes splitting into concurrent flows and join nodes synchronizing them). Comprehensive notations show complete behavior via state machines, allowing inference of all possible paths from initial to final state. UML state machine diagrams represent states as boxes, transitions as arrows labeled with events, with optional guard conditions in brackets and actions following slashes.

Architecture documentation should include mapping between views (showing many-to-many associations across views via tables), documented patterns (identifying which patterns are employed and why), context diagrams (depicting scope and interactions with environment), variability guides (showing how to exercise variation points), rationale (explaining design decisions, tradeoffs, evidence, discarded alternatives, and assumptions), glossary and acronym lists, and document control information (issuing organization, version, date, change history, change request procedure).

Design decisions—selecting among alternatives, creating structures, establishing relationships and interfaces, allocating resources—should be recorded with minimum information (decision description and rationale including discarded alternatives) or richer information documenting evidence, participants, shortcuts, tradeoffs, and assumptions. Recording decisions immediately during design prevents loss of rationale.

Architecture stakeholders have distinct documentation needs: project managers need module views (decomposition, uses, layered) and allocation views (deployment, work assignment) plus context diagrams and system overview; developers need system overview, assigned elements, interfaces, code assets, and constraints plus module views, C&C views, allocation views, variability guides, rationale, and interface documentation; testers and integrators need decomposition, uses, data model, all C&C views, deployment/install/implementation views, context diagrams, interface documentation, and behavior specifications; interoperating system designers need interface documentation and data models; maintainers need everything developers need plus uses views for impact analysis and design rationale for understanding original thinking; end users benefit from C&C views showing control flow and data transformation, allocation views showing deployment, and context diagrams; analysts need information from any documentation area supporting quality attribute evaluation; infrastructure support personnel need decomposition, uses, various C&C views, deployment/install/implementation views, and variability guides; future architects need comprehensive, candid rationale and design information about all key decisions.

Practical considerations include: modeling tools (SysML widely used, offering multiuser interfaces, version control, consistency checking, trace links, and code generation); online documentation using web pages and wikis allowing stakeholder contribution with controlled permissions; release strategy issuing documentation with major milestones ahead of schedule; documenting dynamic architectures by recording invariants across all versions, documenting allowed changes via variability guides, and auto-generating interface documentation from explicit interface mechanisms; traceability linking design decisions to specific requirements and business goals.

## Chapter 23. Managing Architecture Debt

The chapter establishes that not all debt is bad—sometimes principles are violated for worthy tradeoffs such as improved runtime performance or faster time to market. However, without quantitative analysis, it is difficult to secure stakeholder approval for refactoring work that shows "no new functionality." The proposed process requires three types of information: source code to determine structural dependencies, revision history from a version control system to determine co-evolution of code units, and issue information from an issue tracker to determine reasons for changes.

The core analytical tool is the Design Structure Matrix (DSM), an adjacency matrix in which entities (files) are placed on both rows and columns in the same order. Cells are annotated with the type of dependency—structural (inheritance, method calls) or evolutionary (files that change together in commits). A well-designed system exhibits a sparse DSM with entries only below the diagonal, indicating low coupling and no cyclic dependencies. In contrast, a dense DSM with annotations above the diagonal signals high architecture debt.

The chapter identifies six common anti-patterns that contribute to high coupling and low cohesion: (1) Unstable interface—an influential file with many dependents that changes frequently; (2) Modularity violation—structurally decoupled files that change together frequently; (3) Unhealthy inheritance—a base class depending on subclasses or a client depending on both parent and children; (4) Cyclic dependency or clique—a strongly connected group of files forming cycles; (5) Package cycle—two or more packages depending on each other rather than hierarchically; and (6) Crossing—a file with high fan-in, high fan-out, and substantial co-change relations with many dependents.

"Hotspots" are collections of files tightly coupled to each other and implicated in these anti-patterns. The identification process correlates bug fixes, changes, and code churn from issue trackers with anti-patterns detected in DSMs. Files participating in anti-patterns that also experience frequent bug fixes or changes are identified as hotspots and ranked by their contribution to overall system debt.

The refactoring strategy flows directly from the identified anti-patterns: cliques require dependency removal or reversal to break cycles; unhealthy inheritance demands functionality movement from child to parent; modularity violations require encapsulating shared secrets as new abstractions.

The SS1 case study, conducted with SoftServe, analyzed a 797-file system over two years with 2,756 issues and 3,262 commits. The analysis identified three clusters representing 291 files (one-third of the project) that accounted for 89 percent of 265 total defects. The architect estimated 14 person-months of effort to refactor the identified hotspots. Using past bug-fix rates as a conservative predictor, the team calculated average annual bug fixes of 0.33 per file project-wide, versus 237.8 per file in hotspots. Post-refactoring, hotspot files were expected to average 96 annual bug fixes, yielding estimated annual savings of 41.35 person-months. This 3:1 return on investment in a single year demonstrates the financial viability of architecture debt remediation.

The analysis process can be fully automated and integrated into continuous integration pipelines, requiring tools to extract issues and revision logs, reverse-engineer dependencies, build and analyze DSMs, and quantify debt associated with each hotspot.

## Chapter 24. The Role of Architects in Projects

The architect-project manager relationship is foundational to project success. While the project manager maintains external visibility—keeping projects on budget, on schedule, and properly staffed—the architect provides internal technical leadership. Both roles must remain aware of each other's concerns and constraints, ensuring that external stakeholder expectations align with internal technical realities. Table 24.1 maps this relationship across nine PMBOK knowledge areas: in Project Integration Management, the architect creates the design and organizes teams around it while managing dependencies; in Scope Management, the architect elicits requirements and estimates costs and risks; in Time Management, the architect helps define work breakdown structure and resource assignments; in Cost Management, the architect gathers team costs and recommends build-versus-buy decisions; in Quality Management, the architect designs for quality and defines metrics; in Human Resource Management, the architect defines skill sets, mentors developers, and recommends training; in Communications Management, the architect ensures coordination and oversees documentation; in Risk Management, the architect identifies, quantifies, and mitigates risks through architectural adjustment; and in Procurement Management, the architect determines technology requirements and recommends tools.

Incremental architecture releases documentation to stakeholders in a series of planned increments rather than delivering a complete architecture upfront. The first architectural increment should include the module decomposition structure (which defines team structure and project organization), the module "uses" structure (which enables planning of incremental releases), whichever component-and-connector structures best convey the solution approach, and a broad-brush deployment structure addressing major infrastructure questions. Subsequent releases should be guided by stakeholder information needs, with the architect using influence to ensure early releases address the system's most challenging quality attribute requirements, preventing unpleasant architectural surprises late in development.

Architecture and Agile development have historically been in tension, though this relationship has matured significantly. The authors present three approaches: Big Design Up Front (BDUF), a traditional waterfall-style approach with design effort concentrated upfront; emergent design, where architecture supposedly emerges organically during coding; and Iteration 0 (the recommended middle ground). The Iteration 0 approach begins with several iterations of attribute-driven design (ADD) to choose major architectural patterns, reference architectures where appropriate, and frameworks, then transitions to Agile development with careful management of evolving requirements. When new requirements arise, particularly those driving quality attributes, the architecture accommodates them through Agile experimentation using spikes—time-boxed tasks created to answer technical questions and gather information, developed in separate code branches and merged into the main codebase if successful.

The Agile Manifesto's implication that architecture is emergent and need not be planned upfront has proven problematic for medium to large systems. Quality attribute solutions—security, performance, safety, and many others—cannot be bolted on arbitrarily late; they must be designed into the architecture from the beginning, even if the first twenty planned deliveries don't exercise those capabilities. The authors analyze the twelve Agile principles from an architecture-centric perspective: they strongly agree with principles about satisfying customers, welcoming change, delivering frequently, having business and developers work together daily, sustainable development, continuous attention to technical excellence, and regular team reflection. They partially agree with principles about motivated individuals (but emphasize the need for skilled architects to guide less experienced developers), simplicity (but qualify that discarded work must not harm the system), and working software as the primary measure (but reject "only"). They disagree with the principle about face-to-face conversation being the most efficient method, noting that nontrivial systems require written documentation, interfaces, and protocols; otherwise how would maintenance teams learn system details long after original developers have departed? They also strongly disagree with the notion that the best architectures emerge from self-organizing teams, asserting instead that the best architectures are consciously designed by skilled, talented, trained, and experienced architects using methods like ADD.

At enterprise scale, frameworks like the Scaled Agile Framework (SAFe), which emerged around 2007, help large organizations coordinate dozens to hundreds of Agile teams into coherent large-scale systems. SAFe acknowledges "intentional architecture"—purposeful, planned architectural strategies and initiatives that enhance solution design, performance, and usability while providing guidance for inter-team synchronization—balanced with "emergent design" for fully evolutionary and incremental implementation. The authors argue that the qualities SAFe associates with emergent design would naturally emerge from intentional architecture, since the ability to rapidly evolve and support incremental implementations requires careful up-front thinking.

Distributed development—across floors, buildings, time zones, or global subcontractors—presents distinct challenges and opportunities. Cost savings may accrue in the long term but are often offset initially by rework as low-cost-venue developers gain domain expertise; skill set availability and local market knowledge can be significant advantages. The fundamental coordination problem is that when Module A uses an interface from Module B, changes to that interface require coordination between the teams maintaining each module. Methods for coordination include informal contacts (only possible co-located), documentation, scheduled or ad hoc meetings (face-to-face or remote), and asynchronous electronic communication (email, news groups, blogs, wikis). The key architectural implication: minimization of module dependencies takes on heightened importance in distributed development because dependencies among modules owned by globally distributed teams are more likely to be problematic. Documentation likewise becomes critical, as remote teams lack informal coordination possibilities such as hallway conversations or impromptu meetings. The chapter notes that the COVID-19 pandemic accelerated the transition to distributed and work-from-home practices as the norm, eliminating traditional co-located coordination opportunities.

## Chapter 25. Architecture Competence

The chapter establishes that architecture competence rests on a triad of duties, skills, and knowledge. Duties are the activities architects must perform; skills are the abilities needed to perform those duties well; and knowledge is the body of information required. A competent architect is distinguished not by talent alone but by the ability to successfully execute required duties—infinitely talented architects who cannot perform their duties are not competent.

Architects perform diverse technical duties spanning five areas: creating architectures (designing or selecting architectures, creating design plans, identifying patterns and tactics, partitioning systems, building prototypes); evaluating and analyzing (assessing satisfaction of use cases and quality scenarios, performing trade-off analysis); documenting architectures (preparing documents and presentations, producing documentation standards); working with and transforming existing systems (maintaining and evolving systems, measuring architecture debt, refactoring to mitigate risks); and performing other architectural duties (selling the vision, providing technical guidance, leading improvement activities).

Beyond technical duties, architects undertake substantial nontechnical responsibilities organized into four areas. Management duties include supporting project management (budgeting, planning, resource management, scheduling, metrics), and managing the architect's team (building trusted relationships, coordinating, motivating, training). Organization- and business-related duties involve supporting the organization (growing evaluation capability, participating in hiring, developing intellectual property) and supporting the business (understanding business processes, translating strategy, communicating business value). Leadership and team building duties involve providing technical leadership (thought leadership, mentoring other architects) and building teams (aligning developers with architecture vision, mentoring junior architects, educating teams on architecture use, fostering professional development, maintaining morale).

For skills, architects require capabilities across four domains. Communication skills include outward communication (oral and written presentations, explaining to diverse audiences, persuading, selling multiple viewpoints) and inward communication (listening, interviewing, consulting, negotiating). Interpersonal skills focus on team relationships—being diplomatic, building consensus, handling conflict, inspiring collaboration, and mentoring. Work skills encompass leadership (decision-making, initiative, independence, influence), workload management (working under pressure, managing multiple complex tasks concurrently, prioritizing), skills for the corporate environment (strategic thinking, operating under constraints, understanding power dynamics), skills for handling information (detail-oriented yet big-picture vision), and skills for handling the unexpected (tolerating ambiguity, managing risk, adaptability, resilience). The authors emphasize abstract thinking as potentially the most important skill: the ability to see how different things are instances of the same underlying concept.

The knowledge body required spans four areas. Computer science knowledge includes architectural concepts (patterns, tactics, structures and views, reference architectures, quality attributes, evaluation models), software engineering (requirements through maintenance, systems engineering), design knowledge (analysis techniques, object-oriented design, UML and SysML), and programming knowledge (languages, specialized techniques for security, real-time, safety). Knowledge of technologies and platforms includes specific technologies (web applications, Internet technologies, hardware/software interfaces, operating systems) and general knowledge of industry directions and infrastructure impacts. Knowledge about organizational context includes domain knowledge (domain-specific technologies), industry knowledge (best practices, standards, onshore/offshore team environments), business knowledge (company practices, competitive analysis, business strategy, financial models, budgeting), and leadership and management techniques (coaching, mentoring, project management). The authors stress that experience forms an essential part of knowledge acquisition—one gains experience through apprenticeship and on-the-job application, not education alone.

The organizational level of competence is equally critical. The authors define architectural competence of an organization as the ability to grow, use, and sustain the skills and knowledge necessary to effectively carry out architecture-centric practices at individual, team, and organizational levels to produce architectures with acceptable cost aligned with business goals. Organizations perform duties (adequately funding architecture efforts, using the architecture workforce effectively through teaming), develop organizational-level skills (knowledge management, human resource management), and cultivate organizational knowledge (architecture-based life-cycle models).

Organizations can implement three categories of practices to improve architectural success. Personnel-related practices include hiring talented architects, establishing career tracks, making the architect position prestigious through visibility and rewards, encouraging professional organization membership, establishing certification programs, creating mentoring programs, providing training and education, measuring performance, rewarding or penalizing based on project success, and requiring external certifications. Process-related practices establish organization-wide architecture practices, clarify responsibilities and authority, create forums for architects to share information, establish architecture review boards, include architecture milestones in planning, give architects input into product definition, hold organization-wide conferences, measure quality of architectures, bring in external consultants, have architects advise on team structure, and ensure architects have influence throughout the project life cycle. Technology-related practices involve maintaining repositories of reusable architectures and design concepts and providing centralized resources for architecture tools and analysis.

The path to excellence involves mentoring and being mentored. While experience is the best teacher, individuals cannot accumulate in one lifetime all the experience needed to become great architects; mentoring enables secondhand learning. Architects should seek mentoring relationships with skilled architects they respect, either through formal programs or informally, and should extend beyond their organization to professional societies and meetups. Equally important, architects should mentor others—teaching a concept serves as a litmus test for deep understanding, and good teachers invariably report learning substantially from their students' questions and insights.

## Chapter 26. A Glimpse of the Future: Quantum Computing

The chapter establishes the quantum computing context by noting that code written in the 1960s-1970s still runs daily, implying systems architected today could operate for 10-20+ years and face quantum obsolescence. Unlike classical computers that merely perform computations faster, quantum computers leverage quantum physics to tackle problems classical machines cannot. Google's 2019 claim that a quantum computer solved a task in 200 seconds that would take supercomputers 10,000 years illustrates the revolutionary scaling, though quantum computers will not universally outperform classical systems—transaction processing tasks will likely remain classical domains, and quantum machines will never power phones or smartwatches.

The fundamental quantum unit is the qubit, distinguished from classical bits in three ways: (1) A classical bit is definitively 0 or 1; a qubit is characterized by two probabilities and a phase angle, where |α|² represents probability of measuring 0 and |β|² represents probability of measuring 1, with |α|² + |β|² = 1. (2) Classical bit reading is nondestructive; qubit measurement is destructive—the measurement collapses the qubit to either 0 or 1 and destroys the prior state. (3) Classical bits can be copied; qubits cannot (the "no-cloning theorem"), because measuring qubit A to copy it yields only 0 or 1, losing the probability information and phase. Qubits in superposition (non-zero probabilities for both 0 and 1) can be manipulated via phase, which does not affect measurement probabilities but provides another control lever.

Key qubit operations include: READ (destructive measurement producing 0 or 1 per amplitudes), NOT (flips probabilities so P(0) becomes P(1) and vice versa), Z operation (adds Π to phase modulo 2Π), and HAD/Hadamard operation (creates equal superposition from any input, with phase 0 radians for input 0, phase Π radians for input 1). Multi-qubit operations exist; CNOT (controlled not) uses one qubit as a control: if control is 1, it performs NOT on a target qubit; if control is 0, the target remains unchanged. Most quantum operations are invertible (given output, recover input), except READ due to destructive measurement.

Entanglement is the key quantum property without classical analog. Two qubits are entangled if their measurements always match, regardless of time elapsed or physical distance. This enables quantum teleportation: the indirect transfer of one qubit's state to another by accepting the destruction of the original qubit and leveraging entanglement between an intermediate qubit and the destination. Teleportation uses three qubits: A and B entangled, then ψ (payload) entangled with A. By measuring A and ψ (destroying both), two classical bits are propagated to B's location, allowing B to recreate ψ's state. This is inherently secure because eavesdroppers can only intercept two classical bits, not the entanglement itself. NIST is considering quantum-based communication protocols to develop HTTPQ as an HTTPS replacement before quantum computers become powerful enough to break classical encryption—a decadal transition window.

Quantum computers pose severe encryption risks via two algorithms. Grover's algorithm, a probabilistic inverse function calculator, requires ~2¹²⁸ iterations to invert a 256-bit hash, achieving quadratic speedup (quantum time ≈ square root of classical time). This renders password hashing, which assumes inversion is computationally infeasible, vulnerable. Shor's algorithm factors the product of two large primes (each >128 bits, product ~256 bits) in time proportional to log(bit count), compared to NP-hard factorization on classical computers. Since modern encryption (RSA) relies on the difficulty of factoring pq back into p and q, Shor's algorithm threatens all RSA-encrypted data, transforming "practically impossible without keys" into quantum-practical.

Beyond encryption, QRAM (Quantum Random Access Memory) is a theoretical necessity—it would accept superposed memory locations as input and return superposed contents in output, enabling efficient large-data access for machine learning and other applications. However, QRAM requires physical resources scaling linearly with retrieved bits, making very large retrievals potentially impractical; no working QRAM exists yet. The HHL algorithm (Harrow, Hassidim, Lloyd) inverts linear N×N matrices to solve equations Ax = b where A is sparse and well-conditioned (non-zero or near-zero determinant), with b quickly accessible (QRAM's role) and result x appearing in superposition. It uses amplitude magnification via phase manipulation but remains too complex to detail here.

Potential quantum applications span IBM's list: cybersecurity (proven substantial improvement over classical algorithms), drug development, financial modeling, better batteries, cleaner fertilization, traffic optimization, weather forecasting/climate modeling, and AI/machine learning. Except cybersecurity, these remain speculative despite active research with no public results yet. The authors compare quantum computers to the Wright Flyer at Kitty Hawk—enormous promise but tremendous work required to realize it.
