# DTL (Data Transformation) y mensajes HL7

Una **DTL** es una clase generada a partir de un `.dtl`, subclase de `Ens.DataTransformDTL`, que transforma un mensaje de un tipo a otro mediante una lista de `<assign>` (y opcionalmente `<if>`/`<foreach>`/`<code>`) definida en el editor visual. Puntos importantes para diagnosticar por qué un `<assign>` "pierde" un valor:

- **Orden de ejecución**: los `<assign>` corren de arriba hacia abajo, sin "return" implícito. Si dos `<assign>` distintos escriben sobre la misma property de destino (por ejemplo uno dentro de un `<foreach>`/`<if>` posterior), gana el último que se ejecuta — no el que se ve primero al leer el diagrama.
- **Campos repetibles de HL7** (ej. `PID-3`, Patient Identifier List, es un `SET` de `CX`): la sintaxis `{PID:3.1}` no es intercambiable con `{PID:3(1).1}` (repetición explícita) o `{PID:3().1}` (pensada para usarse dentro de un `<foreach>` que itera todas las repeticiones). Si el HIS de origen manda el valor buscado en una repetición distinta de la que asume el path, el `<assign>` no da error — el target simplemente queda vacío.
- **Message Schema Category / DocType**: el mensaje no se interpreta "en crudo", sino contra un schema (ej. `2.5.1:ADT_A03`, o uno custom con Z-segments). Si el DocType configurado en el Test del DTL no coincide con el que usa la Production real (el que detecta/fuerza el Business Service HL7 de entrada), los offsets de campo pueden desalinearse — [un caso reportado en la Developer Community](https://community.intersystems.com/post/hl7-message-test-fine-dtl-message-not-transformed-correctly-end-end-test) muestra exactamente esto: la DTL probaba bien en el Test tool pero fallaba en producción por un mismatch de Schema Category entre ambos contextos (autor por identificar; ver `references/comunidad.json` (`scripts/comunidad.py show`), veredicto sin verificar — el caso ilustra el síntoma pero no fue reproducido empíricamente en un proyecto propio).
- **"Allow empty segments in target"** (activado por defecto): si el motor intenta leer un campo de un segmento/grupo que no existe bajo el schema seleccionado, este flag silencia el error — solo queda una entrada de Trace categoría `xform`, sin error visible en el resultado del Test.

**Cómo debuggear:** usar el panel de Trace del Test de la DTL (Management Portal → Interoperability → Data Transformations → Test) para ver, acción por acción, en qué punto el valor se pierde; y validar el path de origen de forma aislada desde Terminal con `##class(EnsLib.HL7.Message).ImportFromStreamCategory(...)` + `GetValueAt(...)` antes de asumir que el problema está en la lógica de la DTL.
